## 1. Headline outcome

v0.4.5 moved fragment + rewrite generation for `cust build` /
`cust run` into CMake custom commands, so a no-op build spawns
**zero** codegen processes (V45D-12). It explicitly **left the
test path alone** (V45D-11): `cust test` still runs an
unconditional driver pre-pass for every workspace member before
it ever touches CMake.

Today [`run_test_build_path`](../../cust/src/workspace.rs#L793)
does four driver-side generation steps per member, every
invocation:

1. **`build::run_phase1`** with `plan.test_build = true` вЂ” the
   surface fixed-point, which in test mode emits, per lib module,
   both the fragment header **and** a per-module test-discovery
   sidecar `.test-discovery/<crate>/<qname>.cust.tests`
   ([`build_surface_units`](../../cust/src/build.rs#L561) test
   branch).
2. **`build::write_rewrite_tree`** вЂ” the `.rewrite/<crate>/вЂ¦`
   sources (already migrated to per-file `rewrite-file` commands
   for the build path in v0.4.5; the test path still calls the
   driver version).
3. **[`build::write_test_runner_tu`](../../cust/src/build.rs#L341)**
   вЂ” reads every unit sidecar for the crate, renders one
   `cmake/cust_test_main_<crate>.c` runner TU.
4. **[`build::write_integration_runner_tus`](../../cust/src/build.rs#L399)**
   вЂ” for each `tests/<stem>.c`: surface-pass the rewritten TU to
   emit its integration sidecar
   `.test-discovery/<crate>/tests/<stem>.cust.tests`
   ([`surface_pass_integration`](../../cust/src/build.rs#L468)),
   then render `cmake/cust_itest_main_<crate>__<stem>.c`.

Only then does it run one `cmake --build --target <crate>__test
вЂ¦`. So a no-op `cust test` re-runs N+M `clang -fsyntax-only` +
plugin spawns purely to regenerate byte-identical sidecars and
runner TUs вЂ” the exact inefficiency v0.4.5 removed for the build
path, still present on the test path.

After v0.4.6 the test-build path collapses to **emit + configure
+ build**, identical in shape to the v0.4.5 build path. The four
steps above become CMake custom commands with declared
`OUTPUT` / `DEPENDS`, anchored by the `EXCLUDE_FROM_ALL` test
targets that already list the runner TUs as sources.

```console
$ cust test            # cold
   Compiling cstd v0.4.6 (test)
   вЂ¦
    Finished test [unoptimized] target(s)
     Running unittests (cstd)

$ cust test            # nothing changed
    Finished test [unoptimized] target(s)   # 0 clang spawns, 0 plugin spawns
     Running unittests (cstd)
```

## 2. What v0.4.6 (test-codegen) ships

In-scope:

* **Two new hidden `cust internal` leaves** (V46D-1):
  `test-sidecar` (one `.cust.tests` sidecar, unit or integration)
  and `test-runner` (one runner TU from a sidecar set). They join
  the v0.4.5 `internal` group (`rewrite-file`, `surface-module`,
  `crate-header`, `surface-cycle`).
* **Unit test-sidecar + runner commands** (V46D-2/V46D-4): one
  `test-sidecar` `add_custom_command` per lib module, one
  per-crate `test-runner` command. The `<crate>__test` target's
  runner-TU source now points at the `test-runner` `OUTPUT`.
* **Integration test-sidecar + runner commands** (V46D-3): one
  `test-sidecar` + one `test-runner` per `tests/<stem>.c`. The
  `<crate>__itest__<stem>` target's runner-TU source points at
  the `test-runner` `OUTPUT`.
* **Test path collapses to emit + configure + build** (V46D-5):
  `run_test_build_path` drops `run_phase1` /
  `write_rewrite_tree` / `write_test_runner_tu` /
  `write_integration_runner_tus`; it keeps only the residual
  pre-build work v0.4.5 already kept on the build path (prelude +
  `ensure_dirs` + dep-symlink) and computes the
  `test_executable` / `integration_tests` **paths** (no
  generation side effect) for the runner to spawn.
* **No-op `cust test` = zero codegen spawns** (V46D-8): extends
  the V45D-12 property to the test path, with its own regression
  test.
* **cwork test dogfood** (V46D-9): `just test-cwork` passes on the
  new pipeline; the no-op property is verified against it.

Out of scope (deferred / unchanged):

* **No change to sidecar / runner-TU *contents* or on-disk
  layout** (V46D-6). `.cust.tests`, `cust_test_main_<crate>.c`,
  and `cust_itest_main_<crate>__<stem>.c` keep their exact paths
  and bytes. This is a *scheduling* change, like v0.4.5.
* **No test-mode fragment regeneration** (V46D-7). The test build
  consumes the build-mode fragments v0.4.5 already produces; the
  test path emits *sidecars only*, never a second copy of the
  fragments. (See V46D-7 for the one assumption this rests on.)
* **The other v0.4.6 `cust test` features** (`should_panic`,
  timeouts, flag parsing, `tests/common/`, вЂ¦) вЂ” separate design.
* **`cust check`** вЂ” has no CMake graph (V42D-15); unaffected.

## 3. Why now / boundary

1. **It finishes V45D-11.** v0.4.5 deliberately scoped itself to
   the build/run path to keep the diff bisectable, and filed the
   test path as RQ-V45-3. The build/run shape is now proven
   against cwork, so the test path can adopt it with confidence.
2. **It is mechanically the same change.** The test artifacts are
   "more custom commands of kinds we already emit" вЂ” a sidecar is
   a surface pass with a different plugin output, a runner TU is a
   concatenation like the crate header. No new CMake mechanics
   beyond what V45D-14 established.
3. **It is the same observed inefficiency.** A no-op `cust test`
   spawns one surface pass per lib module per member plus one per
   integration TU, every run, for byte-identical output.

**Not a new build system / not a layout change.** Same CMake +
Ninja backend; same files in the same places. The driver/CMake
split moves "test generation" from *before CMake* to *invoked by
CMake*, exactly as v0.4.5 did for build generation.

## 4. Design decisions (V46D-N)

### V46D-1 вЂ” Two new hidden `internal` leaves

**Proposed.** Add `test-sidecar` and `test-runner` to the v0.4.5
`Cmd::Internal` group (still `#[command(hide = true)]`, still
documented as not-a-public-contract). They wrap the *same*
refactored routines the driver uses today, so there is one
implementation per step and two entry points (the V45D-8
no-logic-fork guarantee, applied to the test path).

| subcommand | produces | wraps |
| --- | --- | --- |
| `cust internal test-sidecar --crate <n> --kind {unit\|integration} --module <qname>\|--stem <s> --source <src> --sidecar-out <p> --frag-dir <p> --prelude <p> --plugin <p> --std <s> [--cflag <f>]вЂ¦ [--define <D>]вЂ¦ [--include <p>]вЂ¦ [--dep <d>]вЂ¦ [--profile <p>] [--crate-header <p>]` | one `.cust.tests` | the test-mode body of `build_surface_units` (unit) / `surface_pass_integration` (integration) |
| `cust internal test-runner --crate <n> --out <p> --sidecar <p>вЂ¦` | one `cust_test_main_*.c` / `cust_itest_main_*.c` | `write_test_runner_tu` / `write_integration_runner_tus` rendering (`test_discovery::parse` + `test_runner::render_main_c`) |

`test-sidecar` carries the **full** `build_cflags` set for the
same reason V45D-15 gives for `surface-module`: a sidecar surface
pass that omits a user `-D` can discover a different set of
`[[cust::test]]` functions, and a custom command only re-fires
when its command line changes. The `-DCUST_TEST_BUILD=1` define
is part of that set (it is exactly what makes this a *distinct*
pass from the build-mode `surface-module`, see V46D-7).
`test-runner` is content-only (no compile), like `crate-header`.

**`test-sidecar` always writes its `OUTPUT`.** Ninja errors
(`output вЂ¦ not created`) if a custom command finishes without
producing its declared `OUTPUT`. Today the integration surface
pass runs *only* `if plan.plugin.is_some()`
([`write_integration_runner_tus`](../../cust/src/build.rs#L399))
and the unit sidecar relies on the plugin to write the file вЂ” so
a zero-test or no-plugin module may leave no sidecar on disk
(the driver `read_to_string` then tolerates `NotFound` as "zero
tests"). As a CMake `OUTPUT` that tolerance moves into the leaf:
`test-sidecar` must **unconditionally create the sidecar file**
(empty when the plugin is absent or discovers no tests) so the
downstream `test-runner` `DEPENDS` is always satisfiable. The
leaf writes an empty (or zero-entry) sidecar before invoking
clang, then lets the plugin overwrite it if present.

**Determinism (V45D-15 carry-over).** All repeated args
(`--cflag`, `--define`, `--include`, `--dep`, and the `--sidecar`
list) are emitted sorted/stable so the generated `CMakeLists`
bytes вЂ” and the V42D-8 configure-skip stamp вЂ” are reproducible.

### V46D-2 вЂ” Unit test-sidecar + runner commands

**Proposed.** For each lib module of a member with a lib half:

```cmake
add_custom_command(
    OUTPUT  "<td>/<crate>/<qname>.cust.tests"
    COMMAND "<cust>" internal test-sidecar --crate <n> --kind unit
            --module <qname> --source "<src>"
            --sidecar-out "<td>/<crate>/<qname>.cust.tests"
            --plugin "<plugin.so>" --prelude "<prelude.h>"
            --std <s> [--cflag вЂ¦]вЂ¦ [--define вЂ¦]вЂ¦ [--dep вЂ¦]вЂ¦
    DEPENDS "<src>" "<plugin.so>"
            "<frag>/<crate>/<imp>.cust.h"вЂ¦   # imported modules' BUILD-mode fragments
    VERBATIM)
```

Then one per-crate runner command:

```cmake
add_custom_command(
    OUTPUT  "<cmake>/cust_test_main_<crate>.c"
    COMMAND "<cust>" internal test-runner --crate <n>
            --out "<cmake>/cust_test_main_<crate>.c"
            --sidecar "<td>/<crate>/<q1>.cust.tests"вЂ¦
    DEPENDS "<td>/<crate>/<q1>.cust.tests"вЂ¦   # every unit sidecar
    VERBATIM)
```

The `<crate>__test` target
([`TestTargetView`](../../cust/src/cmake_emit.rs#L356)) already
lists `cust_test_main_<crate>.c` among its `sources` вЂ” that source
listing **is** the anchor (V46D-4), exactly how a `.rewrite`
source anchors its `rewrite-file` command (V45D-3/V45D-14a). No
`add_custom_target` is needed for the runner (unlike the crate
header, which no target lists directly).

The unit sidecar's `DEPENDS` mirror `surface-module`'s minus self:
source + plugin + **imported modules' build-mode fragments**.
Crucially it depends on the *build-mode* fragment outputs (the
v0.4.5 `surface-module` / `surface-cycle` outputs), so building
`<crate>__test` transitively materialises the build-mode surface
first; the test path never produces its own fragments (V46D-7).

### V46D-3 вЂ” Integration test-sidecar + runner commands

**Proposed.** For each `tests/<stem>.c` of a lib-half member:

```cmake
add_custom_command(
    OUTPUT  "<td>/<crate>/tests/<stem>.cust.tests"
    COMMAND "<cust>" internal test-sidecar --crate <n> --kind integration
            --stem <stem> --source "<rw>/<crate>/tests/<stem>.c"
            --sidecar-out "<td>/<crate>/tests/<stem>.cust.tests"
            --crate-header "<build>/<crate>/include/<crate>.h"
            --plugin "<plugin.so>" --prelude "<prelude.h>" --std <s> вЂ¦
    DEPENDS "<rw>/<crate>/tests/<stem>.c"   # the rewrite-file OUTPUT
            "<plugin.so>"
            "<build>/<crate>/include/<crate>.h"   # crate-header OUTPUT
    VERBATIM)

add_custom_command(
    OUTPUT  "<cmake>/cust_itest_main_<crate>__<stem>.c"
    COMMAND "<cust>" internal test-runner --crate <n>
            --out "<cmake>/cust_itest_main_<crate>__<stem>.c"
            --sidecar "<td>/<crate>/tests/<stem>.cust.tests"
    DEPENDS "<td>/<crate>/tests/<stem>.cust.tests"
    VERBATIM)
```

Two differences from the unit case, both inherited from the
existing driver shape:

* The sidecar's **source is the rewritten** `tests/<stem>.c` (a
  `rewrite-file` `OUTPUT`), not an original source вЂ” integration
  TUs are surface-passed after rewrite
  ([`surface_pass_integration`](../../cust/src/build.rs#L468)).
  v0.4.5 already emits a `rewrite-file` command for each
  integration source? **No** вЂ” V45D-3 excluded integration-test
  rewrites (they stayed driver-side under V45D-11). So this
  milestone **also adds the integration `rewrite-file` commands**
  (the natural completion of V45D-3) and points both the sidecar
  command and the `<crate>__itest__<stem>` target source at them.
* The sidecar `DEPENDS` the **crate header** `<crate>.h` (the
  `crate-header` `OUTPUT`), because an integration TU `#include`s
  the published surface rather than re-surfacing the lib.

The runner TU is one-sidecar-in, one-TU-out, anchored by the
`<crate>__itest__<stem>` target's source list
([`IntegrationTestView`](../../cust/src/cmake_emit.rs#L391)).

### V46D-4 вЂ” Anchoring + `EXCLUDE_FROM_ALL` interaction

**Proposed.** Both test targets are `EXCLUDE_FROM_ALL` (V42D-14),
so none of the new test custom commands are reachable from the
default `all` target. Consequences:

* **`cust build` still runs zero test generation** вЂ” the V45D-12
  no-op property is *strengthened*: a build never fires a
  `test-sidecar` / `test-runner` command because nothing in `all`
  depends on a runner TU. (Regression test in slice D.)
* **`cust test` fires exactly the needed commands** вЂ” it builds
  `<crate>__test` / `<crate>__itest__<stem>` via `--target`,
  which lists the runner TU as a source в†’ runs `test-runner` в†’
  whose `DEPENDS` pull the `test-sidecar` commands в†’ whose
  `DEPENDS` pull the build-mode fragments. The whole test
  generation chain is one Ninja sub-DAG rooted at the requested
  target.

This relies on the same "a custom-command `OUTPUT` named as a
target source becomes a build edge" mechanic V45D-3 uses for
rewrites; no new finicky corner beyond V45D-14(a). Slice D's
no-op test asserts the chain actually fires lazily.

### V46D-5 вЂ” Test build path collapses to emit + configure + build

**Proposed (mirror V45D-10).**
[`run_test_build_path`](../../cust/src/workspace.rs#L793) drops
the per-member `run_phase1` + `write_rewrite_tree` +
`write_test_runner_tu` + `write_integration_runner_tus` calls. The
residual per-member work is identical to the v0.4.5 build path:
materialise the prelude + `ensure_dirs` (V45D-14b), dep-symlink
refresh, and `BuildOutputs` synthesis. The `test_executable` /
`integration_tests` fields are filled from
[`layout.test_executable_path`](../../cust/src/target_layout.rs#L182)
/ `integration_test_executable_path` вЂ” **pure path computations**;
the binaries are produced by the single `cmake --build`. The
bin-only-member case keeps its current shape: members without a
lib half contribute no `test_executable` (the path helpers are
only consulted when `kind.has_lib()`, matching today's
`write_test_runner_tu` early `None` return). After this slice the
build and test paths differ only in which
`--target`s the drive step requests (and the `-DCUST_TEST_BUILD`
configure toggle, unchanged from V42D-14).

### V46D-6 вЂ” No content / layout change

**Proposed (mirror V45D / В§2).** A `.cust.tests` sidecar produced
by `internal test-sidecar` is byte-identical to one produced by
today's in-process test surface pass; a runner TU from
`internal test-runner` is byte-identical to today's
`render_main_c` output. Paths
([`test_sidecar_path`](../../cust/src/target_layout.rs#L100),
`cmake/cust_test_main_*.c`) are unchanged. Slice A proves this
with byte-equality tests before any `CMakeLists` change.

### V46D-7 вЂ” Test build consumes build-mode fragments (no test fragments)

**Proposed вЂ” the load-bearing assumption.** The test build
compiles the lib sources with `-DCUST_TEST_BUILD=1`, but the
**published surface** (`[[cust::pub]]` / `[[cust::pub_repr]]`
decls that fragments carry) is **not** conditional on
`CUST_TEST_BUILD` вЂ” that define gates the test *harness*
(`#ifdef CUST_TEST_BUILD` in [prelude.h](../../cust/src/prelude.h#L48)
and the runner template), not public API. Therefore:

* The test path **does not regenerate fragments**. The
  `test-sidecar` commands `DEPENDS` the build-mode fragment
  `OUTPUT`s and consume them directly. This avoids a fatal
  conflict: two custom commands writing the same
  `.h-fragments/<crate>/<qname>.cust.h` path would be a Ninja
  "multiple rules generate" error.
* The `test-sidecar` pass uses `PluginOutputs { fragment: None,
  test_sidecar: Some, module: Some(..) }` вЂ” exactly what
  `surface_pass_integration` already does for integration TUs;
  this milestone applies the same shape to unit sidecars (today
  the unit sidecar pass *also* re-emits the fragment as a side
  effect of `run_phase1`; that side effect is dropped).

**Guard.** Slice A adds a test asserting a build-mode fragment and
the (dropped) test-mode fragment are byte-identical for cwork, so
if a future feature ever makes pub surface conditional on
`CUST_TEST_BUILD` the assumption breaks loudly. If that day comes,
the fix is namespaced test fragments
(`.h-fragments-test/<crate>/вЂ¦`) + test-mode `surface-module`
commands вЂ” tracked as RQ-V46-3, **not** built now (YAGNI).

### V46D-8 вЂ” No-op `cust test` spawns zero codegen processes

**Proposed (verification target, mirror V45D-12).** A second
`cust test` with no source change must spawn **zero** clang and
**zero** plugin processes. Verified with the same `CUST_TRACE_INTERNAL`
trace-file method v0.4.5 uses (each `internal` leaf appends a
line; the second `cust test` must leave the file untouched).

### V46D-9 вЂ” cwork test dogfood

**Proposed (mirror V45D-13).** `just test-cwork` (cstd unit +
integration) passes end-to-end on the new pipeline; the no-op
property (V46D-8) is verified against it, and an edit to one cstd
module's tests is verified to rerun only that module's
`test-sidecar` + the crate `test-runner` + the recompile вЂ” not the
whole crate.

## 5. Open questions (RQ-V46-N) вЂ” resolved

Following the project convention (simple solution, match the
existing grain, or defer):

### RQ-V46-1 вЂ” Separate `test-sidecar`, or a second `OUTPUT` on `surface-module`?

**Resolved: separate command.** Folding the sidecar onto the
build-mode `surface-module` would (a) drag `-DCUST_TEST_BUILD=1`
and the test-discovery plugin output into the build graph that the
`all`-anchored lib target consumes, polluting non-test builds, and
(b) couple two outputs with different invalidation needs. A
distinct `test-sidecar` command keeps the build graph clean and
the test sub-DAG isolated behind `EXCLUDE_FROM_ALL` (V46D-4).

### RQ-V46-2 вЂ” Runner-TU command granularity?

**Resolved: per-crate (unit) / per-stem (integration).** Matches
the existing on-disk runner-TU layout (V46D-6) and the 1:N
sidecarв†’runner dependency. A single workspace-wide runner command
would re-render on any sidecar change (the coarse grain V45D
RQ-V45-1 already rejected).

### RQ-V46-3 вЂ” Test-mode fragments (if surface ever diverges)?

**Resolved: DEFERRED, guarded.** Assume build-mode fragments
suffice (V46D-7) and assert it in slice A. If a future feature
makes published surface conditional on `CUST_TEST_BUILD`,
introduce namespaced test fragments + test-mode `surface-module`
commands then. Not built now.

### RQ-V46-4 вЂ” Integration rewrites: migrate now or keep driver-side?

**Resolved: migrate now.** V45D-3 left integration-test rewrites
driver-side only because the whole test path was deferred. With
the test path migrating, the integration `rewrite-file` commands
are the natural completion of V45D-3 and are a prerequisite for
the integration `test-sidecar` `DEPENDS` edge (V46D-3). They are a
small, mechanical addition to the existing per-source rewrite
emit.

## 6. Verification target

A v0.4.6 test-codegen build is correct when:

1. **Cold test parity.** A clean `cust test` against cwork
   produces byte-identical sidecars, runner TUs, and test exes to
   v0.4.5 вЂ” generation moved, not changed (V46D-6).
2. **No-op test = zero codegen spawns** (V46D-8). A second `cust
   test` with no edits runs no `internal` command and no
   clang/plugin process.
3. **Single-module test incrementality.** Editing one module's
   `[[cust::test]]` body reruns that module's `test-sidecar` + the
   crate `test-runner` + that TU's recompile, and nothing else.
4. **`cust build` runs zero test generation** (V46D-4). A build
   (cold or no-op) fires no `test-sidecar` / `test-runner`
   command вЂ” proven by the trace file staying empty across a
   `cust build`.
5. **Integration parity.** Each `tests/<stem>.c` builds via its
   `rewrite-file` в†’ `test-sidecar` в†’ `test-runner` chain and
   yields the same exe + discovered tests as v0.4.5.
6. **Build-mode fragment reuse holds** (V46D-7). The test build
   consumes the build-mode fragments (no duplicate
   `.h-fragments` rule); the slice-A byte-equality guard passes.
7. **`just test-cwork` green** (V46D-9), with the no-op property
   verified against cwork.
