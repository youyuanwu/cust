# cust v0.4.5 — CMake-owned generation (`add_custom_command` codegen graph)

**Status:** 📝 **draft** (design only — opened 2026-06-13, not yet
sliced).
**Parent doc:** [cust-design.md §17 roadmap](cust-design.md).
**Previous milestone:** [v0.4.4.md](v0.4.4.md) (shipped 2026-06-13).
**Builds on / revisits:** v0.4.2's CMake/Ninja backend — in
particular **V42D-17** ("Who regenerates fragment headers?"),
which locked *driver-owned, in-process* generation and explicitly
costed-and-rejected the CMake-owned `add_custom_command` shape as
"Option B." This milestone **reverses that decision** for the
`cust build` / `cust run` path, because the single assumption that
sank Option B — "the surface pass is whole-crate (fixed-point),
not per-module" (V42D-17 Why-A reason #4) — no longer holds once
generation is decomposed into a per-module DAG (V45D-4).
**Minimum clang / CMake / Ninja:** unchanged from v0.4.4
(clang 19, CMake 3.21, any Ninja).

This file is the v0.4.5 record — scope, in-progress design
decisions (**V45D-N**), open questions (**RQ-V45-N**), and the
verification target. v0.4.5 moves fragment-header + `#cust use`
rewrite generation **out of the Rust driver's unconditional
pre-pass and into CMake `add_custom_command`s** with declared
`OUTPUT` / `DEPENDS` edges, so Ninja owns the
"is this generated file stale?" decision the same way it already
owns "is this object file stale?".

---

## 1. Headline outcome

Today every `cust build` runs an unconditional driver-side
pre-pass for **every** workspace member (not just the `-p`-scoped
one): `build::run_phase1` (the surface-pass fixed-point, one
`clang -fsyntax-only` spawn **per lib module**) followed by
`build::write_rewrite_tree` (re-scan + re-rewrite of every
source). The *writes* are byte-skipped — `writeFragmentIfChanged`
([plugin.cc](../../plugin/src/plugin.cc#L399)) and
`write_if_byte_different` ([build.rs](../../cust/src/build.rs#L375))
both compare bytes and skip the disk write so Ninja's
`OBJECT_DEPENDS` edges don't cascade — but the **computation that
produces those bytes runs every time**. A no-op `cust build`
still spawns N clang+plugin processes purely to regenerate
fragments that come out byte-identical.

After v0.4.5 the generation steps are CMake custom commands:

```cmake
# ---- crate: cstd — generated surface fragments ----
add_custom_command(
    OUTPUT  "…/.h-fragments/cstd/alloc.cust.h"
    COMMAND "/abs/path/to/cust" internal surface-module
            --crate cstd --module alloc
            --source "…/src/alloc.c"
            --frag-out "…/.h-fragments/cstd/alloc.cust.h"
            --prelude "…/prelude.h" --plugin "…/libcust_plugin.so"
    DEPENDS "…/src/alloc.c"            # + upstream fragments (V45D-4)
    VERBATIM)

add_custom_command(
    OUTPUT  "…/.rewrite/cstd/src/alloc.c"
    COMMAND "/abs/path/to/cust" internal rewrite-file
            --crate cstd --in "…/src/alloc.c"
            --out "…/.rewrite/cstd/src/alloc.c"
    DEPENDS "…/src/alloc.c"
    VERBATIM)
```

Ninja then runs each generator **only when its declared inputs
changed**, in **parallel across the DAG**, with **explicit
cross-member edges**. The Rust driver's `cust build` path stops
calling `run_phase1` / `write_rewrite_tree`; it emits the
`CMakeLists`, (re)configures iff the V42D-8 stamp moved, and runs
`cmake --build`. CMake calls **back into `cust`** for each leaf
generation step via a new hidden `cust internal …` subcommand
group.

```console
$ cust build           # cold
   Compiling cstd v0.4.5
   …
    Finished `dev` profile

$ cust build           # nothing changed
    Finished `dev` profile           # 0 clang spawns, 0 plugin spawns
```

## 2. What v0.4.5 ships (MVP)

In-scope:

* **Hidden `cust internal` subcommand group** (V45D-2): three
  leaf generators CMake invokes — `rewrite-file`,
  `surface-module`, `crate-header` — each a thin CLI wrapper over
  a refactored, reusable driver routine. Hidden from `--help`;
  documented as not-a-public-contract (same posture V42D-17
  "Risks accepted" already established for the generated
  `CMakeLists.txt`).
* **Rewrite custom commands** (V45D-3): one
  `add_custom_command(OUTPUT .rewrite/<crate>/<rel>.c DEPENDS
  <src>)` per lib module and bin module source (integration-test
  rewrites stay driver-side — V45D-11). `build::write_rewrite_tree`
  is removed from the `cust build` / `cust run` path.
* **Acyclic fragment DAG** (V45D-4): one
  `add_custom_command` per **lib** module (bins aren't
  surface-passed), `OUTPUT` its
  `.h-fragments/<crate>/<qname>.cust.h`, `DEPENDS` its own source
  **plus the fragment outputs of every module it imports**
  (intra-crate via `Module::imports`, cross-crate via the dep's
  real crate-header `OUTPUT` path). For acyclic crates — **every
  cust crate today** — Ninja's topological ordering over these
  edges replaces the in-process fixed-point loop: each module is
  surfaced exactly once, after its upstream fragments already
  exist on disk.
* **Crate-header custom command** (V45D-5): one
  `add_custom_command(OUTPUT <crate>.h DEPENDS <all fragments>)`
  that concatenates the published surface (today's
  `write_crate_header`), anchored into the build by a
  `<crate>_header` `add_custom_target(... ALL)` (V45D-14).
* **Graph anchoring + prelude/dir ownership** (V45D-14): the
  rules that make Ninja actually *run* each custom command, and
  the driver's residual pre-build responsibilities (materialise
  the prelude, `ensure_dirs`, refresh dep symlinks) that survive
  dropping `run_phase1`.
* **Full cflag threading + invalidation** (V45D-15): the
  `surface-module` command line carries the complete
  `build_cflags` set (`-std`, profile cflags, `[clang]`
  extra-cflags, defines, include dirs) and the plugin `.so` is a
  `DEPENDS`, so any input that can change a fragment also changes
  the command (re-fires it) — no stale fragments.
* **Cyclic-SCC fallback** (V45D-6, Option A): SCCs of size > 1 in
  the import graph (a `[[cust::pub_repr]]` cycle) get **one coarse
  custom command** that regenerates all fragments in the cycle
  together via the existing fixed-point loop. Acyclic modules stay
  fine-grained. No cust crate has such a cycle today; this keeps
  the door open without special-casing the common path.
* **`current_exe()` path-baking** (V45D-7): the generated
  `CMakeLists` references the cust binary by the absolute path
  captured at emit time. Folded into the V42D-8 configure-skip
  stamp automatically (it is part of the emitted bytes).
* **No-op-build = zero codegen spawns** (V45D-12): a second
  `cust build` with no source change spawns zero clang / plugin
  processes. Shipped with a regression test that asserts it.
* **cwork dogfood** (V45D-13): cwork builds end-to-end on the new
  pipeline; the no-op-build property is verified against it.

Out of scope (deferred — see §5 / §6):

* **`cust test` generation migration** (V45D-11 — DEFERRED). The
  test-build path (`run_test_build_path`) keeps its driver-side
  pre-pass for now: per-module test-discovery sidecars
  (`surface_pass` in test mode) + runner-TU rendering
  (`write_test_runner_tu`, `write_integration_runner_tus`) stay
  in the driver. Only `cust build` / `cust run` generation moves
  under CMake in this milestone. (RQ-V45-3 tracks the follow-up.)
* **`cust check` migration** (V45D-8 — DEFERRED, by design).
  `cust check` has no CMake graph (V42D-15); it keeps calling the
  in-process surface pass. It shares the *refactored* surface
  routine with `internal surface-module`, so there is one
  implementation, two entry points — no logic fork.
* **Direct `ninja` invocation as a supported workflow**
  (unchanged from V42D-17 "Risks accepted"): `cust build` / `run`
  / `test` / `check` remain the only supported entry points.

## 3. Why v0.4.5 now

1. **It completes V42D-16's promise.** v0.4.2 moved *codegen +
   link* under Ninja but left *generation* (fragments + rewrites)
   as an unconditional driver pre-pass — the one remaining piece
   of the build that ignores incrementality. v0.4.5 closes the
   gap: every build artifact, generated or compiled, is a Ninja
   node with declared inputs.
2. **It fixes a real, observed inefficiency.** A no-op `cust
   build` today spawns one `clang -fsyntax-only` + plugin **per
   lib module per member**, every invocation, to regenerate
   byte-identical fragments. On a multi-member workspace that is
   the dominant cost of an otherwise-empty build.
3. **The assumption that blocked it in v0.4.2 is gone.** V42D-17
   reason #4 rejected Option B because "the surface pass is
   whole-crate (fixed-point), not per-module — option B just
   moves the trigger into Ninja's scheduler at the cost of a
   subprocess." That was true *as long as the surface pass was an
   atomic fixed-point over the whole crate*. The insight in
   v0.4.5 is that **for an acyclic import graph the fixed-point is
   unnecessary**: order the per-module surface passes
   topologically (which is exactly what `DEPENDS` edges express)
   and each module sees its upstream fragments already written,
   so one pass per module suffices. The DAG genuinely *reduces*
   work (per-module granularity + parallelism), which is what
   reason #4 said Option B couldn't do.
4. **It supersedes the cheaper interim idea.** A driver-side
   input-stamp guarding `run_phase1` was considered first; it
   would have hand-rolled cross-member invalidation (member B's
   surface pass must rerun when upstream member A's surface
   changes). The custom-command DAG expresses that edge
   *declaratively* and lets Ninja — the tool built for it — own
   the invalidation, which is both less code and harder to get
   subtly wrong.

## 4. Boundary & non-goals

* **Not a new build system.** CMake + Ninja stay the backend
  (V42D-13). v0.4.5 only changes *which actor produces fragment
  and rewrite files* — from "driver, eagerly, every build" to
  "Ninja, lazily, on input change." The driver/CMake split V42D-2
  draws is **redrawn one notch**: cust-specific generation is
  still cust's code, but it is now *invoked by* CMake rather than
  *before* CMake. The leaf routines are still pure Rust inside the
  cust binary; CMake never learns anything about plugins or
  fragments beyond an opaque command line.
* **No change to fragment / rewrite *contents* or on-disk
  layout.** `.h-fragments/<crate>/<qname>.cust.h` and
  `.rewrite/<crate>/<rel>.c` keep their V42D-13 paths and exact
  bytes. This milestone is a *scheduling* change; a fragment
  produced by `cust internal surface-module` is byte-identical to
  one produced by today's in-process `surface_pass`.
* **No change to the public CLI.** `cust internal …` is hidden
  and exists only for CMake to call back into. Users keep running
  `cust build` / `run` / `test` / `check`.
* **`cust check` and `cust test` keep their current generation
  path** (V45D-8 / V45D-11). Migrating them is deferred; this
  milestone is scoped to the `cust build` / `cust run` path so the
  diff is bounded and bisectable.
* **No cycle support is *added*** — cyclic `[[cust::pub_repr]]`
  graphs already work via the fixed-point loop and keep working
  via the V45D-6 coarse-command fallback; v0.4.5 neither broadens
  nor narrows what converges.

## 5. Design decisions (V45D-N)

### V45D-1 — Generation moves into CMake `add_custom_command`s

**Locked (reverses V42D-17 for the build/run path).** Fragment
headers and `#cust use` rewrites become declared Ninja build
outputs produced by custom commands, not driver pre-pass
products. Per-TU compiles depend on the rewrite outputs as
*sources*; per-module surface commands depend on upstream
fragments as *inputs*. The driver's `cust build` path no longer
calls `run_phase1` or `write_rewrite_tree` (V45D-10).

This is the same Option B that V42D-17 sketched, refined with
per-module granularity (V45D-4). The "Risks accepted" clause of
V42D-17 (direct `ninja` invocation can see stale fragments — not
a supported workflow; clangd is unaffected because it reads
`compile_commands.json`, not Ninja) **carries forward unchanged**.

### V45D-2 — Hidden `cust internal` subcommand group

**Locked.** A new `Cmd::Internal(InternalArgs)` variant on the
[cli.rs](../../cust/src/cli.rs#L39) `Cmd` enum, itself carrying a
hidden sub-subcommand enum. Every `internal` command is
`#[command(hide = true)]` so `cust --help` never lists it; it is
documented (in the v0.4.5 closeout + a comment in the generated
`CMakeLists`) as internal scaffolding, not a public contract.

The three leaves:

| subcommand | produces | wraps |
| --- | --- | --- |
| `cust internal rewrite-file --crate <n> --in <src> --out <rw> [--dep <d>]… [--own-lib-header <p>] [--bin-half]` | one `.rewrite/<crate>/<rel>.c` | the `write_one_rewrite` body |
| `cust internal surface-module --crate <n> --crate-root <p> --module <qname> --source <src> --frag-out <p> --frag-dir <p> --prelude <p> --plugin <p> --std <s> [--cflag <f>]… [--define <D>]… [--include <p>]… [--dep <d>]… [--profile <p>]` (full `build_cflags` set — V45D-15) | one `.h-fragments/<crate>/<qname>.cust.h` | one iteration of the `surface_pass` per-module body |
| `cust internal crate-header --crate <n> --out <p> --frag <p>…` | one `<crate>.h` | the `write_crate_header` body |

Each leaf re-resolves only what it needs from its arguments (no
workspace re-discovery, no `Cust.lock` write, no dep-symlink
refresh). They are **side-effect-minimal**: read inputs, compute,
atomically write the one output, exit. Arguments are passed
explicitly (not re-derived from `Cust.toml`) so the command line
is self-describing and the emitter — which already has every path
resolved — is the single source of truth.

**Refactor shape:** the three driver functions
(`write_one_rewrite`, the per-module loop body of `surface_pass`,
`write_crate_header`) are extracted into reusable functions that
take explicit paths and return `Result<()>`. Both the in-process
callers (`cust check`, the deferred `cust test` path) and the new
CLI leaves call the *same* functions — no logic fork (this is the
V45D-8 guarantee).

### V45D-3 — Rewrite custom commands (drop `write_rewrite_tree`)

**Locked.** For every **lib module and bin module** source (NOT
integration-test sources — those stay driver-side per V45D-11,
since the test path is unchanged this milestone) the emitter
adds:

```cmake
add_custom_command(
    OUTPUT  "<rw>/<rel>.c"
    COMMAND "<cust>" internal rewrite-file --crate <n>
            --in "<src>" --out "<rw>/<rel>.c" [--dep <d>]…
            [--own-lib-header <p> --bin-half]
    DEPENDS "<src>"
    VERBATIM)
```

The rewrite output **anchors itself** into the build: it is
already listed as a SOURCE of the member's `add_library` /
`add_executable` target (V42D-13), and CMake/Ninja wire a SOURCE
that is a custom-command `OUTPUT` as a build edge automatically.
So no extra anchor is needed for rewrites (contrast V45D-5).

A rewrite output depends **only on its source's bytes**: `#cust
use` lowering substitutes a *static* fragment-path string (e.g.
`#include "…/util.cust.h"`), never the fragment's *contents*, so
there is no edge from a rewrite to any fragment. Clean 1:1. The
existing per-source `OBJECT_DEPENDS` fragment edges (V42D-6) stay
on the compiled object, unchanged — they remain the
"recompile-the-TU-when-an-imported-fragment-changes" edge.
`build::write_rewrite_tree` is deleted from the build/run path
(it survives for the deferred test path until V45D-11 lands).

`--dep`/`--own-lib-header`/`--bin-half` reproduce
`write_one_rewrite`'s directive-lowering inputs exactly: the
bin half of a lib+bin crate lowers `#cust use <crate>;` to its
own lib header (the carve-out at
[build.rs](../../cust/src/build.rs#L762)), so `rewrite-file`
needs the same `is_bin_half` + own-lib-header context to stay
byte-identical.

### V45D-4 — Acyclic fragment DAG (the core change)

**Locked.** Fragment commands are emitted for **lib modules
only** — bins are never surface-passed (`run_phase1` skips them;
nothing downstream consumes a bin's surface), so a bin TU reaches
its lib's fragments purely through the existing `OBJECT_DEPENDS`
edge (V42D-6), not through a surface command of its own.

For each lib module M in an acyclic crate, the emitter adds one
custom command:

```cmake
add_custom_command(
    OUTPUT  "<frag>/M.cust.h"
    COMMAND "<cust>" internal surface-module --crate <n>
            --crate-root "<root>" --module M --source "<src(M)>"
            --frag-out "<frag>/M.cust.h" --frag-dir "<frag>"
            --prelude "<prelude>" --plugin "<plugin>"
            --std <std> [--cflag <f>]… [--define <D>]…
            [--include <dir>]… [--dep <d>]… [--profile <p>]
    DEPENDS "<src(M)>"
            "<plugin>"                       # V45D-15: plugin rebuild re-fires
            "<frag>/N.cust.h"                # for each N in M.imports
            "<build>/<dep>/include/<dep>.h"  # real crate-header OUTPUT (V45D-14)
    VERBATIM)
```

The `DEPENDS` set is exactly M's source, the plugin `.so`, the
fragment outputs of its intra-crate imports (`Module::imports`,
[modules.rs](../../cust/src/modules.rs#L39)), and — for each
cross-crate dep M uses (`dep_imports`) — that dep's **real**
crate-header `OUTPUT` path
`build/<dep>/include/<dep>.h` ([crate_header_path](../../cust/src/target_layout.rs#L140)).
It must be the producing path, **not** the `deps/<dep>/…` symlink
([dep_dir](../../cust/src/target_layout.rs#L151)): Ninja keys
edges by literal path string, so depending on the symlinked path
would fail to connect the upstream dep's crate-header command to
this consumer. (The `#cust use <dep>` lowering inside the
fragment still resolves through the symlink at *compile* time;
that is a baked `#include` string, separate from the *graph*
edge.)

Because Ninja runs a node only after its inputs are built, **each
module is surfaced after its upstream fragments already exist on
disk** — so a single non-fixed-point pass produces the correct
fragment (no implicit-int recovery, no iter-2 re-resolution).
`internal surface-module` therefore runs the per-module body
**once**, not in a loop.

**Require-upstream (no silent blanking).** Today's in-process
`surface_pass` *blanks* a `#cust use crate::X` include when
`X.cust.h` is absent ([build.rs](../../cust/src/build.rs#L760)) —
tolerable only because the fixed-point loop revisits it. In the
one-shot DAG a missing imported fragment means a **missing
`DEPENDS` edge**, i.e. an emitter bug, and blanking it would
silently export the wrong ABI (clang's implicit-int recovery).
So `internal surface-module` **hard-errors** if any
`#cust use crate::X` names a fragment that does not exist on
disk, turning a graph-construction bug into a loud failure
instead of a corrupt fragment. (The fixed-point caller —
`surface-cycle` / `cust check` — keeps the tolerant blanking,
because there the loop is the recovery mechanism.)

This is why the milestone is worth it and why V42D-17 reason #4
no longer applies: granularity drops from whole-crate to
per-module, unchanged modules are skipped by Ninja, and
independent modules surface in parallel — all for free, from the
edges the import graph already encodes.

`cust check` does **not** use this DAG (it has no CMake); it keeps
the in-process fixed-point (V45D-8). The two are consistent:
acyclic crates converge in one fixed-point iteration anyway, so
the per-module DAG and the fixed-point produce identical bytes.

### V45D-5 — Crate-header custom command

**Locked.** One command per lib member concatenates the published
surface to its real header path
`build/<crate>/include/<crate>.h` ([crate_header_path](../../cust/src/target_layout.rs#L140)):

```cmake
add_custom_command(
    OUTPUT  "<build>/<crate>/include/<crate>.h"
    COMMAND "<cust>" internal crate-header --crate <n>
            --out "<build>/<crate>/include/<crate>.h"
            --frag "<frag>/A.cust.h" …
    DEPENDS "<frag>/A.cust.h" …             # every fragment
    VERBATIM)
```

**Anchoring (V45D-14).** Unlike a rewrite (whose output is a
target SOURCE) nothing in CMake consumes `<crate>.h` as a tracked
input — consumers `#include` it by an *absolute path baked into
rewrites*, invisible to the graph. So this `OUTPUT` would never
fire on its own. The emitter anchors it with a per-member
all-target:

```cmake
add_custom_target(<crate>_header ALL
    DEPENDS "<build>/<crate>/include/<crate>.h")
add_dependencies(<crate> <crate>_header)   # lib target needs the header
```

The `ALL` keeps it in the default `cmake --build`; the
`add_dependencies` edge makes any consumer that links `<crate>`
(its own bins, downstream members) order after the header exists.
A cross-member fragment command's `DEPENDS` on the dep header
(V45D-4) provides the finer per-fragment edge; the all-target is
the belt-and-braces anchor so a header-only change still
republishes even when no fragment node is otherwise scheduled.

It depends on every fragment so a surface change in any module
republishes `<crate>.h`; the byte-skip writer (V45D-9) keeps the
output's mtime stable when the concatenation is unchanged, so
downstream consumers don't churn.

### V45D-6 — Cyclic-SCC fallback (Option A)

**Locked: one coarse custom command per cyclic SCC.** A
`DEPENDS` cycle is a hard CMake error, so a `[[cust::pub_repr]]`
import cycle cannot be a fine-grained DAG. The emitter runs
Tarjan's SCC over `Module::imports`; for each SCC of size > 1 it
emits a **single** custom command whose `OUTPUT` is *all*
fragments in the cycle and whose `DEPENDS` is all their sources
(plus fragments of modules the SCC imports from outside itself):

```cmake
add_custom_command(
    OUTPUT  "<frag>/P.cust.h" "<frag>/Q.cust.h"   # whole SCC
    COMMAND "<cust>" internal surface-cycle --crate <n>
            --module P --source "<src(P)>"
            --module Q --source "<src(Q)>" …
            --frag-dir "<frag>" --prelude … --plugin …
    DEPENDS "<src(P)>" "<src(Q)>" <external-frags>…
    VERBATIM)
```

`internal surface-cycle` runs the **existing
`surface_pass_fixed_point`** over just the SCC's modules — the
same loop, same cap (`CUST_FIXED_POINT_CAP`), same §4 verbatim
non-convergence error from
[build.rs](../../cust/src/build.rs#L812). Acyclic modules in the
same crate stay fine-grained (V45D-4); only the SCC is coarsened.
**No cust crate has a cycle today**, so this path is exercised by
a synthetic fixture (slice D) and is otherwise dormant.

### V45D-7 — `current_exe()` path-baking + stamp

**Locked.** The emitter captures the cust binary's absolute path
via `std::env::current_exe()` at generate time and writes it into
every `internal` command line. This is the exact cost V42D-17
Why-A reason #1 wanted to avoid; it is accepted now because the
benefit (incremental generation) is the whole point of the
milestone.

No new stamp mechanism is needed: the baked path is part of the
generated `CMakeLists` bytes, which the V42D-8 configure-skip
stamp already hashes
([cmake_emit.rs](../../cust/src/cmake_emit.rs#L659)). A cust
upgrade that relocates the binary changes the bytes → stamp
mismatch → reconfigure. The cust *version* is likewise already in
the `@generated` header line, so a version bump reconfigures too.

### V45D-8 — `cust check` stays in-process (shared routine)

**Locked / DEFERRED migration.** `cust check` has no CMake graph
(V42D-15) and keeps calling `surface_pass_fixed_point` in-process.
It is **not** a logic fork: per V45D-2 the per-module surface body
is refactored into one reusable function, and both `cust check`
(in-process, wrapped in the fixed-point loop) and `cust internal
surface-module` (one shot, ordered by the DAG) call it. If a
later milestone gives `cust check` a CMake graph it can adopt the
same custom-command DAG; until then in-process is correct and
cheaper for a check-only invocation.

### V45D-9 — Restat + atomic byte-skip writers (no spurious cascade)

**Locked.** The content-skip property that v0.4.2 relied on is
preserved structurally:

* Each `internal` leaf writes its output via tmp-file + atomic
  rename and **skips the write when the bytes are unchanged** —
  the plugin's `writeFragmentIfChanged` already does this for
  fragments; `rewrite-file` and `crate-header` reuse
  `write_if_byte_different`. An unchanged regeneration therefore
  does **not** advance the output's mtime.
* CMake's Ninja generator marks custom-command edges `restat = 1`,
  so when a generator runs but its output mtime did not advance,
  Ninja re-stats and treats downstream nodes as up-to-date.

Net: a source edit that doesn't change a module's *public surface*
reruns that module's `surface-module` command (its source mtime
moved) but, because the emitted fragment bytes are identical, does
**not** recompile the module's downstream consumers — exactly the
v0.4.2 behaviour, now expressed through Ninja's restat instead of
the driver's eager byte-compare.

### V45D-10 — Driver build path collapses to emit + configure + build

**Locked.** On the `cust build` / `cust run` path,
[workspace.rs](../../cust/src/workspace.rs#L687)'s per-member loop
drops the `run_phase1` + `write_rewrite_tree` calls. The remaining
per-member work is: **materialise the prelude + `ensure_dirs`**
(both previously done inside `run_phase1` — see V45D-14), the
dep-symlink refresh (`refresh_dep_symlink`), and `BuildOutputs`
synthesis (`cmake_outputs_for`); fragment/rewrite/header
generation is now entirely inside the single `cmake --build`.
`cust build -p <name>` / `--bin <name>` scoping is unchanged (it
still lowers to `cmake --build --target`); the custom commands
for unscoped members exist in the graph but Ninja only runs the
ones the selected target transitively needs.

### V45D-11 — `cust test` generation stays driver-side (DEFERRED)

**DEFERRED.** Migrating the test-build path
(`run_test_build_path`) is out of scope. Its extra generated
artifacts — per-module test-discovery sidecars (`surface_pass` in
`test_build` mode) and the rendered runner TUs
(`write_test_runner_tu`,
[`write_integration_runner_tus`](../../cust/src/build.rs#L760)) —
add a second class of custom command (`internal test-sidecar`,
`internal test-runner`) and interact with the
`EXCLUDE_FROM_ALL` test targets (V42D-14). Keeping them
driver-side for now bounds the v0.4.5 diff to the build/run path.
The test path already byte-skips its writers, so it is no *worse*
than today; it simply doesn't get the no-op-build win yet.
(RQ-V45-3.)

### V45D-12 — No-op build spawns zero codegen processes

**Locked (verification target).** The headline property: a second
`cust build` with no source change must spawn **zero** clang and
**zero** plugin processes. Shipped with a regression test (slice
E) that builds cwork twice and asserts the second build runs no
generator. Because each `internal` leaf is a **separate process**,
an in-process counter cannot observe them — the test instead
either (a) has each leaf append a line to a trace file named by
`CUST_TRACE_INTERNAL=<path>` and asserts the file is untouched by
the second build, or (b) parses the `cmake --build --verbose` /
Ninja log and asserts no `CUSTOM_COMMAND` edge fired. Option (a)
is preferred (robust to Ninja log-format drift).

### V45D-13 — cwork dogfood

**Locked.** cwork (cstd + hello-cstd, both single- and multi-bin
after v0.4.4) builds end-to-end on the new pipeline; the
no-op-build property (V45D-12) is verified against it, and an
edit to one cstd module is verified to resurface only that module
+ its downstream importers (not the whole crate).

### V45D-14 — Graph anchoring + residual driver pre-build work

**Locked.** Two mechanics the rest of §5 depends on:

**(a) What forces each custom command to run.** CMake executes an
`add_custom_command(OUTPUT x)` only when `x` is reachable from a
built target. The three output classes anchor differently:

| output | anchor |
| --- | --- |
| rewrite `.rewrite/<crate>/<rel>.c` | it **is** a SOURCE of the member's `add_library`/`add_executable` (V42D-13) → automatic edge |
| fragment `.h-fragments/<crate>/<q>.cust.h` | (1) `OBJECT_DEPENDS` on the consuming rewrite source (V42D-6); (2) the crate-header command's `DEPENDS`; (3) a downstream fragment's `DEPENDS` (V45D-4). Any of these pulls it in. |
| crate header `<crate>.h` | **none implicit** → explicit `add_custom_target(<crate>_header ALL DEPENDS <crate>.h)` + `add_dependencies(<crate> <crate>_header)` (V45D-5) |

The fragment case relies on CMake's Ninja generator turning an
`OBJECT_DEPENDS` that names a custom-command `OUTPUT` into a real
order-only + restat build edge. This works but is a known finicky
corner; **slice C must include a test that proves a fragment is
(re)generated purely via its `OBJECT_DEPENDS`/header edges** (no
direct target lists it), so a CMake-version quirk can't silently
break the anchor.

**(b) Residual driver pre-build work.** Dropping `run_phase1`
(V45D-10) also drops the side effects it performed before any
generation: `layout.ensure_dirs()` and `materialise_prelude`
([build.rs](../../cust/src/build.rs#L152)). The `surface-module`
commands need the prelude file and the target dir tree to exist
*before* `cmake --build` runs. The driver keeps doing both, once,
in the build/run path right before driving CMake — the cheapest
correct option (the prelude is a single content-skipped write;
`ensure_dirs` is idempotent). The prelude is therefore **not** a
custom-command output: it has no per-member inputs, never changes
between members, and making it a node would only add a
no-value edge. The dep-symlink (`refresh_dep_symlink`) likewise
stays driver-side (it needs `build/<dep>/` to pre-exist, which
`ensure_dirs` guarantees).

### V45D-15 — Full cflag threading into `surface-module`

**Locked.** A fragment's bytes are a function of the *entire*
clang invocation `surface_pass` builds via `build_cflags`
([build.rs](../../cust/src/build.rs#L800)) — not just the source
+ plugin. The `surface-module` command line therefore carries the
complete set: `--std <std>` (manifest `[clang] std` or the clang
default), every profile cflag, every `[clang] extra-cflags`
entry, any `-D` defines, and the source's own include dir
(`--include`). Two reasons:

1. **Correctness.** A fragment surfaced without the user's
   `-DFOO=1` can differ from one surfaced with it (conditional
   `[[cust::pub]]` decls). The command must reproduce the exact
   flags the codegen compile will use.
2. **Invalidation.** A custom command re-fires only when its
   `DEPENDS` or its *command line* changes. If cflags were
   re-derived inside the leaf from `Cust.toml` (not passed on the
   line), editing `[clang] extra-cflags` would change neither the
   `DEPENDS` nor the command string → **stale fragment** (a
   reconfigure alone does not rerun a custom command). Putting
   the full flag set on the line ties fragment freshness to flag
   changes automatically, and folds into the V42D-8 stamp for
   free (the flags are part of the emitted `CMakeLists` bytes).

The plugin `.so` is additionally a `DEPENDS` (V45D-4) so a plugin
rebuild at the same path re-fires every surface command.

**Determinism.** All repeated args (`--dep`, `--cflag`,
`--define`, `--include`, and the `DEPENDS` fragment list) are
emitted in a **sorted, stable order** so the generated
`CMakeLists` bytes — and thus the V42D-8 configure-skip stamp —
are reproducible across runs.

## 6. Open questions (RQ-V45-N) — resolved

Following the project convention (simple solution, match the
existing grain, or defer):

### RQ-V45-1 — Rewrite-command granularity: per-file or per-crate?

**Resolved: per-file.** One `add_custom_command` per source. A
per-crate multi-output command would retrigger every rewrite on
any source change (the exact coarse-grain V42D-17 reason #4
criticised). Per-file matches the rewrite's true 1:1 dependency
(V45D-3) and costs only emitter loop iterations.

### RQ-V45-2 — One `internal` subcommand or three?

**Resolved: three leaves under one hidden `internal` group**
(V45D-2). They have genuinely different argument shapes and
outputs; collapsing them into one mega-command with a `--mode`
flag would just re-implement subcommand dispatch by hand. Three
named leaves keep each command line self-describing.

### RQ-V45-3 — Migrate `cust test` generation too?

**Resolved: DEFERRED** (V45D-11). Scope v0.4.5 to the build/run
path; file the test-path migration (`internal test-sidecar` +
`internal test-runner`, plus `EXCLUDE_FROM_ALL` interaction) as a
follow-up once the build/run shape is proven against cwork.

### RQ-V45-4 — Path quoting / spaces in generated command lines?

**Resolved: rely on `VERBATIM` + absolute paths.** Every emitted
`internal` command uses `VERBATIM` (CMake handles per-argument
quoting) and absolute paths the emitter already resolves. cust is
Linux-first today; Windows path/quoting is a non-goal until cust
targets Windows at all. No special handling beyond `VERBATIM`.

## 7. Verification target

A v0.4.5 build is correct when:

1. **Cold build parity.** A clean `cust build` against cwork
   produces byte-identical artifacts (fragments, rewrites,
   `<crate>.h`, archives, exes) to v0.4.4 — generation moved, not
   changed.
2. **No-op build = zero codegen spawns** (V45D-12). A second
   `cust build` with no edits runs no `internal` command and no
   clang/plugin process; `Finished` prints with no `Compiling`
   work.
3. **Single-module incrementality.** Editing one cstd module's
   *body* (no surface change) reruns that module's
   `surface-module` + `rewrite-file` + that one TU's compile, and
   **nothing downstream** (restat, V45D-9). Editing its *public
   surface* additionally recompiles its downstream importers.
4. **Cross-member edge.** Editing cstd's public surface reruns
   hello-cstd's affected compiles (the `OBJECT_DEPENDS` /
   `<crate>.h` edge) without any driver-side cross-member
   bookkeeping.
5. **Parallel generation.** Independent modules surface
   concurrently under `cmake --build -jN` (observable in the Ninja
   log); no driver-side serialisation of the surface pass remains
   on the build path.
6. **Acyclic = no fixed-point.** `internal surface-module` runs
   each module's body exactly once; the fixed-point loop is
   reached only via `internal surface-cycle` for a genuine SCC.
7. **Cyclic fallback.** A synthetic 2-cycle `[[cust::pub_repr]]`
   fixture builds via one coarse `surface-cycle` command and
   produces the same fragments the in-process fixed-point would;
   a non-converging 4-cycle still surfaces the §4 verbatim error.
8. **`cust check` unchanged.** `cust check` behaves exactly as in
   v0.4.4 (in-process surface pass, no CMake), sharing the
   refactored per-module routine with `internal surface-module`.
9. **`cust test` unchanged.** The test-build path still produces
   sidecars + runner TUs driver-side (V45D-11); `cust test`
   results are identical to v0.4.4.
10. **Stamp stability.** Toggling `-p` / `--bin` / build↔test does
    not reconfigure (the baked cust path + the graph are stable
    across those); a cust version/relocation bump *does*
    reconfigure (V45D-7).
11. **Anchoring holds (V45D-14).** A fragment whose only graph
    connection is its `OBJECT_DEPENDS` + crate-header edges (no
    target lists it directly) is still (re)generated by
    `cmake --build`; the crate header republishes via its
    `ALL` all-target even on a header-only change. Proven by a
    dedicated slice-C test so a CMake-version quirk can't
    silently break the anchor.
12. **No silent blanking (V45D-4).** `internal surface-module`
    hard-errors (non-zero exit, clear message) when a
    `#cust use crate::X` names a fragment absent on disk, rather
    than blanking the include — verified with a fixture that
    omits the upstream edge.
13. **Flag-change invalidation (V45D-15).** Editing
    `[clang] extra-cflags` (or `[clang] std`) re-fires the
    affected `surface-module` commands on the next build (no
    stale fragment), and reconfigures (the flags are in the
    emitted bytes).
