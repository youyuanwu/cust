# cust v0.4.1 — FFI & system types (typed surface, native linkage, `cust bindgen`)

**Status:** ⏸️ **design locked, implementation deferred**
(locked 2026-06-06; deferred 2026-06-06).
**Parent doc:** [cust-design.md §17 roadmap](cust-design.md).
**Previous milestone:** [v0.4.0.md](v0.4.0.md) (shipped).
**Minimum clang:** 19 (unchanged from v0.4.0).

This file is the v0.4.1 record — scope, design decisions
(**V41D-N**), open questions, and the verification target.
v0.4.1 closes the three FFI gaps that surfaced during the
v0.4.0 post-mortem (2026-06-05): how a cust crate exposes
types whose definitions originate in system headers, how it
records its native-library dependencies in `Cust.toml`, and
how it auto-generates the Pattern 3 shadow typedefs that
v0.4.0's surface-pass typedef fix (commit `28f3b47`) made
ABI-honest.

### ⏸️ Deferral notice (2026-06-06)

**v0.4.1 design is locked but implementation does NOT start
next.** The slot reorders behind v0.4.2 (build scripts),
v0.4.3 (`-jN` parallelism), and v0.4.4 (multi-bin) at
minimum. The full design here is preserved verbatim so it
ships intact when picked up.

**Why deferred:**

The opaque-tag pattern (V41D-2) is the central piece of
v0.4.1's user-facing value — without the
`@cust:requires`-banner-driven diagnostic, the milestone
collapses to "manifest field for `links =` plus a bindgen
CLI" which is real but small. V41D-13 records that the
clean `[[cust::requires("stdio.h")]]` spelling is
impossible due to upstream LLVM
[#45791](https://github.com/llvm/llvm-project/issues/45791)
(open since 2020-06-24, ~6 years, two abandoned patches,
most recent independent bump 2026-05). The workaround —
riding the payload on `[[clang::annotate("cust::requires",
"stdio.h")]]` — works, but ships a user-facing spelling
that is:

1. **Verbose and visually noisy.** Every Pattern 2 decl
   carries a 40-character ceremonial prefix that
   communicates nothing to a reader. cstd-style codebases
   would have hundreds of these.
2. **Easy to misspell silently.** The annotation payload is
   a string literal — `cust::require` /
   `cust::Requires` / `cust:requires` all parse cleanly,
   produce no warning, do nothing. The mitigation
   (V41D-2's plugin-side "warn on unknown `cust::*`
   prefixes") helps but doesn't eliminate the class.
3. **Awkward to teach.** Documentation that says "annotate
   your decl with `[[cust::pub]]` AND
   `[[clang::annotate("cust::requires", "<header>")]]`"
   asks every user to memorise the LLVM bug story to
   understand why two different attribute namespaces
   appear on the same line.
4. **Likely to be reworked anyway** once `[[cust::link(...)]]`
   ships (v0.4.2 with build scripts) and forces the
   project to confront the same workaround for a more
   user-visible attribute. Shipping `cust::requires` first
   commits us to a spelling we'll then have to either
   migrate or duplicate.

The two parts of v0.4.1 that **don't** depend on the
payload-bearing-attribute workaround are:

* **`[package] links = "..."` manifest field + duplicate
  detection** (V41D-3 / V41D-4). Pure metadata + workspace
  validation. No clang interaction.
* **`cust bindgen` CLI** (V41D-6 through V41D-9). Standalone
  subcommand wrapping `clang -ast-dump=json`. The generated
  output *does* use the V41D-2 spelling, so deferring the
  attribute defers the per-decl `cust::requires` emission
  too — but the type-shadow and prototype emission
  (the bulk of bindgen's value) works without it.

We could ship the manifest field + bindgen *without* the
attribute (a "v0.4.1-lite"). Decision: **don't**. The three
pieces compose into one user-facing story ("here's how to
wrap a system library cleanly"); shipping two-thirds of it
leaves the third missing in a way that fragments the
documentation and the v0.4.1 release notes. Better to wait
for either (a) LLVM #45791 to land, or (b) enough cust
dogfooding pressure to justify accepting the
`[[clang::annotate(...)]]` UX tax.

**Resumption criteria** (any one of these flips v0.4.1 back
to active):

1. **Upstream fix.** LLVM #45791 lands, a clang version ≥19

### Why FFI now

Three distinct gaps surfaced from v0.4.0 dogfooding:

1. **The Pattern 3 idiom is undocumented.** cstd's
   `[[cust::pub]] typedef __SIZE_TYPE__ usize;` trick is the
   only worked example of how to expose a stdlib primitive
   in a published `<crate>.h`. The v0.4.0 surface-pass
   typedef fix (`28f3b47`) made it ABI-honest. But §5 of
   cust-design.md treats it as a cstd-specific quirk, not a
   general producer pattern — and there's no story at all
   for opaque-tag system types (`FILE *`, `pthread_t`,
   `struct stat`). A consumer who `#cust use`s a crate whose
   public surface mentions `FILE *` gets a mysterious
   "unknown type name `FILE`" with no diagnostic pointing at
   the missing `#include <stdio.h>`.

2. **`Cust.toml` has no native-dep field.** Cargo's
   `[package] links = "..."` field is the foundation of the
   entire `*-sys` crate ecosystem. Without it, two cust crates
   each wrapping `libssl` cannot detect that they're
   double-linking incompatible versions; the driver has no
   global view of the native-library graph. The
   duplicate-detection guarantee that makes `*-sys` crates
   useful in Cargo doesn't exist in cust yet. The cost of
   adding it is one manifest field + one validation pass at
   workspace load time.

3. **No automation for Pattern 3.** Writing
   `[[cust::pub]] typedef __SIZE_TYPE__ usize;` by hand for
   every primitive is fine; writing `[[cust::pub]] typedef
   struct stat my_stat;` plus signatures for every libc
   function is tedious + error-prone + impossible to keep in
   sync. A small `cust bindgen <header> -o <out.c>` that
   emits a `.c` stub is the v1 answer. cust source is already
   C, so the output is just normal cust source — no separate
   "raw" layer, no `Result<T, BindgenError>`, no FFI boundary
   beyond `[[cust::pub]]`.

### Headline outcome

**Producer side** — exposing system primitives (Pattern 3,
documented in v0.4.1):

```c
// src/types.c — shadow typedefs, no <stdint.h> include
[[cust::pub]] typedef __SIZE_TYPE__   usize;
[[cust::pub]] typedef __INT32_TYPE__  i32;

// src/io.c — uses the shadows in its public surface
#cust use crate::types;

[[cust::pub]] usize crate_count_lines(const char *s);   // ABI-honest
```

→ published `<crate>.h` is self-contained — no `#include
<stddef.h>` injection, no opaque-tag burden on the consumer:

```c
typedef unsigned long usize;       /* clang's pretty-printer desugars */
typedef int i32;
unsigned long crate_count_lines(const char *s);
```

**Producer side** — exposing an opaque system tag
(V41D-2 / V41D-13 — clang::annotate-based payload
mechanism):

```c
// src/file.c
#include <stdio.h>

[[cust::pub]]
[[clang::annotate("cust::requires", "stdio.h")]]
int crate_write_line(FILE *fp, const char *line);
```

(Spelling rationale — see V41D-2 / V41D-13. Short version:
clang's plugin-attribute parser strips arguments from
`[[cust::*]]` bracket-syntax attributes, so the
`requires` payload rides on the built-in `[[clang::annotate(...)]]`
attribute which *does* preserve string-literal args. The
final-form spelling is uglier than v0.4.1's first-draft
`[[cust::requires("stdio.h")]]` but actually works.)

→ published `<crate>.h` carries a machine-readable banner;
the driver checks it when consumers `#cust use` the crate:

```c
/* @cust:requires <stdio.h> */
int crate_write_line(FILE *fp, const char *line);
```

A consumer who forgets the prerequisite gets a clear
diagnostic instead of clang's "unknown type FILE":

```
error: `#cust use crate;` requires `<stdio.h>` to be included
  --> src/main.c:3:1
   |
 3 | #cust use crate;
   | ^^^^^^^^^^^^^^^^
   = note: crate `crate` declares system-header prerequisites
   = hint: add `#include <stdio.h>` before the directive
```

**Manifest** — `[package] links = "..."` for the native-dep
graph (V41D-3 / V41D-4):

```toml
[package]
name    = "openssl-sys"
version = "0.1.0"
links   = "ssl"
```

Two `openssl-sys` crates of different versions ending up in
the same dep graph fail to build with a Cargo-style
diagnostic — same wording as `cargo build` produces for a
duplicate `links` clash:

```
error: failed to select a version for `openssl-sys`.
  ... required by package `app v0.1.0`
versions that meet the requirements `^0.1` are: 0.1.0

the package `openssl-sys` v0.1.0 links to the native library `ssl`,
but the package `app v0.1.0` already links to `ssl` (via openssl-sys v0.2.0).
```

**CLI** — `cust bindgen` (V41D-6 / V41D-7):

```console
$ cust bindgen /usr/include/zstd.h -o src/zstd_raw.c --prefix=zstd_raw_
  Bindgen   /usr/include/zstd.h -> src/zstd_raw.c (47 types, 89 functions)
```

→ generated `src/zstd_raw.c` is checked in alongside the rest
of the crate's source. Re-run when the header changes; the
file is human-editable (handy for trimming the surface or
fixing the occasional printer quirk). No build-script
machinery required.

## Scope

In suggested implementation order. In-progress V41D-N decisions
appear below the scope list. Anything marked **❓ open** is a
question the v0.4.1 design discussion needs to lock before the
first slice ships; anything marked **🔒 locked** is settled and
just listed here for context.

1. **`[package] links = "..."` manifest field.** New optional
   string in `cust/src/manifest.rs`'s `PackageTable`. Parsed,
   validated (`[a-zA-Z][a-zA-Z0-9_-]*`), surfaced on
   `Manifest::package_links() -> Option<&str>`. V41D-3.

2. **Duplicate `links =` detection at workspace load.**
   `cust/src/workspace.rs` gains a post-resolve pass that
   walks the transitive dep graph and rejects with a
   Cargo-shape diagnostic when two distinct crates declare
   the same `links` value. Single-crate (non-workspace)
   builds are unaffected. V41D-4.

3. **`cust::requires` payload via `[[clang::annotate]]`.**
   The plugin's AST consumer walks every decl's `AnnotateAttr`
   list (already done for `getCustPubKind` etc.) and
   recognises payloads whose first string is `cust::requires`,
   collecting the remaining string args as required headers.
   No new `ParsedAttrInfo` recogniser registered (V40D-7
   five-name model is unchanged). Plugin records the
   requirement in the fragment header as a
   `/* @cust:requires <name> */` comment per requirement.
   V41D-2 / V41D-11 / V41D-13.

4. **Driver consumes `@cust:requires` from fragment headers.**
   `cust/src/build.rs` aggregates the `@cust:requires` lines
   from every concatenated fragment when emitting
   `<crate>.h`, deduplicates, emits a single leading
   `/* @cust:requires <name> */` block at the top of the
   published header. `cust/src/mod_scanner.rs` reads this
   block when lowering `#cust use <crate>;` and validates
   that every required system header is `#include`d in the
   importing TU before the directive. Missing prerequisites
   surface as the V41D-2 diagnostic with line:column. V41D-11.

5. **Docs: cust-design.md §5 — three producer patterns.** §5's
   "No `#include` injection" subsection grows a sub-section
   "Producer patterns for system types" naming Pattern 1
   (printer-desugared primitives — no producer action),
   Pattern 2 (raw opaque tag +
   `[[clang::annotate("cust::requires", "<header>")]]`),
   and
   Pattern 3 (shadow typedef — the recommended idiom).
   Cstd's `usize`/`i32` typedefs become the worked example
   for Pattern 3; a libc-wrapper example sketch covers
   Pattern 2. V41D-1 / V41D-2 / V41D-10.

6. **`cust bindgen` subcommand.** New entry in
   `cust/src/cli.rs` Subcommand enum + new
   `cust/src/bindgen.rs` module. Invokes clang via
   `-Xclang -ast-dump=json -fsyntax-only` on the input
   header, parses the JSON in Rust (no libclang FFI), emits a
   `.c` file with `[[cust::pub]] typedef` declarations and
   `[[cust::pub] extern` function prototypes. V41D-6 /
   V41D-7 / V41D-8.

7. **`cust bindgen` name-collision behaviour.** Hard error
   when an emitted decl name collides with a name already
   declared in the destination file (or in any sibling
   module file under `src/`). `--prefix=<prefix>` option
   prefixes every emitted decl name. `--allow-collision`
   suppresses the check (escape hatch; will surface as a
   linker error if you actually call into both). V41D-9.

8. **End-to-end dogfood.** New `cwork/examples/hello-zstd`
   demonstrates the full chain: `cust bindgen
   /usr/include/zstd.h` produces `src/zstd_raw.c`, the rest
   of `src/main.c` calls `zstd_raw_ZSTD_compress(...)`, the
   `Cust.toml` carries `links = "zstd"` + `[clang]
   extra-ldflags = ["-lzstd"]` (the link line itself stays in
   `extra-ldflags` until v0.4.2 ships build scripts; the
   `links =` field tags the dep for the duplicate-detection
   pass). cwork CI runs `cust run -p hello-zstd` and asserts
   the output. V41D-5 framing.

9. **Documentation.** §5 patches per (5) above; §9 attribute
   catalogue grows a `cust::requires` row (documenting the
   `[[clang::annotate("cust::requires", "<header>")]]`
   spelling per V41D-2 / V41D-13); §17 roadmap flips v0.4.1 from "registry + cust add" to
   "FFI & system types"; new §FFI sub-section (in §5 or a
   sibling) summarises the producer/consumer story. v0.4.0.md
   gains a "Subsequent history" callout flagging that the
   surface-pass typedef fix shipped as a v0.4.0 follow-up
   (commit `28f3b47`) and that Pattern 3 is now the
   recommended idiom (cross-link to v0.4.1.md).

## Decisions (V41D-N — locked)

### V41D-1 — Producer pattern for system primitive types?

**Locked: Pattern 3 (shadow typedef) is the recommended
idiom.** Documented in cust-design.md §5 as the official
guidance.

Three patterns are technically available:

* **Pattern 1 — print-through.** Producer writes `size_t` in
  its `[[cust::pub]]` signatures. Clang's pretty-printer
  desugars `size_t` to `unsigned long` (or whatever
  `__SIZE_TYPE__` resolves to on the target) and that's what
  lands in `<crate>.h`. Consumer needs no include. ABI is
  correct because the underlying primitive is the canonical
  resolution on every cust-supported platform.
* **Pattern 2 — raw opaque tag.** Producer writes
  `FILE *` (no typedef-aliasable underlying type). Printer
  preserves the typedef name. Consumer must `#include
  <stdio.h>` before the `#cust use`. Documented + diagnosed
  via V41D-2.
* **Pattern 3 — shadow typedef.** Producer declares a
  `[[cust::pub]] typedef __SIZE_TYPE__ <my_alias>;` (or
  similar `__*_TYPE__` builtin) and uses the alias in its
  public signatures. Consumer sees the alias in `<crate>.h`,
  needs no system include.

Pattern 3 wins as the *recommended* idiom because:

1. **Stable name in the published header.** Pattern 1 leaks
   the target's underlying primitive (`unsigned long` on
   x86-64 Linux, `unsigned long long` on Windows x64). The
   alias gives consumers a portable name.
2. **Survives the freestanding profile (§16 OQ-8).** When
   cust grows `freestanding = true`, Pattern 3 keeps working
   while Pattern 1 (which depends on `<stddef.h>`-style
   system typedefs being available to the consumer's compile)
   breaks for embedded targets.
3. **Producer needs no system include.** `__SIZE_TYPE__` /
   `__INT32_TYPE__` / etc. are clang builtins (independent of
   any system header). The producer's TU doesn't need to drag
   `<stdint.h>` into scope just to declare the alias.
4. **Composes with v0.4.0's surface-pass fix.** The
   `28f3b47` fix made cross-module typedef resolution
   ABI-honest. Pattern 3 is the user-facing payoff of that
   fix.

Pattern 1 is documented as "works without ceremony for the
quick prototype case, but stop using it the moment you care
about your published header reading nicely." Pattern 2 is
documented for the opaque-tag case (Pattern 3 can't handle
those — no `__FILE_TYPE__` builtin exists).

Alternative considered — "have cust auto-inject `#include
<stdint.h>` into `<crate>.h` for crates that use stdint
types" — rejected, restating §5: it silently couples every
cust crate to libc, breaks the freestanding profile, and
creates a hidden contract.

### V41D-2 — Producer pattern for opaque-tag system types?

**Locked: Pattern 2 (raw opaque tag) + payload-carrying
annotation via the built-in `[[clang::annotate("cust::requires",
"<header>")]]` spelling.** Producer writes the type literally
(`FILE *`, `pthread_t`, `struct stat`), annotates the decl
with `[[clang::annotate("cust::requires", "stdio.h")]]`,
plugin walks the resulting `AnnotateAttr` payloads in its
AST consumer and emits a machine-readable banner into the
fragment header, driver enforces the prerequisite at
`#cust use` lowering time.

> **Why this spelling, not `[[cust::requires("stdio.h")]]`?**
> The clean form was the first-draft design. Empirical
> testing on 2026-06-06 (clang 21.1.8, see V41D-13 below)
> proved it impossible to implement: clang's plugin-attribute
> parser strips arguments from `ParsedAttrInfo`-registered
> attributes, including string literals. The built-in
> `[[clang::annotate(...)]]` attribute uses a different
> internal parser path that *does* preserve string args, so
> riding on it is the only way to get payload-bearing
> annotations to the plugin from the C23 bracket syntax.
> Tradeoff: ugly UX (two attributes instead of one), no
> ergonomic improvement over the underlying clang built-in.
> If LLVM ever fixes the plugin parser we'll add the clean
> `[[cust::requires("stdio.h")]]` spelling as an alias.

The published `<crate>.h` is still pure declarations (§5's
no-`#include`-injection invariant survives intact). The
difference from today's behaviour is that the prereq is
**named**, **published in the header itself**, and
**checked by the driver**. Before v0.4.1 the consumer got:

```
error: unknown type name 'FILE'
```

— with no hint of what to include or which crate's surface
needed it. After v0.4.1:

```
error: `#cust use crate_foo;` requires `<stdio.h>` to be included
  --> src/main.c:3:1
   |
 3 | #cust use crate_foo;
   | ^^^^^^^^^^^^^^^^^^^^
   = note: crate `crate_foo` declares system-header prerequisites
   = hint: add `#include <stdio.h>` before the directive
```

The annotation can apply to:

* A function decl — requires apply to that function's
  signature.
* A typedef / record decl — requires apply to the type's
  body (e.g. a struct containing `FILE *`).
* A whole TU (via `[[clang::annotate("cust::requires",
  "stdio.h")]]` at file scope, which clang accepts as an
  empty declaration with the attribute attached) — every
  `[[cust::pub*]]` decl in this TU inherits the requirement.
  Shorthand for libc-wrapper crates where every exported
  function takes `FILE *`.

Multiple requirements stack two ways, both supported:

```c
// one call, multiple string args
[[clang::annotate("cust::requires", "stdio.h", "stdlib.h")]]
[[cust::pub]] void f(FILE *fp, void *buf);

// repeated annotations
[[clang::annotate("cust::requires", "stdio.h")]]
[[clang::annotate("cust::requires", "stdlib.h")]]
[[cust::pub]] void f(FILE *fp, void *buf);
```

Alternative considered — "wrap the opaque tag in a cust
handle (`typedef void cust_FILE; cust_FILE *crate_fopen(...)`)
so the published header is self-contained" — rejected
because it forces every consumer to go through wrapper
functions even when they want to interleave cust-wrapper
calls with raw libc calls on the same `FILE *`. The
annotation mechanism keeps the user in control.

Alternative considered — "infer the requirement from any
non-cust typedef in scope (opt-out via
`[[clang::annotate("cust::no_requires")]]`)" — rejected
because the plugin would need to walk include trees and
decide which includes "count" as system vs. user, which is
a heuristic mess. Opt-in is cleaner.

Alternative considered — "use a `#pragma cust requires
\"stdio.h\"`-style pragma instead of an attribute" —
rejected because pragmas have file-scope-only semantics in
practice; per-decl precision (the common case for
function-by-function library wrappers) needs the attribute
form. A pragma-shaped *aliasing* form for the TU-scope case
is on the v0.5+ wishlist if dogfooding shows the attribute
spelling is too verbose at file scope.

Alternative considered — "ship the design without
`[[cust::requires]]` entirely; document the bare 'consumer
includes the system header' rule from §5 and move on" —
rejected because the driver-checked diagnostic is the whole
point. Without it, the user contract is "read the upstream
crate's docs to figure out which `#include`s you need before
the `#cust use`" — i.e. the v0.4.0 status quo. The
annotation infrastructure is small (one banner format, one
deduplication pass, one diagnostic) and pays for itself.

#### Implementation contract

`plugin/src/plugin.cc` slice B:

1. The `CustASTConsumer` already walks the TU for
   `AnnotateAttr` payloads (`getCustPubKind` etc.). Extend
   it to recognise `AnnotateAttr` payloads starting with
   `cust::requires` and accumulate their string-arg lists.
2. Plugin v1 already emits a per-module fragment header
   (V40D-4 / V40D-5). Add a pre-decl banner block per
   V41D-11 listing each accumulated `cust::requires`
   header.
3. **No new `ParsedAttrInfo` recogniser registered for
   `cust::requires`.** It rides on the built-in
   `clang::annotate`. Five-name model (V40D-7) is unchanged.
4. The V40D-7 sentinel marker (`__cust_v40_marker__`)
   discipline does **not** apply to `cust::requires`
   payloads — user-written `[[clang::annotate("cust::pub")]]`
   is rejected by the sentinel guard precisely because
   anyone can write annotate; for `cust::requires` we *want*
   the user to write it directly, so no marker check.

Mandatory slice-B day-one regression test
(`plugin/test/test_requires_arg_survival`): fixture
declares `[[clang::annotate("cust::requires", "a.h",
"b.h", "c.h")]] [[cust::pub]] void f(void);`, runs the
plugin, asserts the fragment header contains exactly three
`/* @cust:requires <X> */` lines. If this test fails on
any clang version in the supported range (19, 20, 21) the
milestone halts and we fall back to V41D-13 option C
(per-header zero-arg attributes). RQ-V41-11 tracks the
contingency.

### V41D-13 — Why `[[cust::requires("...")]]` doesn't work — empirical record

**Locked: documented restriction. v0.4.1 ships using the
`[[clang::annotate(...)]]` workaround (see V41D-2).
Clean `[[cust::*("arg")]]` syntax remains impossible
until an upstream LLVM fix.**

This decision records the empirical finding that prompted
V41D-2's spelling change so it isn't re-discovered in
v0.4.x / v0.5.x.

#### Background

V40D-7 (v0.4.0 slice A) committed to a "five-name model"
(`cust::pub`, `cust::pub_crate`, `cust::pub_repr`,
`cust::test`, `cust::test_ignore`) because clang's C23
bracket-syntax parser silently dropped *identifier* args
(`crate`, `repr`) from plugin-registered attributes. The
v0.4.0 doc inferred that string-literal args would work
because they don't need scope binding.

#### The experiment

Tested 2026-06-06 against clang 21.1.8 with a probe build
of `plugin/src/plugin.cc` that logged `Attr.getNumArgs()`
from inside `handleDeclAttribute`. Five `ParsedAttrInfo`
configurations × five input shapes:

| Recogniser config | Input | `handleDeclAttribute` saw |
|---|---|---|
| `NumArgs=0, OptArgs=0` | `[[cust::pub]]` | NumArgs=0 (baseline) ✓ |
| `NumArgs=0, OptArgs=1` | `[[cust::pub(crate)]]` | NumArgs=0 — ident dropped |
| `NumArgs=0, OptArgs=1` | `[[cust::pub("crate")]]` | NumArgs=0 — **string dropped** |
| `NumArgs=0, OptArgs=15, AcceptsExprPack=1, NumArgMembers=15` | `[[cust::pub("a","b")]]` | NumArgs=0 — all knobs flipped, still 0 |
| `NumArgs=1, OptArgs=0` | `[[cust::pub("crate")]]` | clang error: `'cust::pub' attribute takes one argument` (saw 0) |
| `NumArgs=1, OptArgs=0` | `[[cust::pub]]` | Same error |
| GNU spelling not registered | `__attribute__((cust__pub("crate")))` | `unknown attribute` warning |

For comparison, the built-in `[[clang::annotate("payload",
"stdio.h", "stdlib.h")]]` (itself a `ParsedAttrInfo`
recogniser internal to clang) preserves all three string
args — visible in `-Xclang -ast-dump` as separate
`StringLiteral` nodes attached to the `AnnotateAttr`.

#### Conclusion

The v0.4.0 V40D-7 finding was understated. Clang 21's
plugin-attribute parser **strips every kind of argument**
from C++/C23 bracket-syntax attributes registered via the
public `ParsedAttrInfo` API — identifiers, string literals,
integer literals, all expressions. Built-in clang
attributes (including `clang::annotate`) reach the same
internal `AnnotateAttr` storage through a different parser
path that does preserve args; the plugin API doesn't grant
access to that path.

Root cause: confirmed by upstream as
[LLVM #45791](https://github.com/llvm/llvm-project/issues/45791)
("Attribute plugin example in trunk does not seem to
recognize attribute arguments"), filed by Jonathan Protzenko
on **2020-06-24** — i.e. the bug has been open for **almost
six years** at the time of this v0.4.1 lock. Confirmed by a
second party (label `confirmed`) and labelled `clang` /
`bugzilla`. The thread (15 comments through 2026-05) names
the exact code path:

> Looks like this is caused by this in
> `Parser::ParseCXX11AttributeArgs` in `ParseDeclCXX.cpp`:
>
> ```cpp
> // If the attribute isn't known, we will not attempt to parse any arguments.
> if (!hasAttribute(LO.CPlusPlus ? AttrSyntax::CXX : AttrSyntax::C,
>                   ScopeName, AttrName, getTargetInfo(), getLangOpts())) {
>   // Eat the left paren, then skip to the ending right paren.
>   ConsumeParen();
>   SkipUntil(tok::r_paren);
>   return false;
> }
> ```
>
> `hasAttribute` doesn't know about attributes added by
> plugins, so the arguments are skipped.
>
> — john-brawn-arm, 2020-07-08

Aaron Ballman (clang maintainer) acknowledged the bug
the same day and identified the underlying design tension:
C++11 bracket attributes have no GNU-style "always parse
as expression" rule, so without a parser-extension hook the
plugin can't be given the arg tokens. A
[draft patch by Protz](https://github.com/llvm/llvm-project/issues/45791#issuecomment-paste)
(2020-08-11, see issue activity) added a custom-parser hook
to `ParsedAttrInfo`; the patch did not land. A follow-up
review [D99861](https://reviews.llvm.org/D99861) took a
different approach (token-bag passed to the plugin); it too
was closed without merging. Most recent activity:
**2026-05** (TomMurray reports hitting the same issue six
years later). The bug remains open with no assignee, no
milestone, no recent patch.

This means v0.4.1's V41D-2 workaround is **not** routing
around a one-LLVM-release transient — it is the only
strategy available today, and based on the upstream history
it will remain so for the foreseeable future. The cust
project does not need to file a new issue; we should
**subscribe to #45791** and reference it from this
section, and from any future cust attribute that wants to
take arguments.

#### Options considered for v0.4.1

| # | Approach | Verdict |
|---|---|---|
| A | Ship V40D-7-style: one `[[cust::requires_<header>]]` zero-arg attribute per header name, decode the suffix. | Rejected. Doesn't scale — recognisers are explicit, can't catch-all; header names with `/`, `.` need mangling (`<sys/socket.h>` → `cust::requires_sys_socket_h`?); every new header is a plugin-code change. |
| B | Use `[[clang::annotate("cust::requires", "<header>")]]` directly; plugin walks `AnnotateAttr` payloads in the AST consumer. | **Selected.** Works today (string args survive), no new plugin recogniser, no upstream LLVM dependency. Tradeoff: verbose user-facing spelling. |
| C | `#pragma cust requires "<header>"` via `PragmaHandler`. | Rejected for v0.4.1: file-scope only (works for TU-level, not per-decl); fragments the design surface (some cust metadata as attributes, some as pragmas). Reconsidered for v0.5+ as a TU-scope shorthand if dogfooding warrants. |
| D | Drop `[[cust::requires]]` entirely; document the "consumer includes the system header" rule from §5 as-is. | Rejected. Removes the driver-checked diagnostic which is the whole user-facing value of v0.4.1's opaque-tag work. |
| E | Block v0.4.1 on an upstream LLVM patch fixing the plugin parser ([#45791](https://github.com/llvm/llvm-project/issues/45791)). | Rejected. Issue open ~6 years with multiple proposed patches abandoned; treating it as a hard blocker means cust v0.4.1 ships when (and if) someone lands a non-trivial change to clang's attribute parser. Not a meaningful timeline. |

#### Implications beyond v0.4.1

* **V40D-7 stands.** The five-name model is the practical
  answer for cust's own decl-annotation surface and
  remains correct.
* **v0.4.0's `[[cust::pub_crate]]` / `[[cust::pub_repr]]`
  spellings stay.** The v0.4.0 doc currently says "clang
  silently drops identifiers"; that wording is *not wrong*
  but is *less general* than the truth. A small v0.4.0
  doc patch (see Roadmap impact) records the broader
  finding.
* **Future `[[cust::*(arg)]]` attributes are blocked.** Any
  future attribute that wants payload (e.g. a hypothetical
  `[[cust::since("0.5")]]` or `[[cust::link("ssl", kind =
  "static")]]`) must use the `[[clang::annotate(...)]]`
  workaround until the LLVM parser is fixed. The
  `[[cust::link]]` row in §9 (catalogued but unimplemented)
  inherits this restriction — V40D-2's deferral list now
  has an "and they have to use clang::annotate when they
  land" caveat.
* **Upstream LLVM ticket.** v0.4.1 slice F includes a
  documentation step: add a backlink to LLVM
  [#45791](https://github.com/llvm/llvm-project/issues/45791)
  from the plugin source where the workaround actually
  lives (one `// See LLVM #45791 for why this rides on
  clang::annotate.` comment in `plugin/src/plugin.cc`).
  **No new LLVM issue filed by the cust project** — the
  existing one is the right place for any future upstream
  engagement. If/when a fix lands, v0.5+ can flip to the
  clean `[[cust::requires("stdio.h")]]` spelling via a
  plugin recogniser alias (the `clang::annotate` pathway
  stays as a back-compat layer for crates that encoded
  the workaround).

### V41D-3 — `[package] links = "..."` field shape?

**Locked: bare string, single value, optional.** Same as
Cargo:

```toml
[package]
name    = "openssl-sys"
version = "0.1.0"
links   = "ssl"            # OR: links = "ssl-3"
```

* Type: `Option<String>`. Single value (one crate links one
  native library, by convention).
* Validation: pattern `^[A-Za-z][A-Za-z0-9_-]*$`. Reject
  empty strings, leading digit, characters that wouldn't
  survive an `-l<name>` flag.
* No `links = ["a", "b"]` array form. A crate that genuinely
  wraps two native libs gets two cust crates (the Cargo
  pattern). Keeps the duplicate-detection invariant
  one-tag-per-crate.

Why a string vs. table: the table shape (`[links] name =
"ssl", version = "3"`) anticipates `pkg-config`-style version
constraints that v0.4.1 doesn't ship. Adding the table form
in v0.4.2+ alongside build scripts is additive and safe.
v0.4.1 ships the minimum.

The field is **metadata only** in v0.4.1 — it gates the
duplicate-detection pass (V41D-4) and the future build-script
`cust:cust-link-lib=` lines (v0.4.2), but doesn't itself add
anything to the link line. Users wanting to actually link
`-lssl` in v0.4.1 still use `[clang] extra-ldflags =
["-lssl"]` (see V41D-5).

### V41D-4 — Duplicate-link detection — when and how?

**Locked: workspace-load-time pass, hard error, Cargo-shape
diagnostic.**

`cust/src/workspace.rs` already resolves the transitive dep
graph for every workspace member. v0.4.1 adds a post-resolve
pass that walks every reachable crate, collects each
`(crate_name, package_links)` pair where `package_links` is
`Some`, and rejects with a clear diagnostic if any
`package_links` value appears twice with different
`crate_name` values.

Diagnostic shape (mirrors Cargo's wording — RQ-V41-1
records the verbatim text):

```
error: native library `ssl` is linked by multiple packages
  package `openssl-sys v0.1.0` links to `ssl`
  package `openssl-sys-fork v0.3.0` links to `ssl`
  = note: each native library may be linked by only one package
  = hint: choose a single openssl-sys variant in your dependency tree
```

Why workspace-load time (not link time): we want the error
*before* a single object file gets compiled — the wasted
work otherwise would be the whole build. Cargo does the
same thing.

Why hard error (not warning + first-wins): two crates each
calling `ssl_init()` from different `ssl` versions is an
ODR violation waiting to crash at runtime. Cargo's posture
is "this is the user's problem to resolve"; v0.4.1 inherits
that posture.

Single-crate (non-workspace) builds skip the pass — there
can't be a duplicate when there's only one crate. The pass
is a no-op for crates with no `links =` field anywhere in
their dep graph.

### V41D-5 — Link flags: manifest table vs. `[[cust::link]]` attribute?

**Locked: neither, in v0.4.1.** The existing `[clang]
extra-ldflags = [...]` field is the v0.4.1 escape hatch.
`[[cust::link("name", kind = "static")]]` (catalogued in
cust-design.md §9, never implemented) ships in v0.4.2
alongside build scripts.

Reasoning: actually generating link flags from declarative
metadata is the build-script job (cust-design.md §12).
Cargo's pattern is "`build.rs` emits `cargo:rustc-link-lib=`
lines; cargo aggregates and adds them to the rustc invocation."
The cust analogue (`cust:cust-link-lib=` lines from
`build.cust.c`) is on the v0.4.2 roadmap. Shipping a parallel
manifest-driven flag-generation path in v0.4.1 would mean
two mechanisms doing the same job for two milestones — a
documentation tax and a future-deprecation hazard.

`[clang] extra-ldflags = ["-lssl", "-lcrypto"]` keeps working
unchanged. Users wanting to link `-lzstd` in v0.4.1 use:

```toml
[package]
name    = "zstd-sys"
version = "0.1.0"
links   = "zstd"          # NEW — tags the dep

[clang]
extra-ldflags = ["-lzstd"]   # unchanged from v0.3.x
```

The `links = "zstd"` tag has effect — duplicate detection —
even though the actual `-lzstd` comes from `extra-ldflags`.
v0.4.2 will let users replace `extra-ldflags = ["-lzstd"]`
with a `cust:cust-link-lib=zstd` line in `build.cust.c` (or
a `[[cust::link]]` attribute on an extern decl).

### V41D-6 — `cust bindgen` invocation: standalone, build subcommand, or both?

**Locked: standalone subcommand only, in v0.4.1.** Build-time
integration deferred to v0.4.2.

```console
$ cust bindgen <header> -o <out.c> [--prefix=PREFIX] [--allow-collision]
```

* `<header>` is the input header path or `-` for stdin.
* `-o <out.c>` writes the generated cust source.
* `--prefix=PREFIX` prefixes every emitted decl name (typically
  the crate name + an underscore — e.g. `--prefix=zstd_raw_`).
* `--allow-collision` suppresses the V41D-9 name-collision
  check.

The generated file is checked into the user's repo (NOT
regenerated every build). Rationale: bindgen output is
sometimes wrong (printer quirks, macro-defined constants
that bindgen can't reach, platform-specific signatures);
having the file in version control means the user can patch
it. Cargo's `bindgen` ecosystem does the same thing for
crates that ship pre-generated bindings (`*-sys-rs`,
`openssl-sys`).

Build-time auto-regeneration (`build.cust.c` calls into
`cust bindgen` and writes to `OUT_DIR`) is the natural
follow-up in v0.4.2 once `build.cust.c` exists. The
standalone subcommand stays in v0.4.2+ — both paths will be
supported.

Alternative considered — "ship build-time only, no standalone
CLI" — rejected because v0.4.2 isn't shipped yet and we want
the bindgen ergonomics available for v0.4.1 users.

Alternative considered — "make bindgen part of `cust new
--bindings <header>`" — rejected because `cust new` is for
new crates, not for adding bindings to existing crates.
Keeps responsibilities separate.

### V41D-7 — `cust bindgen` output shape?

**Locked: single `.c` file, `[[cust::pub]] typedef` + extern
function prototypes, no bodies, no macros.**

Generated file template:

```c
/* @generated by `cust bindgen` v0.4.1 from /usr/include/zstd.h
 *
 * DO NOT EDIT by hand if you intend to re-run bindgen — your
 * changes will be lost. Otherwise this file is normal cust
 * source; feel free to trim or annotate.
 *
 * Source header:    /usr/include/zstd.h
 * Bindgen version:  cust 0.4.1
 * Generated:        2026-06-06 14:33 UTC
 * Prefix:           zstd_raw_
 */

[[cust::pub]] [[clang::annotate("cust::requires", "zstd.h")]]
typedef unsigned long zstd_raw_ZSTD_compressBound_arg;

[[cust::pub]] [[clang::annotate("cust::requires", "zstd.h")]]
typedef struct ZSTD_CCtx_s zstd_raw_ZSTD_CCtx;
/* ... */

[[cust::pub]] [[clang::annotate("cust::requires", "zstd.h")]]
extern unsigned long zstd_raw_ZSTD_compressBound(unsigned long src_size);

[[cust::pub]] [[clang::annotate("cust::requires", "zstd.h")]]
extern unsigned long zstd_raw_ZSTD_compress(
    void *dst, unsigned long dst_capacity,
    const void *src, unsigned long src_size,
    int compression_level);
/* ... */
```

* Every emitted decl carries `[[cust::pub]]
  [[clang::annotate("cust::requires", "<source-header>")]]`
  so consumers get the V41D-2 diagnostic if they forget the
  include. The two-attribute pairing is the V41D-13
  workaround for the plugin-arg parser limitation; bindgen
  hides the verbosity behind generated output, which is the
  largest single user of the spelling.
* Typedef shadows for every typedef in the input header
  (Pattern 3) — desugared if primitive-aliasable, kept
  opaque-tag otherwise.
* Function prototypes are `extern` because there's no body —
  the implementation lives in the linked native library.
* No bodies, no macro constants, no inline functions. v0.4.1
  bindgen is structural only. Macro-constant extraction
  (`#define ZSTD_VERSION_MAJOR 1`) is a v0.5+ enhancement —
  the cust analogue needs `[[cust::pub]] static const int
  zstd_raw_ZSTD_VERSION_MAJOR = 1;` which requires v0.4.0's
  pub-var support (shipped) but also needs macro
  introspection that the JSON AST dump doesn't expose.
* No anonymous-struct flattening, no bitfield packing —
  whatever the input header looks like, the output mirrors
  it. If clang's pretty-printer can't represent something
  (rare in C), bindgen emits a `/* @cust:bindgen-skip <reason>
  */` comment and continues.

The destination file path is the user's choice. By
convention `src/<libname>_raw.c` (mirrors Rust's `*-sys`
crate convention of one TU named `<lib>_raw.rs`).

### V41D-8 — `cust bindgen` implementation: libclang, external bindgen, or `-ast-dump=json`?

**Locked: clang's `-Xclang -ast-dump=json -fsyntax-only`
JSON.** Pure subprocess + JSON parsing. No libclang FFI, no
external `bindgen` Rust crate.

Reasoning:

1. **No new dependencies.** cust already depends on clang
   being on PATH. `-ast-dump=json` is a stable clang flag
   (since clang 8). Adds zero crates to `Cargo.toml`.
2. **No libclang version drift.** libclang's ABI changes
   across major LLVM versions; adopting it would tie cust to
   a specific LLVM major release. The JSON dump is
   forward-compatible (clang adds fields, never removes).
3. **The Rust `bindgen` crate generates Rust bindings.** It
   doesn't help us emit C source. The output transformation
   we want (clang AST → `[[cust::pub]] typedef` + extern
   protos in cust source) is small enough to write directly.
4. **Easier to debug.** A user reporting a bindgen issue can
   re-run `clang -Xclang -ast-dump=json -fsyntax-only
   <their-header>` themselves and look at the JSON.

Implementation sketch (`cust/src/bindgen.rs`):

```rust
fn bindgen(input: &Path, opts: &BindgenOpts) -> Result<String> {
    let json = clang_ast_dump_json(input)?;          // subprocess
    let ast: BindgenAst = serde_json::from_str(&json)?; // serde + jsonc-tolerant
    let decls = walk_top_level(&ast)?;                // filter pub-eligible decls
    let out = render_cust_source(&decls, opts);       // string template
    Ok(out)
}
```

JSON parsing uses `serde_json` (already in cust's transitive
dep tree via the `clap`/`anyhow` ecosystem? — RQ-V41-2
verifies; if not, it's a small additional dep, fine for a
CLI subcommand). The walker handles `FunctionDecl`,
`TypedefDecl`, `RecordDecl`, `EnumDecl`. Everything else
(macro definitions, `VarDecl`, `FieldDecl` nested in records
we don't export) is skipped with a `/* @cust:bindgen-skip */`
comment when it would have shown up in the public surface.

System headers (`#include`d in the input header transitively)
are walked but **not** emitted by default — bindgen filters
to decls whose source location is in `<input>` itself or a
header it brought in. The user can pass `--include-system`
to emit decls from system headers too (e.g. for a
`stdio-sys` crate that intentionally wraps `<stdio.h>`).

### V41D-9 — Name-collision behaviour in `cust bindgen` output?

**Locked: hard error by default, `--prefix=` is the
recommended workaround, `--allow-collision` is the escape
hatch.**

The check runs at output time: before writing
`<out.c>`, bindgen walks every other `.c` file under `src/`
in the destination crate, scans for `[[cust::pub*]]` decl
names (cheap regex — bindgen doesn't run the plugin), and
hard-errors when any emitted name matches:

```
error: `cust bindgen` would emit decl `ZSTD_compressBound` which collides with:
  - `src/util.c:42`: `[[cust::pub]] int ZSTD_compressBound(int);`
help: use `--prefix=zstd_raw_` to namespace the generated decls
help: or pass `--allow-collision` to suppress this check (you'll get a linker error if both are reachable)
```

Reasoning: cust has no symbol namespacing today (OQ-13, see
cust-design.md §6 + §16). Letting bindgen silently emit a
colliding name produces a confusing ld error at the next
build. Catching it at generation time costs nothing and
keeps the error close to the cause.

`--prefix=<prefix>` is the recommended workaround because
it both avoids the collision AND addresses OQ-13's
"manual crate-name prefix" convention (see cust-design.md
§6 Status subsection). The bindgen invocation typically
looks like `cust bindgen <header> -o src/<name>_raw.c
--prefix=<cratename>_raw_`.

### V41D-10 — Documentation home for the FFI patterns?

**Locked: cust-design.md §5 extension + new top-level
"§FFI" sub-section.**

§5 gets a "Producer patterns for system types" subsection
that names Patterns 1 / 2 / 3, with cstd's `usize` typedef
as the worked example for Pattern 3 and a sketched
libc-wrapper for Pattern 2.

A new §FFI sub-section (likely §5.5 to keep the numbering
local) describes:

* `[package] links = "..."` (V41D-3) + duplicate-detection
  guarantee (V41D-4).
* `[[clang::annotate("cust::requires", "<header>")]]`

### V41D-11 — `@cust:requires` banner format in fragment / published headers?

**Locked: machine-readable comment, deduplicated, leading
position.**

Plugin emits one line per requirement at the top of the
fragment header (after the existing `/* @generated by cust
plugin v1 */` banner):

```
/* @generated by cust plugin v1 — DO NOT EDIT */
/* Forward declarations of [[cust::pub]] items. */
/* @cust:requires <stdio.h> */
/* @cust:requires <stdlib.h> */

extern int crate_foo_open(FILE *fp);
extern void *crate_foo_alloc(size_t n);
```

Driver concatenation step (`<crate>.h` emit):

1. Read every `<module>.cust.h` in the crate.
2. Collect all `/* @cust:requires <name> */` lines, dedupe.
3. Emit them at the top of `<crate>.h`, between the
   `/* @generated */` banner and the first decl:

```c
/* @generated by cust — DO NOT EDIT */
/* Public surface of crate `crate_foo`. */
/* @cust:requires <stdio.h> */
/* @cust:requires <stdlib.h> */

#ifndef CRATE_FOO_H
#define CRATE_FOO_H
/* ... */
```

The comment format is deliberate:

* **Leading `@cust:` namespace** — distinguishes from
  human-readable comments, easy to grep, no collision with
  doxygen / clang-format markers.
* **Header name in angle brackets** (`<stdio.h>`) — matches
  how the user writes the `#include`. Driver compares
  *literally* — `<stdio.h>` ≠ `"stdio.h"`, `<sys/socket.h>`
  ≠ `<socket.h>`.
* **One requirement per line** — deduplication is a
  textual set-union, no parsing involved.
* **Comments not pragmas** — pragmas would interact with
  clang's pragma handling (whitespace sensitivity, vendor
  prefixes); a `/* ... */` comment is inert in every
  pipeline.

The driver's `mod_scanner` lowering of `#cust use <crate>;`
reads the `<crate>.h`'s `@cust:requires` block and verifies
each requirement appears as an `#include` earlier in the
TU. Missing prereqs produce the V41D-2 diagnostic with
`line:column`. The check is at directive-lowering time
(not link time) so the error is close to the cause.

### V41D-12 — `cust::requires` propagation through `[[cust::pub_repr]]` body export?

**Locked: yes, recursively, with the requirement attached
to the struct/enum decl.**

A `[[cust::pub_repr]] struct foo { FILE *fp; … };` exports
the body into the fragment header. If the body mentions
`FILE *`, the struct declaration *itself* needs `<stdio.h>`
visible to consumers. The plugin enforces this:

* If the user writes
  `[[cust::pub_repr]] [[clang::annotate("cust::requires",
  "stdio.h")]] struct foo { FILE *fp; };`, the requirement
  propagates normally.
* If the user writes `[[cust::pub_repr]] struct foo { FILE
  *fp; };` *without* the annotation, the plugin walks the
  struct body, detects the use of a typedef from a system
  header (heuristic: the typedef's source location is in
  `/usr/include/` or `/usr/local/include/`), and **warns**
  (not errors — opt-in is the v0.4.1 posture) with:

```
warning: `[[cust::pub_repr]] struct foo` field `fp` has type `FILE *` from `<stdio.h>`
  hint: add `[[clang::annotate("cust::requires", "stdio.h")]]` so consumers know what to include
```

`-Werror=cust-missing-requires` upgrades the warning to a
hard error. v0.5+ may flip the default once dogfooding
shows the pattern is stable.

The walk is one level deep: it doesn't recursively peer
through nested typedefs from non-system headers. (A struct
field of type `mycrate_handle` doesn't get flagged even if
`mycrate_handle` itself transitively uses `FILE *` — that's
`mycrate`'s `cust::requires` annotation responsibility.)

## Open questions (RQ-V41-N — locked)

These are the smaller "lock the wording, then move on"
questions surfaced during design. Each carries the locked
answer plus a one-line rationale.

* **RQ-V41-1 — Verbatim text of the V41D-4 duplicate-`links`
  diagnostic?** Locked: see V41D-4 above (mirror of Cargo's
  E0469-ish wording with cust-flavoured field names). The
  literal string is checked against by integration test (3)
  via `assert_failure_with`.

* **RQ-V41-2 — Does cust already transitively depend on
  `serde_json`?** Locked: verified at slice E start. If
  yes, use it directly. If no, add as a direct dep with the
  `default-features = false` minimal feature set. Either
  way, ride the existing serde version cust already pulls
  in (no version bump).

* **RQ-V41-3 — Does `cust bindgen` invoke the user's
  configured clang, or the cust driver's clang?** Locked:
  the user's configured clang — same resolution path as
  `cust build` (`CC` env var, `[clang] path` in `Cust.toml`,
  PATH search). `cust bindgen --clang=<path>` overrides.
  Reason: bindgen output is a function of the clang version
  used to introspect; using the user's clang keeps the
  output reproducible across machines.

* **RQ-V41-4 — Comment format: `/* @cust:requires <h> */`
  or `// @cust:requires <h>`?** Locked: block comments
  (`/* ... */`). C source uses block comments throughout
  the cust ecosystem; line comments (`//`) are technically
  C99+ but less universal in old-school C. No correctness
  difference; consistency wins.

* **RQ-V41-5 — Does the `cust::requires` annotation need
  to compose with `[[cust::pub_macro]]`?** Locked: no —
  `pub_macro` is deferred (V40D-13 from v0.4.0, no
  extractor yet). When `pub_macro` ships in v0.5+, it'll
  pick up `cust::requires` naturally because it's just
  another `AnnotateAttr` on the macro decl.

* **RQ-V41-6 — `cust::requires` annotation on a non-pub
  decl?** Locked: silently accepted, no effect (the
  requirement doesn't propagate to anything consumers see).
  Future warning is fine; v0.4.1 doesn't diagnose.

* **RQ-V41-7 — Does the V41D-12 walk through `typedef`
  chains?** Locked: one level deep. `typedef FILE *foo;`
  in a system header propagates the requirement
  (`<stdio.h>`); a user-crate typedef defined in a non-
  system header is treated as user-owned (the producer is
  responsible for marking the typedef itself with the
  `cust::requires` annotation if needed).

* **RQ-V41-8 — Does the duplicate-`links` check care about
  the version specifier?** Locked: no. Two crates both
  named `openssl-sys` at different versions still produce
  the V41D-4 error if both end up in the resolved graph.
  This matches Cargo's behaviour and protects against the
  multi-version-of-`-lssl` ODR hazard.

* **RQ-V41-9 — Does `cust bindgen` honour the input
  header's `#cust use` directives?** Locked: no. cust's
  `#cust mod` / `#cust use` directives only run in the
  driver pre-pass (cust-design.md §4); a header processed
  by clang via `-ast-dump=json` would have the directives
  passed straight through and rejected as syntax errors.
  Bindgen is for *system / third-party* headers, which
  don't contain `#cust` directives in any case. Documented
  as a restriction in the `cust bindgen` man page.

* **RQ-V41-10 — Does `cust bindgen` re-write the
  destination file's existing contents, or fail if it
  exists?** Locked: re-writes with `--force` flag (default:
  fails with "destination file exists; pass --force to
  overwrite"). Matches `cust new`'s posture.

* **RQ-V41-11 — Contingency if the `[[clang::annotate(...)]]`
  workaround stops working.** Locked: fall back to one
  zero-arg `ParsedAttrInfo`-registered attribute per header
  (`[[cust::requires_stdio_h]] [[cust::requires_stdlib_h]]`)
  with the header name decoded from the suffix. Ugly,
  doesn't scale to arbitrary header names, and would limit
  the user to a fixed alphabet (see V41D-13 option A);
  but it fits the V40D-7 envelope and the plugin owns the
  decoding. The mandatory slice-B contract test
  (verification (4) above, `test_requires_arg_survival`)
  is the trigger; failing the test pauses the slice and
  re-opens V41D-2.

  Confidence today (2026-06-06): high. The built-in
  `[[clang::annotate(...)]]` recogniser is part of clang's
  permanent attribute surface; an LLVM minor version that
  silently changed its behaviour would break a far larger
  ecosystem than cust. The residual risk is a clang major-
  release rewrite of attribute parsing — in which case
  every plugin in the world rewrites simultaneously, and
  the per-header fallback is one of several options we'd
  reconsider then.

## Risks

* **LLVM plugin-attribute parser is buggy and unfixed.**
  V41D-13 records the empirical finding and identifies it
  as upstream issue
  [#45791](https://github.com/llvm/llvm-project/issues/45791)
  (open since 2020-06-24, ~6 years at v0.4.1 lock time,
  two proposed patches abandoned, most recent independent
  bump 2026-05). v0.4.1 works around it via
  `[[clang::annotate(...)]]`. The risk is that the
  workaround stops working too — unlikely, since
  `clang::annotate` is permanent surface used by
  thousands of downstream projects. Mitigation: the
  mandatory slice-B day-one regression test (verification
  item 4) catches a workaround regression on the first
  build of every supported clang version; failure pauses
  the slice and triggers V41D-13 option A (per-header
  zero-arg attributes, ugly but functional).
* **User-written `[[clang::annotate("cust::requires",
  ...)]]` is ugly and easy to misspell.** The V40D-7
  sentinel-marker discipline catches `[[clang::annotate(
  "cust::pub")]]` impersonations precisely because *any*
  user could write that string. For `cust::requires` we
  opted out of the sentinel check (V41D-2 implementation
  contract) because there's no malicious-use angle (worst
  case: extra header in the diagnostic), but typos
  (`cust::require`, `cust::Requires`, `cust:requires`) are
  silently ignored. Mitigation: in slice B, the plugin
  warns on `[[clang::annotate("cust::*", ...)]]` payloads
  starting with `cust::` that don't match a known prefix
  (`cust::requires`, `cust::pub`, `cust::pub_crate`, etc.).
  Catches the common typos at compile time.
* **`-ast-dump=json` schema drift.** Clang's JSON AST format
  is "stable" in the colloquial sense but isn't formally
  versioned. A clang minor-version bump that renames a key
  silently breaks bindgen. Mitigation: bindgen treats
  unknown keys as ignorable, hard-errors only on missing
  required keys (`kind`, `loc`, `inner`). Pin a
  CI matrix of clang versions (19, 20, 21) to catch breaks
  early.
* **`cust::requires` heuristic for V41D-12 false
  positives.** The "system header" detection (file path
  matches `/usr/include/` etc.) is OS-dependent. macOS uses
  `/Library/Developer/CommandLineTools/SDKs/.../usr/include/`,
  Windows uses MSVC's `INCLUDE` env var. v0.4.1 ships
  Linux-only detection (matches v0.4.0's Linux-only stance
  per cust-design.md §1). Cross-platform detection is a
  v0.5+ enhancement, tracked separately.
* **`cust bindgen` on `<stdint.h>` produces an unexpected
  flood of decls.** Clang's `<stdint.h>` transitively
  includes `<bits/...>` headers full of internal-only
  declarations. Default behaviour (only emit decls whose
  source location is the input header file itself, not
  transitively-included files) keeps the output small. The
  `--include-system` flag exists for the rare case where
  the user wants the full transitive surface.
* **Workspace duplicate-`links` check has a false-negative
  hole.** A crate whose `Cust.toml` has no `links =` line
  but whose `[clang] extra-ldflags` includes `-lssl`
  bypasses the check. v0.4.1 doesn't parse `extra-ldflags`
  for `-l` flags (too brittle — flags can be hidden inside
  `pkg-config` outputs, env-var expansions, etc.).
  Documented: `links =` is the cust contract for "I link
  this native library"; `extra-ldflags` is the escape
  hatch and the user is on their own for collisions there.
  v0.4.2's `cust:cust-link-lib=` from build scripts will
  participate in the same check.

## Backward compatibility

v0.4.1 is purely additive at the user contract level:

* `[package] links = "..."` is a new optional field. Crates
  without it behave identically to today.
* `[[clang::annotate("cust::requires", "<header>")]]` is
  a new *recognised payload* of the existing built-in
  `clang::annotate` attribute. Decls without it behave
  identically to today (the `@cust:requires` banner
  is absent from their fragment header → no driver
  enforcement). No new attribute is registered with clang;
  see V41D-2 / V41D-13 for the spelling rationale.
* `cust bindgen` is a new subcommand. No existing
  subcommands change shape.
* cwork's cstd has no Pattern 2 use → cstd's published
  `<cstd.h>` is byte-identical to its v0.4.0 form (verification
  target 21).

The one behavioural change is the V41D-2 diagnostic for
crates that DID rely on opaque-tag types without
`[[clang::annotate("cust::requires", ...)]]` — they shipped
today by virtue of the consumer happening to already
`#include <stdio.h>` for their own reasons; in v0.4.1 the
producer needs to add the annotation. This is a docs / migration note in v0.4.1's
release notes; no automated migration tool ships (the
annotation is one line, the warning V41D-12 emits hints at
it).

The fragment header byte format gains the `@cust:requires`
banner lines for crates that use the new annotation. This
breaks the "byte-identical" stamping for those specific
fragments, but only on the first build after upgrading
(subsequent rebuilds re-stamp). Documented as a one-time
rebuild cost.
