# cust v0.3.1 — binary crates & `cust run`

**Status:** ready for implementation (2026-06-04).
**Parent doc:** [cust-design.md §17 roadmap](cust-design.md).
**Previous milestone:** [v0.3.md](v0.3.md) (shipped).

This file is the v0.3.1 record — scope, locked design decisions
(V31D-N), and the verification target. A small single-purpose
milestone wedged between v0.3 (workspaces & path deps) and v0.4
(registry / tests / plugin v1 / build scripts).

### Why a 0.3.1, not part of v0.4

* The change is mechanical: add a `CrateKind::Bin` codepath through
  manifest + module discovery + build + workspace + cli. No new
  contracts, no plugin work, no network code.
* v0.4 already carries five big-ticket items (registry, `cust add`,
  semver, build scripts, plugin v1, fork-isolated `[[cust::test]]`).
  Bolting bins onto v0.4 doubles that milestone.
* Bin crates as a discrete milestone gives us a working dogfood
  story (`cust run -p hello-cstd`) *before* any v0.4 risk lands.
* The contracts it introduces (`[[bin]]` table shape, executable
  output path, `cust run` arg-forwarding) are independently
  versionable and don't interact with the v0.4 registry/lock work.

### Headline outcome

```sh
cd cwork && cust run -p hello-cstd
   Compiling cstd
   Compiling hello-cstd
    Finished hello-cstd [dev]
     Running cwork/target/debug/hello-cstd
hello from cstd v0.3.0: max(3, 7) = 7
```

Plus the corresponding scaffolder:

```sh
cust new --bin examples/foo
```

## Scope

In suggested implementation order. Locked V31D-N decisions appear
below the scope list.

1. **Crate kind detection.** Auto-infer from disk: `src/lib.c` →
   has lib, `src/main.c` → has bin, both → lib+bin, neither →
   existing "library source not found" error. Optional `[[bin]]`
   table for overrides (single entry only — multi-bin via
   `src/bin/*.c` is **V31D-3** deferred). See **V31D-1** and
   **V31D-2** for the precise rules.

2. **Manifest validation.** Reject explicit `[lib] crate-type` on
   a bin-only crate (only meaningful for libs). Reject `[[bin]]`
   when no `src/main.c` exists (or the configured `[[bin]].path`).
   `cdylib` / multiple `crate-type` entries stay rejected (v0.4+).

3. **Bin codegen path.** Same module-discovery + per-TU compile
   pipeline as libs (the existing `mod_scanner` / `modules` /
   per-TU compile loop is kind-agnostic), rooted at `src/main.c`
   instead of `src/lib.c`. Step 6 (archive) is replaced for the
   bin path with a clang **link** invocation producing
   `target/<profile>/<crate-name>` (V31D-4). Lib+bin crates build
   the lib archive first, then link the bin against it + any dep
   archives.

4. **Workspace edges.** Bins are first-class workspace members.
   Bins can depend on lib members via `[dependencies]` and `#cust
   use <name>;`; lib → bin and bin → bin edges are rejected at
   edge-resolution time (V31D-6). Topological ordering treats bins
   exactly like libs.

5. **`cust run`.** New subcommand: `cust run [-p <member>]
   [--release] [-- <args>...]`. Builds first, then spawns the
   resulting executable with anything after `--` as argv. Exits
   with the subprocess's exit code. `cust run -p <lib-member>` is
   a clear error (V31D-7, V31D-8).

6. **`cust check` for bins.** Same `-fsyntax-only` pass as libs.
   No link step. Cheap. (V31D-9.)

7. **`cust build -p <bin>`.** Already works once slices A+B land;
   verified in a test rather than newly implemented (V31D-8).

8. **`cust new --bin`.** Finally lights up the reserved `--bin`
   flag in [`cust/src/new.rs`](../../cust/src/new.rs). Emits
   `src/main.c` with `int cust_main(void) { ... }` returning 0
   and a minimal printf. (V31D-11.)

9. **Lock format unchanged.** `Cust.lock` v1 records member name /
   version / path / deps. The bin-vs-lib distinction is a
   property of the member's `Cust.toml`, not the resolved graph.
   No schema bump. (V31D-10.)

## Decisions (V31D-N — locked)

### V31D-1 — How is "this is a bin crate" declared?

**Locked: A. Auto-infer from `src/main.c`** (Cargo parity), with
optional `[[bin]]` table for path overrides. Concretely:

| `src/lib.c` | `src/main.c` | `[lib]` | `[[bin]]` | Result |
|---|---|---|---|---|
| present | absent | any | absent | **lib-only** (v0.3 behaviour) |
| absent | present | absent | absent | **bin-only** |
| present | present | any | absent | **lib+bin** |
| any | any | any | present (path override) | bin path determined by `[[bin]] path` |
| absent | absent | absent | absent | **error** (existing "library source not found") |
| any | absent | any | present (path override, file missing) | **error** with the resolved path |

Cargo's `[[bin]]` is an array; ours is a single-entry table in
v0.3.1 (V31D-3 defers multi-bin). The accepted subkeys are `path`
only — `name` is always derived from the package name.

### V31D-2 — Can a crate be both lib + bin?

**Locked: A. Yes.** `src/lib.c` AND `src/main.c` is the natural
Cargo shape; the bin links against its own lib. Build order
within the crate is lib first (produces `lib<name>.a`), then bin
linking that archive + any dep archives.

**Intra-crate self-import syntax (Cargo parity).** The bin half
reaches the lib half's `[[cust::pub]]` surface with the *same*
syntax it would use for an external dep:

```c
// src/main.c in a lib+bin crate named `demo`
#cust use demo;

cust_pub int cust_main(void) {
    return demo_answer();   // declared cust_pub in src/lib.c
}
```

This mirrors Rust's `use my_crate::*;` in `src/main.rs`. The
driver lowers `#cust use <own-package-name>;` in the bin half to
an `#include` of the local `<crate>.h` header (rather than the
dep-symlink-farm path that an external `#cust use` would resolve
through). Mechanism:

* The lib half always produces
  `target/<profile>/build/<crate>/include/<crate>.h` from the
  concatenated module fragments (cust-design.md §5).
* The bin half is compiled with `-I<that-include-dir>` (so a
  raw `#include "<crate>.h"` would also work, but isn't the
  canonical form).
* The build pipeline carves out `name == own_package_name` from
  the `#cust use <name>;` "must be a declared dependency" check
  whenever the kind is `LibAndBin` and the module being
  compiled is in the bin half.

The carve-out is **bin-half only**: writing `#cust use <self>;`
in `src/lib.c` is rejected with the standard "not listed in
[dependencies]" error. Self-importing a library is meaningless
(there's no sibling crate to import from), and the bin-half
specific lowering path doesn't apply there.

### V31D-3 — Multiple bins per crate?

**Locked: B. No in v0.3.1.** Single `src/main.c` (or `[[bin]] path
= "..."`) only. Cargo's `src/bin/*.c` directory convention and
multi-entry `[[bin]]` array stay v0.4+. Rationale: every example
this milestone unlocks is single-bin, and the multi-bin shape
opens schema design questions (per-bin `[dependencies]`?
per-bin `[clang]`?) that don't have to be answered now.

### V31D-4 — Bin output location?

**Locked: A. `target/<profile>/<crate-name>`** — sibling to the
existing `target/<profile>/build/<name>/lib<name>.a`. Matches
Cargo, which writes `target/debug/<name>` not
`target/debug/bin/<name>`. `cust run` knows where to find it.
Alternative considered: `target/<profile>/bin/<name>` — rejected
for breaking Cargo muscle memory with no upside.

### V31D-5 — `crate-type` validation?

**Locked: A. Reject explicit `crate-type` on bin crates entirely
in v0.3.1.** `crate-type` only makes sense for libraries
(`staticlib` today, `cdylib` v0.4+). A bin's output kind is
implicit. Error message:

```
error: `[lib] crate-type` set on bin-only crate `<name>` —
  remove the `[lib]` table or add `src/lib.c` to make it a
  lib+bin crate
```

### V31D-6 — Workspace edges involving bins?

**Locked: A. lib → bin and bin → bin edges are rejected at
edge-resolution time.** Only bin → lib and lib → lib edges are
permitted. Rationale:

* **lib → bin** would mean a library depending on an executable —
  meaningless; there's no surface to import.
* **bin → bin** is Cargo-disallowed too. If app A wants to call
  app B, the right answer is to extract the shared code into a
  lib member that both bins depend on.

Error message:

```
error: workspace member `<name>` (bin) cannot be a dependency
  — only library members may appear in [dependencies]
  hint: extract the shared code into a separate lib member
```

### V31D-7 — `cust run` arg forwarding?

**Locked: A. `cust run [-p <member>] [--release] [-- <args>...]`**
matching Cargo. The `--` separator is required to pass argv to
the spawned binary; anything before `--` is parsed as cust flags.

```sh
cust run -p hello-cstd                 # no args
cust run -p hello-cstd -- --verbose    # passes --verbose to hello-cstd
cust run --release -- arg1 arg2        # release build, two argv
```

Stdin is inherited (the spawned binary can read from the
terminal). Stdout/stderr are inherited. The cust process exits
with the subprocess's exit code; signals are not currently
forwarded (v0.5+).

### V31D-8 — `-p / --package` on bin operations?

**Locked: A.** `cust build -p <bin>` works (same machinery as for
libs — the orchestrator only cares about member names + transitive
deps). `cust run -p <bin>` works. `cust run -p <lib-member>` is a
clear error:

```
error: workspace member `<name>` is a library — `cust run`
  requires a binary crate
```

`cust run` with no `-p` requires the workspace to contain exactly
one bin member; otherwise:

```
error: workspace contains multiple binary members; pass
  `-p <name>` to choose one (found: app, server, tool)
```

(Same shape as Cargo's "could not determine which binary to
run".)

### V31D-9 — What does `cust check` do for bins?

**Locked: A. Same as lib** — `-fsyntax-only` pass on every TU; no
link step. The orchestrator already runs check per member; bin
members just become another kind of member to check.

### V31D-10 — Does the lock change?

**Locked: B. No.** `Cust.lock` v1 records member name / version /
path / deps. The bin-vs-lib distinction is a property of the
member's `Cust.toml`, not the resolved graph. No schema bump, no
spec doc update.

### V31D-11 — `cust new --bin` scaffolder?

**Locked: A. Yes.** Lights up the reserved `--bin` flag in
[`cust/src/new.rs`](../../cust/src/new.rs). Generates:

```
<path>/
├── .gitignore        # /target
├── Cust.toml         # [package] name + version, no [[bin]] (auto-inferred)
└── src/
    └── main.c        # int cust_main(void) { ... return 0; }
```

The `cust_main` name is deliberate — keeps a stable entry-point
seam if we ever decide to wrap `main` in a runtime (signal
handling, panic install, etc.). For v0.3.1 the cust prelude
defines `cust_main` as a plain alias for `main`.

## Deferrals — what is NOT in v0.3.1

| Topic | Owner |
|---|---|
| Multiple bins per crate (`src/bin/*.c`, `[[bin]]` array) | v0.4 |
| `cdylib` output | v0.4+ |
| `[[example]]` / `tests/` / `benches/` directory conventions | v0.4 (with `[[cust::test]]`) |
| Per-bin features / per-bin `[clang]` overrides | v0.4+ |
| `cust install` (copy executable to `~/.cust/bin/`) | v0.5+ |
| `cust run` ENV plumbing for build-script-set vars | v0.4 (with build scripts) |
| Signal forwarding to the spawned subprocess | v0.5+ |
| `cust run --example <name>` | v0.4 (with examples convention) |

## Verification target

Each bullet maps to an integration test that lands in the slice
that owns it.

* [ ] Single bin crate (`src/main.c` only) builds and runs.
* [ ] Lib+bin crate builds both artifacts; bin links against own lib.
* [ ] Bin crate path-deps on a lib member; `cust run` executes
      against the symlink-farm-published archive.
* [ ] Bin crate uses `#cust use <dep>;` and the dep's
      `[[cust::pub]]` surface.
* [ ] `cust run` with no args.
* [ ] `cust run -- arg1 arg2` forwards argv.
* [ ] `cust run --release` builds and runs the release executable.
* [ ] `cust run -p <bin>` in a multi-member workspace.
* [ ] `cust run -p <lib-member>` is a clear error.
* [ ] `cust run` in a no-bin workspace is a clear error.
* [ ] `cust run` in a multi-bin workspace without `-p` is a clear error.
* [ ] `cust check` runs syntax-only on bin TUs (no link).
* [ ] `cust build -p <bin>` builds the bin + its transitive deps.
* [ ] `cust new --bin <path>` scaffolds a buildable bin crate.
* [ ] lib → bin and bin → bin edges are rejected at edge-resolution.
* [ ] `[lib] crate-type` on a bin-only crate is rejected.
* [ ] `[[bin]] path = "..."` pointing at a missing file is rejected.

Total target: existing 83 unit + 43 integration = 126 tests, plus
roughly +15 unit and +12 integration for v0.3.1 → ~98 unit + ~55
integration ≈ 153 tests.

`cargo clippy --all-targets` stays silent under the workspace
pedantic+nursery profile.
