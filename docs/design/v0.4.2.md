# cust v0.4.2 — CMake-driven build backend

**Status:** ✅ **shipped** (closed 2026-06-11). Opened
2026-06-11; designed, implemented, and closed in one cadence.
**Parent doc:** [cust-design.md §17 roadmap](cust-design.md).
**Previous milestone:** [v0.4.0.md](v0.4.0.md) (shipped); v0.4.1 (⏸️ deferred).
**Supersedes:** the "CMake is never a build-time dependency"
rule in [cmake-and-portability.md](cmake-and-portability.md) §1 /
§2.2 / §2.3 / §5 (see superseded banner in that file).
**Minimum clang:** 19 (unchanged from v0.4.0).
**Minimum CMake:** 3.21 (matches the export-doc floor in
[cmake-and-portability.md](cmake-and-portability.md) §7).
**Minimum Ninja:** any (no minimum exercised; the generator
features we use are all 1.10-era).

This file is the v0.4.2 record — scope, in-progress design
decisions (**V42D-N**), open questions, and the verification
target. v0.4.2 replaces the direct-clang invocation path in
`cust/src/build.rs` (`compile_one_module`, `compile_tree`, the
phase-2 codegen + link steps) with a generated `CMakeLists.txt`
that cust writes into the build tree and drives via
`cmake -G Ninja` + `cmake --build`.

The previously-planned v0.4.2 scope (build scripts, `build.cust.c`
with the §12 hang-protection timeout) is **dropped from the
v0.4.x ordered line** and pushed back behind v0.4.1 (FFI), see
"Roadmap impact" below.

### Why CMake now — the reversal

[cmake-and-portability.md](cmake-and-portability.md) argues at
length (§2.2, §2.3, §5 rows 1–2) that CMake should never be in
the build hot path: the *long-term* answer was "emit Ninja
directly from the in-memory build graph cust already maintains,
skip CMake entirely." That position is preserved as the long-term
direction in [cmake-and-portability.md](cmake-and-portability.md);
v0.4.2 reverses it for the *near term* for four reasons that
have hardened during v0.4.0 dogfooding:

1. **The direct-Ninja-emitter is undated and unstaffed.** §2.3 of
   the original doc budgeted "a few hundred lines for a minimal
   emitter, 1000–2000 lines with tests for a production one." That
   budget is real, the emitter has no slice in the v0.4.x ordered
   line, and every milestone after v0.4.0 has produced a fresh ask
   that touches the build pipeline (v0.4.1 FFI: `links =`
   metadata, system-header banners; v0.4.3: `-jN`; v0.4.4:
   multi-bin; v0.4.5: integration tests, per-test timeouts). The
   in-house scheduler we already have (`cust/src/build.rs`
   `compile_tree` / `compile_one_module`) is single-threaded,
   has no parallelism story, and grows new ad-hoc rules every
   milestone. Shipping each one against a direct-Ninja emitter
   we haven't started is more work than shipping it against
   CMake we already know how to drive.
2. **The `-jN` problem isn't actually a `-jN` problem.** v0.4.3
   was scoped as "parallelism within and across crates." With a
   CMake/Ninja backend and one workspace-level CMakeLists
   (V42D-13), both kinds of parallelism collapse into Ninja's
   single scheduler — `cust build -j N` lowers to `cmake
   --build -j N` and Ninja makes one set of decisions over the
   union of every member's TUs. v0.4.3 becomes "pass the flag
   through" rather than "write a scheduler that handles depfile
   parsing + work stealing + path-rewriting." This is the
   largest concrete win from the reversal.
3. **`compile_commands.json` quality.** The driver currently
   synthesises it by hand in `cust/src/build.rs` line ~163, and
   it omits the per-TU `OBJECT_DEPENDS`-equivalent fragment-header
   edges (cust-design.md §4 "Incremental compilation"). Routing
   compilation through CMake means `CMAKE_EXPORT_COMPILE_COMMANDS=ON`
   produces the canonical file for free, and clangd / clang-tidy
   pick up *exactly* the flags the build uses.
4. **Diagnostics layering — actually fine.** §2.3 row 4 of the
   original doc worried that "clang error → ninja line → CMake
   script line → user" would be worse than direct invocation.
   In practice (verified against cwork by hand on 2026-06-10),
   CMake + Ninja errors that originate in user `.c` files show
   the clang diagnostic *first* and the Ninja "command failed"
   note *last*; the CMake-script layer doesn't appear because the
   generated `CMakeLists.txt` has no user-edited content for an
   error to trace back to. The worry was theoretical and the
   reality is "clang's output is on top, ninja's is at the
   bottom, no CMake noise." Acceptable.

The original counterarguments that the reversal **explicitly
accepts**:

* **Configure-step latency** (§2.3 of the original doc, 0.5–2s).
  v0.4.2 mitigates it (V42D-8: stamp-based configure-skip when the
  generated `CMakeLists.txt` is byte-identical and the source list
  hasn't changed). Worst-case "edit `Cust.toml`, re-run `cust
  build`" pays the configure cost once. Worst-case "edit one .c
  file, re-run `cust build`" pays nothing — Ninja takes over.
* **CMake becomes a runtime dep of cust.** Accepted. v0.4.2 looks
  for `cmake` and `ninja` on `PATH` and errors with a clear
  install hint if either is missing (V42D-10, V42D-11). The
  "vendor a pinned Ninja under `~/.cust/toolchain/`" question
  from the original doc §6 stays open and is *not* in v0.4.2
  scope.
* **Two sources of truth (`Cust.toml` vs `CMakeLists.txt`).**
  Avoided by construction: the generated `CMakeLists.txt` lives
  under `target/` (V42D-4), is regenerated every build, is
  gitignored, and is **not** intended for users to read or edit.
  If a user does edit it, the next `cust build` overwrites the
  changes silently — same posture as the existing generated
  `<crate>.h` (cust-design.md §5).
* **ThinLTO bitcode rlibs awkward in CMake.** Out of scope for
  v0.4.2 (V42D-7). The bitcode rlib format ships in v0.6
  (cust-design.md §17). v0.4.2 emits `.a` static archives only,
  which map trivially onto `add_library(... STATIC ...)`. When
  v0.6 arrives, the bitcode path will need either a custom CMake
  rule (`add_custom_command` driving `clang -emit-llvm` + custom
  archive layout) or the direct-Ninja-emitter resurrected for
  that specific case. v0.4.2 does not predetermine which.

The original "CMake as *export* format for consumers"
(`cust export cmake --consumable`) is unaffected and ships in
v0.8 per §17. The two CMake roles — *export-only for consumers*
and *driver for our own builds* — are now both active; the
companion doc's §3 still describes the export-only side.

### Headline outcome

A `cust build` against cwork's `hello-cstd`:

```console
$ cust build -p hello-cstd
   Compiling cstd v0.4.0
   Compiling hello-cstd v0.4.0
   Configuring workspace (cmake, target/debug/cmake)
   Building   workspace (ninja -j8)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.84s
$ ./target/debug/hello-cstd
Hello from cstd 0.4.0!
```

The `Configuring …` line appears only on the first build, on a
`Cust.toml` change anywhere in the workspace, or when the
module set changes. Subsequent builds run just the `Building
…` line (Ninja-only, no CMake reconfigure — see V42D-8).

The build tree gains one workspace-level CMake directory at
`target/<profile>/cmake/`; per-crate `target/<profile>/build/<crate>/`
still holds driver-generated artifacts (concatenated headers,
archives placed there via CMake's `ARCHIVE_OUTPUT_DIRECTORY`):

```
target/debug/
├── cmake/                              # workspace-level CMake tree (V42D-13)
│   ├── CMakeLists.txt                  # one file, all members
│   ├── build/                          # CMake build dir; ninja + .o
│   │   ├── build.ninja
│   │   ├── compile_commands.json       # source of truth from v0.4.2 onward
│   │   ├── CMakeCache.txt
│   │   └── …
│   └── stamp/                          # cust-owned: configure-skip stamps
│       └── cmakelists.sha256
├── build/                              # driver outputs, per-crate (unchanged)
│   ├── cstd/
│   │   ├── include/cstd.h               # concatenated fragment headers
│   │   └── libcstd.a                    # ARCHIVE_OUTPUT_DIRECTORY landing zone
│   └── hello-cstd/
│       └── include/hello-cstd.h
├── .h-fragments/cstd/…                 # plugin output, driver-owned (unchanged)
├── .rewrite/                           # `#cust use` lowering output (unchanged)
└── hello-cstd                          # bin output (Cargo parity, unchanged)
```

The generated `target/debug/cmake/CMakeLists.txt`
(shape — full byte-format pinned in V42D-4):

```cmake
# @generated by cust v0.4.2 — DO NOT EDIT
# Regenerated every `cust build`. Edits are lost.
cmake_minimum_required(VERSION 3.21)
project(cust_workspace LANGUAGES C)

set(CMAKE_C_STANDARD 23)
set(CMAKE_C_STANDARD_REQUIRED ON)
set(CMAKE_EXPORT_COMPILE_COMMANDS ON)

# ---- crate: cstd (library) ----
add_library(cstd STATIC
    "${CMAKE_CURRENT_LIST_DIR}/../.rewrite/cstd/src/types.c"
    "${CMAKE_CURRENT_LIST_DIR}/../.rewrite/cstd/src/lib.c"
)
set_target_properties(cstd PROPERTIES
    ARCHIVE_OUTPUT_DIRECTORY "${CMAKE_CURRENT_LIST_DIR}/../build/cstd"
    OUTPUT_NAME cstd
)
set_source_files_properties(
    "${CMAKE_CURRENT_LIST_DIR}/../.rewrite/cstd/src/lib.c"
    PROPERTIES OBJECT_DEPENDS
    "${CMAKE_CURRENT_LIST_DIR}/../.h-fragments/cstd/cstd__types.cust.h"
)

# ---- crate: hello-cstd (binary) ----
add_executable(hello-cstd
    "${CMAKE_CURRENT_LIST_DIR}/../.rewrite/hello-cstd/src/main.c"
)
set_target_properties(hello-cstd PROPERTIES
    RUNTIME_OUTPUT_DIRECTORY "${CMAKE_CURRENT_LIST_DIR}/.."   # Cargo parity
)
set_source_files_properties(
    "${CMAKE_CURRENT_LIST_DIR}/../.rewrite/hello-cstd/src/main.c"
    PROPERTIES OBJECT_DEPENDS
    "${CMAKE_CURRENT_LIST_DIR}/../.h-fragments/cstd/cstd__lib.cust.h;\
${CMAKE_CURRENT_LIST_DIR}/../.h-fragments/cstd/cstd__types.cust.h"
)
target_include_directories(hello-cstd PRIVATE
    "${CMAKE_CURRENT_LIST_DIR}/../build/cstd/include"
)
target_link_libraries(hello-cstd PRIVATE cstd)        # CMake intra-graph edge

# ---- shared compile options ----
# Plugin invocation — absolute path baked at emit-time (V42D-5).
foreach(_t cstd hello-cstd)
    target_compile_options(${_t} PRIVATE
        "SHELL:-fplugin=/home/user1/code/cust/target/debug/libcust_plugin.so"
        -Wno-unknown-attributes
    )
endforeach()
```

The driver runs:

```console
$ cmake -G Ninja -S target/debug/cmake -B target/debug/cmake/build
$ cmake --build target/debug/cmake/build -j 8 --target hello-cstd
```

(`-p <member>` lowers to `--target <member>`; bare `cust
build` omits `--target` so Ninja builds every default target.
`-jN` lowers to `cmake --build -j N` — Ninja handles
intra-workspace parallelism across crates and TUs in one
scheduler, not split between cust and Ninja.)

Phase 1 (surface extraction, plugin-driven fragment-header
emission) **stays in the driver, unchanged.** It's a fast,
no-codegen `clang -fsyntax-only` invocation per module that the
driver runs before CMake configure (V42D-2). Only phase 2
(codegen) and phase 3 (link) move under CMake.

## Scope

In suggested implementation order. In-progress V42D-N decisions
appear below the scope list. Anything marked **❓ open** is a
question the v0.4.2 design discussion needs to lock before the
first slice ships; anything marked **🔒 locked** is settled and
just listed here for context.

1. **`CMakeLists.txt` emitter.** New module
   `cust/src/cmake_emit.rs` that takes a `CompiledWorkspace`
   view (every member's sources after `#cust use` rewriting,
   include dirs, link inputs, cflag set, plugin path) and
   writes one deterministic `CMakeLists.txt` for the whole
   workspace. Byte-stable on no-input-change (V42D-4 pins the
   format). One `add_library` / `add_executable` per member;
   intra-workspace deps lowered as CMake target links
   (V42D-13).

2. **CMake + Ninja discovery.** `cust/src/build.rs` gains
   `find_cmake()` and `find_ninja()` helpers — `PATH` lookup
   with `--version` validation. Missing tool produces:
   ```
   error: `cmake` not found on PATH
     hint: install cmake >= 3.21 (apt install cmake / brew install cmake)
     note: cust v0.4.2+ requires cmake as a build-time dependency
   ```
   (V42D-10 / V42D-11.) No vendored fallback in v0.4.2.

3. **Configure-skip stamp.** `cust/src/build.rs` writes a
   SHA-256 of the generated `CMakeLists.txt` to
   `<build-dir>/cmake/stamp/cmakelists.sha256` after each
   configure. On subsequent builds, regenerate the
   `CMakeLists.txt` in a tempfile, hash it, compare; if equal
   skip `cmake -G Ninja` and go straight to `cmake --build`.
   V42D-8.

4. **`compile_one_module` / `compile_tree` rewrite.** The phase-2
   codegen path in `cust/src/build.rs` (lines ~534–800) is
   replaced by the emit → configure → ninja sequence. The
   surface pass (`surface_pass_fixed_point`) stays direct-clang
   (V42D-2). The test-runner build (`run_test_build` line
   ~342) also moves under CMake (V42D-14) as a second
   `add_executable` target in the same `CMakeLists.txt`.

5. **`compile_commands.json` switchover.** Delete the
   driver-side synthesiser in `cust/src/build.rs` line ~163;
   point cust's published path
   (`target/<profile>/compile_commands.json`) at the
   CMake-emitted file via symlink (or atomic rename copy on
   Windows, when we get there). V42D-12.

6. **`cust check` — surface pass only, no CMake.** `cust
   check` runs `surface_pass_fixed_point` and stops. That
   already invokes `clang -fsyntax-only -fplugin=…` per
   module (V40D-5) — which is exactly what `cust check` is
   supposed to validate. No CMake configure, no second
   build tree, no second stamp. V42D-15.

7. **Plugin path injection.** The generated `CMakeLists.txt`
   bakes the absolute path to `libcust_plugin.so` as a string
   literal in `target_compile_options(... -fplugin=...)`. cust
   resolves the path from `cust/src/plugin.rs::discover()`
   (existing) before emit-time. No CMake `find_package`
   indirection; the plugin is not a CMake target. V42D-5.

8. **`cust build -p <member>` lowering.** The Rust driver
   resolves `-p <member>` against the workspace member list
   (existing v0.3 behaviour) and passes `--target <member>`
   to `cmake --build`. `-jN` lowers to `cmake --build -j N`
   on the same invocation; Ninja handles both intra-TU and
   cross-crate scheduling in one graph. No per-crate fanout
   in `cust/src/workspace.rs` (V42D-13).

9. **Docs + cwork rewrite-free.** cust-design.md §4 "Build
   pipeline" gains a paragraph noting CMake/Ninja owns phases
   2–3 from v0.4.2 onward; §13 "Tooling integration"
   `compile_commands.json` bullet updated; §14 "CMake &
   compiler portability" headline decisions list is rewritten;
   §17 v0.4.x bullet is reslotted (build scripts pushed back).
   `cmake-and-portability.md` already carries the superseded
   banner (landed pre-implementation, in the same commit as
   this doc).

10. **Diagnostic rewriter** (`cust/src/diag_rewrite.rs`). New
    pure-function module classifying each line from CMake /
    Ninja stdout/stderr as `{compiler, ninja, cmake, other}`
    and rewriting accordingly. Closes most of RQ-V42-2 by
    construction. V42D-18.

Cwork sources are **not modified**. Phase 1's fragment-header
output, the published `<crate>.h`, and the test-discovery
sidecars are all byte-identical to v0.4.0 — only the per-TU
compile + link execution moves. cstd and hello-cstd version
bumps to 0.4.2 are mechanical.

## Decisions (V42D-N — locked)

### V42D-1 — Does CMake replace direct-clang invocation, or augment it?

**Locked: A. Replace (no driver-side fallback).** v0.4.2 is the
only path; there is no `CUST_BACKEND=direct-clang` escape hatch.

Considered: B. Augment — keep the direct-clang scheduler behind
a flag for the cases where CMake/Ninja aren't installed (CI
hardening, locked-down developer machines).

Why A:

* The whole win of the reversal (free `-jN`, free
  `compile_commands.json`, no in-house Ninja-emitter to
  write/maintain) requires that the direct-clang path actually
  go away. Keeping both means every milestone after v0.4.2 has
  to think about two backends, which is exactly the situation
  cmake-and-portability.md was right to warn against — just
  with the labels swapped.
* The "no CMake on the machine" case is solved by the install
  hint (V42D-10). cust is already in a position where it
  requires clang ≥ 19 and the cust plugin .so; adding cmake
  and ninja to that list is at most one apt-line of additional
  install friction.
* Backward-compat burden is zero — no external users yet.

The phase-1 surface pass (`surface_pass_fixed_point` in
`cust/src/build.rs`) stays direct-clang because it produces no
artifacts CMake would need to schedule; it's a fast
`-fsyntax-only` driven by the plugin. That's not "two
backends," it's "the driver runs the plugin once before
configuring CMake" — same as how the driver runs `mod_scanner`
today. V42D-2 makes the boundary explicit.

### V42D-2 — Driver / CMake boundary — what stays in the Rust driver?

**Locked.** The Rust driver retains exclusive ownership of:

1. **Manifest parsing + workspace resolution** (`manifest.rs`,
   `workspace.rs`). Unchanged.
2. **Module discovery + `#cust mod` walking** (`mod_scanner.rs`,
   `modules.rs`). Unchanged.
3. **`#cust use` lowering** (`mod_scanner.rs::rewrite`).
   Rewrite output goes to
   `target/<profile>/.rewrite/<crate>/<module>.c` — same
   location as today; CMake consumes these as its sources.
4. **Surface pass** (`surface_pass_fixed_point`). Phase 1
   plugin-driven fragment-header generation. Direct-clang
   invocation; runs once per crate before CMake configure.
   Fragment headers under `target/<profile>/.h-fragments/<crate>/`
   are inputs to CMake's build graph, not outputs — V42D-17
   covers why.
5. **Plugin discovery** (`plugin.rs::discover`). Result is
   baked into the generated `CMakeLists.txt` as a string
   literal (V42D-5).
6. **`<crate>.h` concatenation** (`build.rs` "Crate header"
   step). Topological over `#cust use crate::<mod>` edges,
   per cust-design.md §4 phase 3. Runs after phase 1, before
   CMake configure (so the concatenated header exists for
   `target_include_directories(... PRIVATE include)`).
7. **Test discovery + per-crate test executable plan**
   (`test_discovery.rs`, `test_runner_template.c`,
   `test_runner.rs`). Plugin sidecars (V40D-6) are read by
   the driver to produce one `cust_test_main.c` per member;
   each becomes one `add_executable(<crate>__test ...)`
   target in the single workspace CMakeLists (V42D-14).
8. **Test execution** (`test_runner.rs`). The fork-per-test
   harness (V32D-7) runs each member's test binary directly
   — no `ctest` involvement. CMake is a build-system layer,
   not a test runner.
9. **Workspace concurrency policy.** `cust/src/workspace.rs`
   resolves `-p` / `-jN` / `--target` semantics and passes
   the result to one `cmake --build` invocation. Inter-crate
   parallelism lives inside Ninja's graph, not in a cust-
   side fanout (V42D-13).

CMake (via Ninja) owns:

1. **Per-TU compile scheduling** (phase 2 codegen).
2. **Fragment-header invalidation** via `OBJECT_DEPENDS`
   (V42D-6).
3. **Depfile parsing** (`-MMD -MF` is what CMake configures
   Ninja to use natively).
4. **Link command** (phase 3 codegen).
5. **`compile_commands.json` emission** (V42D-12).
6. **Incremental decisions** below the CMakeLists granularity.

This boundary keeps the cust-specific machinery (plugin,
fragment headers, test discovery, `#cust use` lowering) in
Rust where it has access to the manifest model, and pushes
everything that's "scheduling C compilation" to CMake/Ninja.
The driver is smaller, not bigger, after v0.4.2.

### V42D-3 — Which CMake generator?

**Locked: Ninja only.** `cmake -G Ninja`. No support for
`Unix Makefiles`, `Xcode`, `Visual Studio`, or any other
generator in v0.4.2.

Why:

* Ninja's depfile semantics match what we'd want from any
  executor (cust-design.md §4 "Incremental compilation").
  CMake-generated Makefiles use a different, slower
  re-stat-everything model.
* Single generator = single set of bugs to chase, single set
  of edge cases for spaces-in-paths, generator-expression
  escaping, etc.
* `ninja` is small (~200 KB), trivially installed, and is the
  generator the rest of the C/C++ tooling world has
  consolidated on. The original doc's §6 question "vendor a
  pinned Ninja" stays open but unblocked by this milestone.

We do NOT use `--build` with a `-G` switch on subsequent
builds; configure is run with `-G Ninja` exactly once
(initial build / cache reset), and subsequent `cmake --build`
invocations re-use the cached generator. The configure-skip
stamp (V42D-8) is the bigger win.

### V42D-4 — Generated `CMakeLists.txt` location + byte format

**Locked.** Generated CMakeLists lives at:

```
target/<profile>/cmake/CMakeLists.txt
```

(workspace-level, not per-crate — V42D-13). The CMake build
dir (cache, `build.ninja`, object files) is *inside* that
directory:

```
target/<profile>/cmake/build/
```

This keeps the entire CMake-owned tree under one directory
the driver can `rm -rf` (or `cust clean` can sweep). Per-crate
driver-generated paths (`build/<crate>/include/`,
`build/<crate>/lib<crate>.a` via `ARCHIVE_OUTPUT_DIRECTORY`)
sit as siblings of the cmake tree, unchanged from v0.4.0
layout — important so that anything downstream that reads
`target/<profile>/build/<crate>/include/<crate>.h` still finds
it at the same path.

Byte format is pinned by a golden-file test in
`cust/src/cmake_emit/tests.rs`. Determinism rules:

* **Deterministic source order.** `CUST_SOURCES` list is in
  topological order over `#cust use crate::<mod>` edges (same
  order as the `<crate>.h` concatenation in cust-design.md §4
  phase 3). Stable on ties.
* **Deterministic member order.** Workspace members appear in
  topological order over inter-crate dep edges. Stable on
  ties (alphabetical by name). Same rule the driver applies
  for the workspace build plan today (V3D-N from v0.3.0).
* **Deterministic include / link orders.** Workspace deps in
  topological order; system include dirs and link libs in
  manifest declaration order.
* **Deterministic `OBJECT_DEPENDS` fragment-header lists.**
  Each per-source `OBJECT_DEPENDS` value (V42D-6) lists the
  importee fragment headers in the same topological order as
  the `<crate>.h` concatenation (cust-design.md §4 phase 3),
  stable on ties. Fragment-header set membership and order
  are both stamp-relevant; a re-order alone shouldn't churn
  the stamp but a set change must.
* **No timestamp in the generated file.** The `@generated by`
  comment names the cust version (e.g. `cust v0.4.2`) but no
  date or hash — those would make the SHA stamp churn on every
  driver upgrade unnecessarily. Version is enough for human
  readers; the configure-skip stamp (V42D-8) handles
  driver-version invalidation explicitly via the version
  string in the file.

The format is **not stable across cust versions.** A cust
0.4.2 → 0.4.3 upgrade will regenerate the CMakeLists with
whatever shape 0.4.3 wants and re-run configure (the SHA-256
stamp will mismatch because the embedded version string
changed). Users don't read the file; the format is internal.

### V42D-5 — Plugin invocation from CMake

**Locked: bake the absolute path as a string literal.**

The generated CMakeLists contains:

```cmake
target_compile_options(<target> PRIVATE
    "SHELL:-fplugin=/abs/path/to/libcust_plugin.so"
    -Wno-unknown-attributes
)
```

The path is resolved by the driver's `plugin.rs::discover()`
*before* CMakeLists generation. The plugin is **not** a CMake
`add_library` target, not a `find_package` lookup, not a
`-D` cache variable — the driver knows where the plugin is
because it owns plugin discovery, and the absolute path is the
cheapest way to communicate that to clang invocations CMake
schedules.

The `SHELL:` prefix bypasses CMake's de-duplication /
re-ordering logic on `target_compile_options`, which is
important: clang's `-fplugin=` flag is positionally
significant and CMake's normaliser otherwise mangles it.

Considered: B. Add a `cust-plugin` `add_library(IMPORTED)`
target that resolves `IMPORTED_LOCATION` from
`-DCUST_PLUGIN_PATH=…`, then reference `$<TARGET_OBJECTS:…>`.
Rejected because the plugin is loaded by clang at compile
time (it's a clang plugin, not a CMake plugin); CMake's
target machinery has nothing to add.

The `-Wno-unknown-attributes` is defensive — if a TU somehow
escapes the plugin's recognition path, the build degrades to
a warning rather than a hard error. Matches the v0.4.0
`--no-plugin` semantics for `cust check` (V40D-10).

### V42D-6 — Fragment-header invalidation through CMake

**Locked.** Each cust-rewritten TU lists every importee's
fragment header as a `OBJECT_DEPENDS` source-file property:

```cmake
set_source_files_properties(
    "${CMAKE_CURRENT_LIST_DIR}/../.rewrite/<crate>/<module>.c"
    PROPERTIES OBJECT_DEPENDS
    "<frag-1>;<frag-2>;…"
)
```

(`${CMAKE_CURRENT_LIST_DIR}` resolves to
`target/<profile>/cmake/` under V42D-13's single-CMakeLists
layout; `../.rewrite/...` reaches the driver's rewrite tree.)

The fragment list is the same list cust-design.md §4
"Incremental compilation" describes — one entry per `#cust use
crate::<importee>` edge resolved from the importee's
fragment-header location under
`target/<profile>/.h-fragments/<crate>/`.

This makes fragment-header changes propagate into Ninja's
dirty-set without depending on the depfile (which records
`#include`s, not "re-exported decls changed"). Same invariant
cust-design.md §4 "Incremental compilation" calls out;
v0.4.2 just lowers it to `OBJECT_DEPENDS` instead of an
in-house executor edge.

IMPORTANT: `OBJECT_DEPENDS` is a **rebuild-on-change** edge,
not a **produce-this-first** edge. If a fragment header is
missing, Ninja errors out — nothing in the CMake graph knows
how to produce one. v0.4.2 closes this by having the Rust
driver regenerate fragments before invoking `cmake --build`,
not by teaching CMake to produce them. The trade-off is
locked in V42D-17.

Considered: B. Generate a `add_custom_command` per fragment
with `BYPRODUCTS` listing the importing TUs' objects.
Rejected — too verbose, doesn't compose with the standard
`add_executable` / `add_library` generation, and breaks
Ninja's restat optimisation. (The whole-crate variant of B
is a different question — see V42D-17.)

### V42D-7 — ThinLTO / bitcode rlibs — interaction?

**Locked: out of scope for v0.4.2.** v0.4.2 emits `.a` static
archives only. The bitcode rlib format (cust-design.md §7,
shipping in v0.6) is deferred from this milestone's design.

When v0.6 lands, the bitcode path has two CMake-shaped options:

* **A.** Custom rule: `add_custom_command(OUTPUT foo.bc COMMAND
  clang -emit-llvm -c foo.c)` per TU, plus a custom archive
  layout for `metadata.json` + `.bc` files. Glue, but
  bounded.
* **B.** Per-crate generator switch: emit a different
  `CMakeLists.txt` shape entirely for bitcode-format crates,
  treating them as `add_custom_target` with full custom
  invocations and skipping CMake's compile / link rules
  entirely.

v0.4.2 doesn't predetermine which; it just ensures the static-
archive shape (the v0.4.x default) works end-to-end. v0.6's
design doc revisits this.

The "ThinLTO awkwardness" warning in
[cmake-and-portability.md](cmake-and-portability.md) §2.2 stands.
It's a real cost we'll pay at v0.6; v0.4.2 doesn't pretend it
goes away.

### V42D-8 — Configure-skip stamp

**Locked.** After each successful `cmake -G Ninja` configure,
the driver writes:

```
target/<profile>/cmake/stamp/cmakelists.sha256
```

containing the SHA-256 of the generated `CMakeLists.txt` at
configure time. One stamp for the whole workspace (V42D-13).

On subsequent `cust build`:

1. Driver regenerates the CMakeLists into a tempfile.
2. Computes its SHA-256.
3. If equal to the stamp → skip `cmake -G Ninja`, go straight
   to `cmake --build <build-dir>`.
4. If different → write the new CMakeLists into place, run
   `cmake -G Ninja`, update the stamp.

This is the answer to the "configure-step latency" concern in
[cmake-and-portability.md](cmake-and-portability.md) §2.3. In
the common edit-one-`.c`-file case, the source list, include
dirs, plugin path, link inputs, and cflag set are all
unchanged → CMakeLists is byte-identical → SHA matches → no
configure runs → Ninja takes over directly.

Configure runs when:

* `Cust.toml` changes (manifest → cflags, deps, include
  dirs).
* Module set changes (`#cust mod foo;` added/removed).
* Cust version changes (the `@generated by cust vX.Y.Z`
  string in the file changes).
* Plugin path changes (rare; happens on a cust upgrade or a
  `~/.cust/toolchain/` rebuild).

This is the same coarse-grained "config inputs changed →
reconfigure" decision Cargo makes when `Cargo.toml` changes;
we lean on CMake's own cache validation (`CMakeCache.txt`)
underneath as a second-level guard against compiler version
drift.

Considered alternative (recorded for future revisit): hash a
**canonicalised `Cust.toml`** instead of the generated
CMakeLists. The [musto](../../../musto/) project takes this
approach — [`manifest::canonicalise()`](../../../musto/musto/src/manifest.cppm)
re-parses the manifest through toml++ and re-emits with
sorted keys, then [`paths::sha256_hex()`](../../../musto/musto/src/paths.cppm)
hashes the bytes (see [prior-art-musto.md](prior-art-musto.md)
§3). Advantage: a cust version bump that changes only the
`@generated by cust vX.Y.Z` comment string doesn't churn the
stamp, so version bumps don't cause spurious reconfigures.
Cost: requires implementing a canonical TOML serialiser
(Rust `toml` crate doesn't sort keys by default).

v0.4.2 picks the simpler CMakeLists-hash because we don't
yet have the canonical-TOML machinery and version bumps are
rare enough that the spurious reconfigure cost is
negligible. Revisit when (a) the `Cust.lock` work in v0.4.6
forces us to write a canonical TOML serialiser anyway, or
(b) version-bump churn becomes a real complaint.

### V42D-9 — CMake version floor

**Locked: 3.21.** Matches the floor for the
`cust export cmake --consumable` work in
[cmake-and-portability.md](cmake-and-portability.md) §7.

Why 3.21:

* `IMPORTED` target features cust-design.md §6 export work will
  rely on (consistency between driver-emitted and export-emitted
  CMakeLists).
* Ships in Ubuntu 22.04 LTS (`cmake 3.22.1`), Debian 12 stable
  (`3.25.1`), Fedora 38+, recent macOS Homebrew, recent
  Windows installers. The ubuntu-24.04 GitHub runner cust CI
  uses (per `.github/workflows/ci.yml`) ships CMake 3.28.
* `cmake_minimum_required(VERSION 3.21)` in the generated file
  triggers a clean `cmake` error if a user's CMake is older,
  before any cust-specific machinery runs.

The driver's `find_cmake()` (V42D-10) parses `cmake --version`
and rejects older installs with an install hint, so the error
surface is consistent regardless of whether CMake fails on its
own version check.

### V42D-12 — `compile_commands.json` ownership

**Locked.** v0.4.2 deletes the driver-side
`compile_commands.json` synthesiser
(`cust/src/build.rs` lines ~163–185). CMake emits the
canonical file via `set(CMAKE_EXPORT_COMPILE_COMMANDS ON)`
into:

```
target/<profile>/cmake/build/compile_commands.json
```

The driver publishes the canonical path
`target/<profile>/compile_commands.json` as a symlink to that
file. Single workspace CMakeLists (V42D-13) means a single
`compile_commands.json` covering every member's TUs — no
driver-side merge step.

The driver-side synthesiser dies because the CMake-emitted
version is *more accurate*: it captures the exact post-
expansion compile invocation, including CMake-injected flags
that the driver's synthesiser would have had to mirror by
hand. v0.4.2 makes clangd's view match the build's view
exactly, for free.

### V42D-13 — Workspaces: one CMakeLists for the whole workspace

**Locked.** v0.4.2 emits ONE `CMakeLists.txt` for the entire
workspace at `target/<profile>/cmake/CMakeLists.txt`. Each
member is one `add_library(<crate> STATIC ...)` or
`add_executable(<crate> ...)` target (or both, for lib+bin
crates); intra-workspace deps lower to
`target_link_libraries(<bin> PRIVATE <lib>)` CMake target
edges. One `cmake -G Ninja` per workspace; one `cmake --build`
per `cust build` invocation; Ninja owns inter-crate scheduling
in a single build graph.

Why single-CMakeLists (reversal of the per-member position):

* **Ninja gets the whole workspace dep graph in one
  scheduler.** With per-member CMake trees we had two layers
  of parallelism (cust-side fanout across crates, Ninja-side
  within each crate) that couldn't load-balance against each
  other. RQ-V42-3 was the open question about that mismatch;
  with a single CMakeLists it dissolves — a `cust build -j 8`
  lowers to `cmake --build -j 8` and Ninja makes one set of
  scheduling decisions over the union of every member's
  TUs.
* **One configure cost, not N.** Per-member trees pay
  `cmake -G Ninja` startup once per crate per relevant
  manifest edit. Workspaces with many small crates were the
  worst case. The configure-skip stamp (V42D-8) softened
  this but didn't eliminate it; one CMakeLists eliminates
  N−1 of the configures by construction.
* **`compile_commands.json` is one file, no merge.** V42D-12
  loses the driver-side merge step.
* **`cust build -p <member>`** lowers to `cmake --build
  --target <member>`. Ninja walks the same workspace graph
  and builds only the requested target + its transitive
  deps. Same semantics as the per-member fanout produced,
  with less driver code.
* **Smaller cust driver.** No `workspace.rs` fanout, no
  per-crate stamp set, no per-crate CMake invocation
  orchestration. The driver does one configure-skip check
  and one `cmake --build` call.
* **CMake users actually do this.** A single top-level
  CMakeLists with `add_subdirectory` (or, in our case, flat
  inline `add_library` / `add_executable` per member) is
  the idiomatic CMake-workspace shape. The per-member
  fanout was idiomatic for cust's pre-CMake direct-clang
  scheduler; it isn't for CMake.

Considered (rejected):

* **A. Per-member CMakeLists with cust-side fanout** (the
  previous V42D-13 lock). Rejected for the five reasons
  above. The original motivation — preserving per-crate
  isolation — was a worry inherited from the direct-clang
  scheduler that CMake doesn't share. CMake target
  visibility rules (`PRIVATE` / `INTERFACE` / `PUBLIC`) and
  per-target `target_compile_options` already give us all
  the per-crate isolation we need within a single graph.
* **B. Per-member CMakeLists glued together via
  `add_subdirectory`.** The CMake-idiomatic shape. This is
  what the [musto](../../../musto/) project ships today —
  see [prior-art-musto.md](prior-art-musto.md) §2 for the
  full cross-reference. Their umbrella is ~13 lines
  (`cmake_minimum_required` + `project` + a few `set()`
  lines + one `add_subdirectory(<name>)` per member); per-
  member CMakeLists hold the bulk. Inter-crate parallelism
  is identical to option C (one `cmake -G Ninja` + one
  `cmake --build` against the umbrella; Ninja walks the
  add_subdirectory cascade into one graph) and the file-
  system shape mirrors the source tree, which is easier to
  inspect when debugging.

  We picked C over B on emitter simplicity (one `emit()`
  function and one golden file vs two of each) and on the
  "one file to read when debugging" property of the flat
  shape. Both are defensible. Migration C → B is
  straightforward (see the migration paragraph at the end of
  this decision) if option C ever surfaces a real cost
  musto's experience hasn't.

  Updated 2026-06-11 from "rejected as worst of both
  worlds" — musto's 0.1.0–0.5.2 shipped experience does not
  support that framing.

What we keep from the previous lock:

* **Per-crate driver-generated outputs stay per-crate.**
  `target/<profile>/build/<crate>/include/<crate>.h` (the
  concatenated fragment header) and
  `target/<profile>/build/<crate>/lib<crate>.a` (CMake-
  emitted via `ARCHIVE_OUTPUT_DIRECTORY`) keep the v0.3+
  layout. Downstream code that reads either path doesn't
  notice the backend swap.
* **cust-design.md §4 `target/<crate>/.lock` model** — still
  applies, just rarely contended. The lock matters when two
  separate `cust` invocations race on the same crate (e.g.
  parallel `cust check` + `cust build`). Inside a single
  workspace `cust build`, all writes happen under one
  `cmake --build` and don't race. v0.4.2 swaps the lock
  granularity from per-crate to *workspace-level for the
  build*: a single advisory lock at
  `target/<profile>/cmake/.lock` held by the cust process
  for the duration of the cmake invocation. Per-crate locks
  in cust-design.md §4 stay defined for the cases that still
  use them (`cust test` reads + writes per-crate output
  dirs for fork-per-test isolation; that's still per-crate).

Migration if a real constraint forces per-member CMakeLists
back (huge workspaces where one CMakeLists hits CMake's
configure-time scaling cliff, say): swap `cmake_emit::
generate()` to emit per-member files + a root
`add_subdirectory` umbrella; restore the per-member configure-
skip stamps; everything else stays the same. The boundary
(driver owns surface pass + concatenation; CMake owns codegen
+ link) is unchanged either way.

### V42D-14 — Test runner under CMake

**Locked.** Each member's test executable is one
`add_executable(<crate>__test ...)` target in the single
workspace CMakeLists (V42D-13), guarded by
`if(CUST_TEST_BUILD)`:

```cmake
# Test-build only: appears when CUST_TEST_BUILD is ON.
if(CUST_TEST_BUILD)
    # One block per member, emitted as part of the per-member section.
    add_executable(<crate>__test
        "${CMAKE_CURRENT_LIST_DIR}/../.rewrite/<crate>/src/lib.c"
        "${CMAKE_CURRENT_LIST_DIR}/../.rewrite/<crate>/src/<other-modules>.c"
        "${CMAKE_CURRENT_LIST_DIR}/cust_test_main_<crate>.c"
    )
    target_compile_definitions(<crate>__test PRIVATE CUST_TEST_BUILD=1)
    target_include_directories(<crate>__test PRIVATE
        "${CMAKE_CURRENT_LIST_DIR}/../build/<crate>/include"
    )
    # Same -fplugin invocation as the library target.
endif()
```

The cust driver passes `-DCUST_TEST_BUILD=ON` when running
`cust test`. The `CUST_TEST_BUILD=1` preprocessor macro is the
existing v0.3.2 V32D-4 signal that activates the prelude's
test branch.

The generated `cust_test_main_<crate>.c` files (one per
member, from `test_runner_template.c` + that member's plugin
sidecars) are written into
`target/<profile>/cmake/cust_test_main_<crate>.c` by the
driver before CMake configure. Filename includes the crate
name to keep the workspace CMakeLists's source list
unambiguous.

`ctest` is **not** used. The driver still spawns each
`<crate>__test` binary directly via the fork-per-test harness
(V32D-7), iterating over members in dep order. CMake's role
here is build-only.

For `cust test -p <member>`, the driver passes
`--target <member>__test` to `cmake --build` so Ninja builds
only that member's test executable + its transitive deps;
then runs only that member's tests.

Test isolation under `target/<profile>/test/<crate>/` from
v0.3.2 V32D-4 still applies for the **runtime** side
(per-test cwd, output capture); the test executable itself
lives in `target/<profile>/cmake/build/` like every other
CMake-built artifact and is invoked from the isolated
per-crate dir.

### V42D-15 — `cust check` stays out of CMake entirely

**Locked: `cust check` runs the surface pass and stops. No
CMake, no second tree, no second stamp.**

What `cust check` is supposed to validate — that every TU
parses cleanly and the plugin's phase-1 checks pass — is
exactly what `surface_pass_fixed_point` already does. V40D-5
locks phase 1 as `clang -fsyntax-only -fplugin=…` per
module, with the plugin emitting fragment headers + running
AST checks and **no codegen**. `cust check` simply calls
that path and reports.

Concrete shape after v0.4.2:

```rust
// cust/src/cli.rs (sketch)
Check => {
    let plan = BuildPlan::resolve(&workspace, &cli)?;
    build::surface_pass_fixed_point(&plan)?;   // existing entry point
    // No header concat, no cmake_emit, no cmake invocation.
    Ok(())
}
```

Considered:

* **A. Drop `cust check` entirely.** Rejected. Cargo-parity
  command, real CI/headless use case, clangd doesn't
  substitute (clangd is in-editor only).
* **B. Separate `cmake-check/` tree with `add_library(...
  OBJECT EXCLUDE_FROM_ALL ...)` + `-fsyntax-only`** (the
  previous V42D-15 lock). Rejected. The cmake-check tree
  duplicates the build tree's structure to do work that the
  surface pass already does, and pays a CMake configure cost
  every cust upgrade or manifest change. The previous lock
  defended this on the grounds that "after slice D nothing
  in cust shells out to clang directly" — but
  `surface_pass_fixed_point` shells out to clang directly
  regardless (V42D-2 makes that boundary explicit), so the
  purity argument was illusory.
* **C. Reuse the main `CMakeLists.txt` with a
  `-DCUST_CHECK_ONLY=ON` switch.** Rejected. Same reasons B
  was originally rejected (would flap the configure-skip
  stamp every time the user alternates `cust check` and
  `cust build` against the same crate) — *plus* it pays CMake
  configure cost for what's already a fast surface-pass
  invocation. Strictly dominated by A.

Why this is the right boundary:

1. **The work cust check does is exactly the work the
   surface pass does.** Both want `-fsyntax-only` per
   module, both want the plugin loaded, both want fragment
   headers regenerated as a side effect, neither wants
   codegen or link. Reusing the surface pass is not a
   shortcut; it's the natural home for the work.
2. **`cust check` becomes the fastest command in the
   toolbox by construction.** No CMake configure, ever —
   edit-check loop feels instant. Cargo's `cargo check`
   tries to approximate this with a `--emit=metadata` short-
   circuit; cust's structural answer is simpler.
3. **V40D-10 `--no-plugin` collapses to one line.** The flag
   becomes "omit `-fplugin=...` from the surface pass's
   clang argv." No CMake cache variable, no per-target
   conditional, no second tree. RQ-V42-5 dissolves.
4. **One fewer build tree to invalidate.** v0.4.2 already
   has the SHA-256 configure-skip stamp for the build tree
   (V42D-8) and the test-build invocation (V42D-14, sharing
   the same tree); a separate stamp for the check tree was
   pure overhead.

What this gives up:

* **Phase-2 semantic checks** that the plugin only runs in
  codegen mode (V40D-2's deferred slate: `must_use`,
  `no_panic`, `const` constexpr-equivalence, `unsafe` call-
  site enforcement). These ship in v0.7. When they land,
  `cust check` either (a) extends to also run a non-link
  codegen pass with the plugin in phase-2 mode (probably via
  the same CMake emitter, with a check-specific target shape
  resembling B above), or (b) declares phase-2 semantic
  checks the responsibility of `cust build` only.
  Decision deferred to v0.7's design doc; v0.4.2 does not
  predetermine the answer.

Revisit if any of the following fire:

* v0.7 plugin checks make `cust check` meaningfully weaker
  than `cust build`, and users notice.
* A real workflow surfaces where `cust check` needs
  per-target build invalidation (e.g. project-wide lint
  rules that the surface pass can't express).

### V42D-16 — Migration of `cust/src/build.rs`

**Locked.** Slice B deletes:

* `compile_one_module` (lines ~582–740)
* `compile_tree` (lines ~534–580)
* `argv_with_clang` helper (used only by the above)
* `compile_commands.json` synthesiser (lines ~163–185)

…and replaces them with one ~80-line `emit_and_drive_cmake()`
that wires `cmake_emit::generate()` → configure-skip-stamp →
`cmake -G Ninja` (conditional) → `cmake --build`.

The surface pass (`surface_pass` / `surface_pass_fixed_point`,
lines ~747–950) is **unchanged.** It calls `clang
-fsyntax-only -fplugin=… <src>` directly and is fast enough
that routing it through CMake would only add latency. V42D-2
documents this as the deliberate boundary.

The test-runner-template TU compile (lines ~440–485) is
**deleted** — the runner TU is now compiled as part of the
test executable's CMake target (V42D-14). The two separate
clang invocations (one for the runner TU, one for the crate's
sources) collapse to one CMake build.

Slice plan ordering ensures the build never breaks: slice A
adds `cmake_emit.rs` without wiring it; slice B does the wire-
up + delete in one atomic commit. Bisect lands cleanly on
either side.

### V42D-17 — Who regenerates fragment headers?

**Locked: the Rust driver, in-process, before each `cmake
--build` invocation.** Fragment headers under
`target/<profile>/.h-fragments/<crate>/` are NOT CMake/Ninja
build outputs — they are pre-existing inputs from CMake's
point of view, referenced via `OBJECT_DEPENDS` (V42D-6) for
change propagation only.

The gap V42D-6 leaves: `OBJECT_DEPENDS` is a rebuild-on-change
edge, not a produce-this-first edge. If a fragment header is
missing, Ninja has no rule that produces it and the build
errors out. V42D-17 closes that gap by making the driver
responsible for fragment existence + freshness before CMake
is ever asked to schedule a per-TU compile.

Considered:

* **A. Driver-owned (locked).** Each `cust build` /
  `cust test` / `cust check` runs `surface_pass_fixed_point`
  in-process before invoking `cmake --build`. The generated
  CMakeLists references fragment headers as `OBJECT_DEPENDS`
  inputs only. The driver also runs the `<crate>.h`
  concatenation step (cust-design.md §4 phase 3) immediately
  after, before configure.

* **B. CMake-owned via whole-crate `add_custom_command`.**
  Add a per-crate multi-output custom command:
  ```cmake
  add_custom_command(
      OUTPUT ${CUST_FRAGMENT_HEADERS}    # all frags for the crate
      COMMAND /abs/path/to/cust internal-surface-pass <crate>
              --out ${CUST_FRAGMENTS_DIR}
      DEPENDS ${CUST_SOURCES}
  )
  ```
  Per-TU compiles list fragment headers as regular source
  inputs (via the custom command's outputs, which CMake/Ninja
  wire up as proper build edges). Driver no longer runs the
  surface pass itself; build orchestration moves entirely into
  CMake's graph. Requires a new hidden CLI subcommand
  `cust internal-surface-pass <crate-dir> --out <dir>
  --plugin <path>` that re-resolves manifest + plugin state
  from CLI args and runs the fixed-point loop.

Why A:

1. **Smaller surface area.** No new subcommand, no
   `current_exe()` path-baking for the cust binary in the
   generated CMakeLists, no need for the fixed-point loop to
   be a self-contained CLI entry with its own arg parsing.
   The driver/CMake boundary V42D-2 establishes stays intact:
   anything cust-specific (plugin, fragment headers, test
   discovery, `#cust use` lowering) is the Rust driver's job;
   anything that's scheduling C compilation is CMake's job.
2. **`cust build` is the only supported entry point.** Same
   posture as today — users don't run `clang` directly to
   advance the build, and they won't run `ninja` directly
   either. Documented as unsupported (see "Risks accepted"
   below).
3. **Faster on the common path.** The in-process surface pass
   is one Rust method call against an already-loaded manifest
   + plugin. Option B pays subprocess startup + arg parsing +
   manifest re-load + plugin re-discovery + fixed-point loop
   every time Ninja decides fragments need regen. Negligible
   absolute cost per invocation (~10–30ms), but multiplies
   across crates in a workspace and against the SHA-256
   configure-skip stamp (V42D-8) where the in-process variant
   has nothing to do at all on a no-source-change rebuild.
4. **The custom-command shape doesn't reduce work.** Surface
   pass is whole-crate (fixed-point), not per-module — a
   single multi-output `add_custom_command` works, but
   `DEPENDS ${CUST_SOURCES}` triggers a full re-run on any
   source change. Granularity matches option A; option B just
   moves the trigger into Ninja's scheduler at the cost of a
   subprocess.
5. **Workspace `-jN`** (V42D scope item 8) already
   parallelises across crates at the driver level. Option B's
   "Ninja could overlap one crate's surface pass with another
   crate's codegen" doesn't materialise without restructuring
   the workspace driver to interleave per-crate phases (much
   bigger change than v0.4.2 wants). (V42D-13's single-
   CMakeLists lock means Ninja already overlaps codegen
   across crates trivially — the only thing the driver still
   serialises is surface-pass-before-cmake-build, and option
   A keeps that simple.)

Risks accepted:

* **Direct `ninja -C target/.../build` invocation can see
  stale fragments.** Not a supported workflow. If a user runs
  Ninja directly after editing source, dependent TUs compile
  against yesterday's surface and the link either succeeds
  with old behaviour or fails with a clear undefined-symbol
  error. Visible, not silent. Documented in v0.4.2 closeout:
  `cust build` / `cust test` / `cust check` / `cust run` are
  the only supported entry points; the generated
  `CMakeLists.txt` is internal scaffolding, not a public
  contract.
* **IDE integrations that invoke Ninja directly.** clangd uses
  `compile_commands.json` and does NOT invoke Ninja, so it's
  unaffected. CLion / Qt Creator / VS could in principle
  bypass `cust build`; the answer is "configure the IDE's
  build command to `cust build`, not `ninja`." Same answer
  Cargo users give for projects with build scripts.
* **First-time `cmake --build` on a freshly-cloned checkout
  with prebuilt `target/`.** If someone unpacks a `target/`
  tarball with `.h-fragments/` populated but stale relative to
  the source, and runs `cmake --build` directly without
  `cust build`, the build uses the stale fragments. Same
  shape as the previous bullet; same answer.

Migration path if option B becomes necessary later (because
of a real IDE/workflow constraint we don't currently see):
add the `internal-surface-pass` subcommand to
`cust/src/cli.rs`, change the `cmake_emit::generate()`
template to use the custom command, drop the in-driver
`surface_pass_fixed_point` call from `emit_and_drive_cmake()`.
Mechanical, contained, ~150 LoC across two files. Today's
decision keeps v0.4.2 small; the door stays open.

### V42D-18 — Diagnostic rewriter from slice B

**Locked: ship `cust/src/diag_rewrite.rs` in slice B,
classifier + verbatim passthrough as the minimum shape.**

RQ-V42-2 asked whether the v0.4.2 build path needs an
output filter over `cmake --build`'s stdout/stderr. The
[musto](../../../musto/) project answered yes and shipped one
([`musto/src/diag_rewrite.cppm`](../../../musto/musto/src/diag_rewrite.cppm),
~80 LoC, golden-file tested) since v0.1.0. See
[prior-art-musto.md](prior-art-musto.md) §4 for the full
cross-reference; the short version is that the rewriter pays
for itself the moment Ninja or CMake emit anything that
doesn't lead with `<file>:<line>:<col>:`.

Shape:

```rust
pub enum Source {
    Compiler,   // `<file>:<line>:<col>: (error|warning|note):`
    Ninja,      // line starts with `ninja: `
    Cmake,      // line starts with `CMake Error` or `CMake Warning`
    Other,      // unrecognised
}

pub fn classify(line: &str) -> Source { ... }
pub fn rewrite(line: &str, crate_name: &str) -> Cow<'_, str> { ... }
```

Four cases, one non-trivial transform:

1. **`Source::Compiler`** — verbatim passthrough. Clang's
   diagnostics are already correct and source-relative; we
   don't touch them.
2. **`Source::Ninja`** — strip the `ninja: error: ` prefix
   and reformat the missing-input case into a cust-shape
   diagnostic:
   ```
   error[cust]: missing build input
     --> src/util.c.o (needed by `<crate>` staticlib)
   ```
   (Pattern stolen verbatim from musto.) Other ninja lines
   pass through.
3. **`Source::Cmake`** — v0.4.2 ships passthrough only. The
   long-term ambition (point CMake configure errors back at
   the offending `Cust.toml` key, the way musto does) is
   in scope for a follow-up but not required to ship v0.4.2.
4. **`Source::Other`** — verbatim passthrough.

Pure function (`&str -> Cow<str>`), zero I/O, golden-file
testable. Wired in at the cmake/ninja child-process spawn
site introduced by slice B (the same call site that replaces
`compile_one_module`'s direct `Command::new("clang")` invocations
in `cust/src/build.rs`) as a per-line stdout+stderr
transformer. The spawn site is a small inline helper in
`build.rs`, not a new `subprocess.rs` module — promote to a
shared module only if a second caller appears.

Why ship in slice B, not a follow-up:

* **The `compile_one_module` direct-clang path being deleted
  in slice B has no rewriter** (it didn't need one — clang's
  output was already user-facing). Without the diag rewriter,
  slice B regresses the diagnostic experience for any error
  that surfaces via CMake/Ninja rather than clang. Shipping
  the rewriter in the same slice means the regression never
  exists.
* **Minimum shape is cheap.** Classifier + verbatim
  passthrough is ~30 LoC. Even with the one Ninja
  missing-input rewrite, the whole module is ≤ 80 LoC,
  testable as a pure function, no integration test needed.
* **Closes most of RQ-V42-2 by construction.** The open
  question becomes "which additional rewrite rules to add"
  rather than "do we need a rewriter at all," which is a
  much lower-stakes follow-up.

What slips:

* The CMake-error-to-`Cust.toml`-key mapping. Needs a
  manifest-parse-time table of key locations; non-trivial.
  Tracked as a v0.4.x follow-up; not blocking v0.4.2 ship.
* Multi-line block handling (musto notes the same limitation).
  Per-line classifier loses block context. Same posture: ship
  per-line, grow if needed.

Considered: A. Ship slice B without a rewriter; defer to a
follow-up. Rejected because of the regression point above.
B. Ship the full key-mapping rewriter in slice B. Rejected as
over-scoped — the table-of-key-locations work is a separate
chunk that has no v0.4.2-critical use.

## Open questions (RQ-V42-N — locked)

1. **RQ-V42-1 — Configure-skip stamp completeness.**
   **Locked: hash the plugin .so's bytes into the same
   SHA-256 V42D-8 already computes.** The plugin path is
   already in the generated CMakeLists, so a path change
   triggers a SHA mismatch via V42D-8's existing logic. The
   remaining gap — same path, different `.so` contents —
   closes by changing V42D-8's hash input from "CMakeLists
   bytes" to "CMakeLists bytes ‖ plugin .so bytes" (single
   `Hasher::update()` call after reading the .so via
   `std::fs::read`). Five extra lines of driver code, no
   new stamp file, no new failure mode. clang version drift
   stays CMakeCache.txt's responsibility (it already does
   this well — `CMakeCache.txt` records `CMAKE_C_COMPILER`
   + `CMAKE_C_COMPILER_VERSION` and CMake itself triggers
   reconfigure on mismatch). cust binary mtime is NOT in
   scope — cust upgrades that change build behaviour also
   change the `@generated by cust vX.Y.Z` string in the
   CMakeLists, which already mismatches the stamp.

   Considered: separate `plugin.sha256` stamp file (more
   moving parts), in-toto-style attestation (over-
   engineered for v0.4.2), trust CMake's discovery entirely
   (doesn't notice in-place .so swaps with matching mtime).
   Hashing into the existing stamp is the simplest path
   that covers the failure mode.

2. **RQ-V42-2 — Diagnostic rewrite rule set.**
   **Locked: ship the three rules in V42D-18 and stop.**
   No additional rules in v0.4.2. The per-line classifier
   has no block-context awareness; that's acceptable
   because (a) clang diagnostics — the multi-line case
   that matters most — pass through verbatim under
   `Source::Compiler` and never lose context, and (b) the
   ninja/cmake multi-line cases are rare and tolerable as
   passthrough.

   Additional rules land **on demand** — when a real bug
   report or a cwork dogfooding session surfaces a
   diagnostic that's genuinely worse than the v0.4.0
   direct-clang shape, we add a rule. Each new rule is
   ≤ 20 LoC + one golden-file test. Tracked as a v0.4.x
   follow-up, not a v0.4.2 blocker.

   Block-context awareness deferred indefinitely; revisit
   only if (a) someone reports a diagnostic where the
   per-line classifier produces actively misleading output,
   not just suboptimal, or (b) a v0.5+ milestone needs the
   machinery for another reason.

3. **RQ-V42-3 — *deleted*.** This was the per-crate-fanout
   vs Ninja-N×N concurrency mismatch question. V42D-13's
   reversal (single workspace CMakeLists, single `cmake
   --build`) dissolves it: Ninja owns all parallelism in one
   scheduler. Workspace concurrency policy is now "pass
   `-jN` straight through to `cmake --build -j N`". Tuning
   knobs that v0.4.3 was going to add against the fanout
   model become Ninja's job, not cust's.

4. **RQ-V42-4 — `cwork/`-style nested workspaces.**
   **Locked: no special handling.** cwork has its own
   `Cust.toml` workspace root, its own `target/`, its own
   workspace-level CMakeLists, and runs in a directory the
   cust binary build never visits. There is nothing for the
   v0.4.2 emitter to coordinate.

   Verification is the slice B integration test (already in
   the plan) — `just test-cwork` after the slice lands. If
   that passes, nested-workspace support shipped. No new
   code, no new tests beyond what slice B already requires.
