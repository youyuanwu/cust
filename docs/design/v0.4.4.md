# cust v0.4.4 — multi-bin per crate (`src/bin/*.c`, `[[bin]]` arrays)

**Status:** � **shipped** (opened 2026-06-13, closed 2026-06-13).
**Parent doc:** [cust-design.md §17 roadmap](cust-design.md).
**Previous milestone:** [v0.4.3.md](v0.4.3.md) (shipped 2026-06-13).
**Builds on:** v0.3.1 binary crates & `cust run` (V31D-1 … V31D-11),
v0.4.2 CMake/Ninja build backend (V42D-13, V42D-14), and the
v0.4.3 `tests/`-discovery cadence (V43D-1 stem walk, V43D-8 stem
validity) which this milestone mirrors for `src/bin/`.
**Minimum clang / CMake / Ninja:** unchanged from v0.4.3
(clang 19, CMake 3.21, any Ninja).

This file is the v0.4.4 record — scope, in-progress design
decisions (**V44D-N**), open questions (**RQ-V44-N**), and the
verification target. v0.4.4 lifts the v0.3.1 **V31D-3** deferral:
a crate may now produce **more than one executable**, via Cargo's
two conventions — `src/bin/<name>.c` auto-discovery and a
multi-entry `[[bin]]` array. The single-bin shapes from v0.3.1
(`src/main.c`, single `[[bin]]` table) keep working unchanged.

---

## 1. Headline outcome

A crate gains a `src/bin/` directory alongside `src/main.c`:

```
cwork/examples/hello-cstd/
├── Cust.toml
└── src/
    ├── main.c                  # package-named bin: hello-cstd
    └── bin/                    # NEW in v0.4.4
        ├── bench.c             # one extra bin per file: bench
        └── dump.c              # …and dump
```

`cust build` against cwork now produces three executables for
this member:

```console
$ cust build
   Compiling cstd v0.4.3
   Compiling hello-cstd v0.4.3
   Configuring workspace (cmake, target/debug/cmake)
   Building   workspace (ninja -j8)
    Finished `dev` profile [unoptimized + debuginfo]

$ ls target/debug
hello-cstd  bench  dump  build/  cmake/  deps/  test/
```

`cust run` selects a bin by name; with more than one bin in the
selected member it **requires** `--bin` (Cargo parity):

```console
$ cust run -p hello-cstd
error: `cust run` could not determine which binary to run in
  package `hello-cstd`: available binaries are `bench`, `dump`,
  `hello-cstd`. Use `--bin <NAME>` to select one.

$ cust run -p hello-cstd --bin bench
     Running `target/debug/bench`
bench: 1000000 iters in 4.2ms
```

Bin **output paths** stay flat at `target/<profile>/<bin-name>`
(V31D-4 generalised): the package-named bin keeps
`target/<profile>/<crate>`; each `src/bin/<name>.c` lands at
`target/<profile>/<name>`. No `bin/` subdirectory.

## 2. What v0.4.4 ships (MVP)

In-scope:

* New `src/bin/<name>.c` discovery convention (V44D-1): one `.c`
  file at the top level of `<crate>/src/bin/` = one executable
  named after its stem, alongside the existing `src/main.c`
  package-named bin.
* Multi-entry `[[bin]]` array (V44D-4): the v0.3.1 single-entry
  table relaxes to a real array; each entry gains a required
  `name` and an optional `path`. Explicit entries add bins or
  override the path of an auto-discovered one.
* Bin-name validity + collision rules (V44D-3), mirroring the
  v0.4.3 V43D-8 stem rules: `[A-Za-z][A-Za-z0-9_-]*`, no
  duplicate names, no `src/bin/<pkg>.c` shadowing the
  `src/main.c` package bin.
* One CMake `add_executable(<bin-target>)` per bin (V44D-5),
  emitted into the same workspace `CMakeLists.txt` v0.4.2
  generates (V42D-13). Each bin links the member's own lib
  half (if any) + workspace deps, exactly like the v0.3.1
  single bin.
* `cust run --bin <name>` selection (V44D-6) with the Cargo
  "ambiguous binary" error when a multi-bin member is targeted
  without `--bin`. `cust build --bin <name>` scopes the build
  to one executable (V44D-7).
* `CrateKind` carries a `Vec<BinTarget>` instead of a single
  optional bin source (V44D-8).
* cwork dogfood (V44D-10): `hello-cstd` grows a `src/bin/`
  with one extra bin exercising the multi-bin path end-to-end.

Out of scope (deferred — see §5/§6):

* `src/bin/<name>/main.c` multi-file bin subdirectories
  (V44D-2 — DEFERRED, mirrors v0.4.3 V43D-2).
* Per-bin `[dependencies]` / per-bin `[clang]` config
  (V44D-11 — DEFERRED; all bins share the member's
  `[dependencies]` + `[clang]`).
* Strict `cust check` over bin TUs (V44D-9 — DEFERRED at
  slice C; `cust check` is a tolerant lib-surface pass, same
  finding as v0.4.3 V43D-13). Bin syntax errors surface at
  `cust build` / `cust run` time.
* `default-run` (RQ-V44-1), `required-features`
  (RQ-V44-2), and unit tests **inside** `src/bin/*.c`
  (RQ-V44-3) — all deferred.

## 3. Why v0.4.4 now

v0.4.3 shipped `tests/` integration tests (pulled forward from
the old v0.4.5 slot when v0.4.2 slice D absorbed the `--jobs`
work). With integration tests landed, v0.4.4 returns to its
long-standing slot from the cust-design.md §17 roadmap: the
V31D-3 multi-bin deferral.

Multi-bin is a self-contained schema + emission change:

1. **No new contracts with the registry/lock work.** V31D-10
   already established that the bin-vs-lib distinction is a
   property of a member's `Cust.toml`, not the resolved graph.
   Multiple bins are *more* of the same — `Cust.lock` is
   untouched (V44D-12).
2. **Reuses the v0.4.2 CMake backend wholesale.** The emitter
   already produces one `add_executable` for the single bin;
   v0.4.4 makes that a loop over a `Vec<BinView>`. No plugin
   changes, no `tests/` interaction beyond V43D-3 staying put.
3. **Independently shippable in either order vs. v0.4.5+.**
   Multi-bin doesn't block the v0.4.5 `cust test` follow-ups,
   and they don't block it.

The one cross-cutting constraint: a multi-bin member's
**library** half is still the only thing unit-tested (V32D-11)
and integration-tested (V43D-3). Bin halves — single or
multiple — are never the crate-under-test. V44D-9 only
syntax-checks them; it does not test them.

## 4. Boundary with v0.3.1 bins & v0.4.2 CMake backend

v0.3.1 (V31D-1) resolves a crate to one of three `CrateKind`
shapes — `Lib`, `Bin { bin_source }`, `LibAndBin { lib_source,
bin_source }` — each carrying **at most one** bin source. The
v0.4.2 CMake backend (`cmake_emit::MemberView`) mirrors that
with single `bin_sources` / `bin_target_name` /
`runtime_output_dir` fields and one `emit_executable` call.

v0.4.4 makes the bin half plural:

```text
manifest::CrateKind            v0.3.1            v0.4.4
  Bin     { bin_source }   →   Bin     { bins: Vec<BinTarget> }
  LibAndBin{ …, bin_source}→   LibAndBin{ …, bins: Vec<BinTarget> }
```

where `BinTarget { name: String, source: PathBuf }`. A
single-bin crate is just `bins.len() == 1` — every v0.3.1
codepath becomes the `len == 1` special case, so behaviour is
byte-identical for existing crates (verified by the unchanged
cwork golden in slice B until the dogfood bin is added).

The v0.4.2 CMake emission (`emit_executable`) becomes a loop:

```cmake
# ---- crate: hello-cstd (bin: hello-cstd) ----
add_executable(hello-cstd "…/src/main.c")
set_target_properties(hello-cstd PROPERTIES
    RUNTIME_OUTPUT_DIRECTORY "…/target/debug" OUTPUT_NAME hello-cstd)
target_link_libraries(hello-cstd PRIVATE cstd)
# …compile options, include dirs (shared across this member's bins)…

# ---- crate: hello-cstd (bin: bench) ----
add_executable(hello-cstd__bin__bench "…/src/bin/bench.c")
set_target_properties(hello-cstd__bin__bench PROPERTIES
    RUNTIME_OUTPUT_DIRECTORY "…/target/debug" OUTPUT_NAME bench)
target_link_libraries(hello-cstd__bin__bench PRIVATE cstd)
# …same shared options…
```

CMake **target names** are workspace-unique, so each bin needs
a distinct identifier even though the on-disk `OUTPUT_NAME` is
the plain bin name. v0.3.1 already solved the lib+bin clash by
suffixing the package bin `<name>-bin` (`bin_target_name`
field). v0.4.4 generalises to `<crate>__bin__<name>` for every
non-default bin and keeps the v0.3.1 rule for the package bin
(`<name>` pure-bin, `<name>-bin` lib+bin) to preserve the
golden file shape for existing single-bin members. See V44D-5
for the exact naming scheme.

The plugin needs **zero changes**: bins have never carried
surface extraction or test discovery, and that stays true.

## 5. Decisions (V44D-N)

### V44D-1 — Discovery: `src/bin/*.c` top level, recursion forbidden

**Proposed: A. Auto-discover one bin per `.c` file directly
under `<crate>/src/bin/`**, named after the file stem. Mirrors
Cargo's `src/bin/*.rs` and the v0.4.3 V43D-1 `tests/*.c` walk:

* Only the **top level** of `src/bin/` is scanned. A
  `src/bin/sub/foo.c` is silently ignored (no error, no
  target) — reserved for the V44D-2 subdirectory form.
* Non-`.c` files are ignored.
* Discovery order is **sorted by stem** (deterministic CMake
  emission, V42D-4).
* `src/bin/` is independent of `src/main.c`: a crate may have
  `src/main.c` (package bin) **and** `src/bin/*.c` (extra
  bins), or `src/bin/*.c` with no `src/main.c` (bins but no
  package-named bin), or neither.

The union of {`src/main.c` if present} ∪ {`src/bin/*.c`} ∪
{explicit `[[bin]]` entries} is the member's bin set.

### V44D-2 — `src/bin/<name>/main.c` subdirectory form — **DEFERRED**

**Deferred to a later v0.4.x.** Cargo supports multi-file bins
as `src/bin/<name>/main.c` (the directory name is the bin
name). v0.4.4 keeps the flat `src/bin/<name>.c` form only.
Rationale (mirrors v0.4.3 V43D-2): the flat form covers every
dogfood case; the subdirectory form opens module-discovery
questions (does `src/bin/<name>/` get its own `#cust use
crate::<mod>` scan?) that don't have to be answered now. A
`src/bin/<name>/` directory is silently ignored in v0.4.4
(consistent with V44D-1 no-recursion).

### V44D-3 — Bin naming, validity, and collisions

**Proposed:**

* The `src/main.c` bin is named after `[package].name`
  (unchanged from V31D-1).
* A `src/bin/<stem>.c` bin is named `<stem>`.
* A `[[bin]]` entry is named by its required `name` field
  (V44D-4).
* **Stem/name validity** reuses the v0.4.3 V43D-8 rule:
  `[A-Za-z][A-Za-z0-9_-]*`. A leading digit, empty name, or
  disallowed character is a hard manifest/discovery error with
  the offending path.
* **Duplicate bin names are an error.** Two bins resolving to
  the same name — whether from `src/bin/foo.c` + `[[bin]]
  name = "foo"` without a path (collision, not override — see
  V44D-4), or two `[[bin]]` entries with the same `name` — are
  rejected:

  ```
  error: package `hello-cstd` defines binary `bench` more than
    once (src/bin/bench.c and [[bin]] name = "bench"); give the
    [[bin]] entry a `path` to override, or rename one.
  ```

* **`src/bin/<pkg>.c` shadowing the package bin is an error**
  when `src/main.c` also exists (both would be named `<pkg>`).
  With no `src/main.c`, `src/bin/<pkg>.c` is allowed — it
  simply *is* the package-named bin.

### V44D-4 — `[[bin]]` array schema (relaxing V31D-3)

**Proposed: `[[bin]]` becomes a real array; each entry is
`{ name, path? }`.** The v0.3.1 `Bin { path }` struct gains a
`name: Option<String>` field:

```toml
[[bin]]
name = "bench"
path = "src/tools/bench.c"   # optional; default below

[[bin]]
name = "dump"                # path defaults to src/bin/dump.c
```

Resolution rules:

* `name` **omitted** is allowed only for a **single** `[[bin]]`
  entry, where it defaults to `[package].name` (exact v0.3.1
  back-compat — existing single-`[[bin]]` manifests parse
  unchanged). Two or more entries each require an explicit
  `name`; an omitted `name` with ≥2 entries is an error.
* `path` **omitted** defaults to `src/bin/<name>.c`, except the
  package-named entry defaults to `src/main.c` (V31D-1 parity).
* An explicit `[[bin]]` entry whose `name` matches an
  auto-discovered `src/bin/<name>.c` and **supplies a `path`**
  *overrides* the auto-discovered path (Cargo behaviour). With
  **no `path`**, it collides (V44D-3 duplicate error) — the
  user must either drop the redundant entry or point it
  elsewhere.
* A `[[bin]]` with an explicit `path` that doesn't exist on
  disk is an error (V31D-1 parity: explicit ⇒ required).

`#[serde(deny_unknown_fields)]` stays; `name` + `path` are the
only accepted subkeys (no per-bin `[dependencies]`/`[clang]` —
V44D-11).

### V44D-5 — One CMake target per bin; naming scheme

**Proposed: one `add_executable` per bin**, target-named to
stay workspace-unique while keeping the v0.3.1 golden shape for
existing single-bin members:

| bin role | CMake target name | `OUTPUT_NAME` (on disk) |
|---|---|---|
| package bin, pure-bin member | `<crate>` | `<crate>` |
| package bin, lib+bin member | `<crate>-bin` | `<crate>` |
| `src/bin/<n>.c` / `[[bin]] name=<n>` | `<crate>__bin__<n>` | `<n>` |

The first two rows are exactly v0.3.1's `bin_target_name`
rule, so a single-bin member emits **byte-identical**
`CMakeLists` to v0.4.3 (regression-guarded by the unchanged
golden in slice B). Every non-package bin gets the
`<crate>__bin__<n>` prefix — same disambiguation pattern as
v0.4.3's `<crate>__itest__<stem>` (V43D-5), so the two
namespaces (`__bin__` / `__itest__`) never clash.

Bins are **not** `EXCLUDE_FROM_ALL` (unlike test/itest
targets): `cust build` builds every bin by default, matching
Cargo. `cust build --bin <n>` (V44D-7) lowers to `cmake
--build --target <that-one-target>`.

### V44D-6 — `cust run --bin <name>` selection

**Proposed: add a `--bin <NAME>` flag to `cust run`.** Member
resolution is unchanged (`-p` picks the member; bare `cust run`
uses the V31D-8 sole-member rule). The bin-selection rules
below then apply **unconditionally** — whether or not `-p` was
given — which is the key change from v0.3.1, where the no-`-p`
path ran the sole bin-member's single executable directly
without any bin-count check ([cli.rs](../../cust/src/cli.rs)
`run` no-`-p` branch). After the member is resolved:

1. `--bin <n>` given ⇒ run that bin; error if the resolved
   member has no bin named `<n>` (list available names sorted).
2. No `--bin`, member has exactly **one** bin ⇒ run it
   (v0.3.1 behaviour preserved).
3. No `--bin`, member has **>1** bin ⇒ the Cargo
   "could not determine which binary to run" error from §1,
   listing all bin names sorted, exit non-zero.

The `--bin <n>` flag may be used **without** `-p`: it resolves
the member by searching for the (unique) member that owns a bin
named `<n>`. If two members each own a bin named `<n>`, that's
an "ambiguous across packages — pass `-p <member>`" error
(symmetric with V44D-7). `cust run -p <lib-only-member>` stays
the V31D-7 error (no bins at all).

### V44D-7 — `cust build --bin <name>` scoping

**Proposed: add `--bin <NAME>` to `cust build`**, scoping the
Ninja build to that one bin's CMake target (and its transitive
lib deps, which CMake pulls automatically). Without `--bin`,
`cust build [-p <member>]` builds all of the member's bins +
lib (current behaviour). `--bin <n>` resolution mirrors V44D-6
exactly:

* With `-p <member>`: `<n>` must name a bin in that member;
  otherwise the "available binaries" error.
* Without `-p`: search all members for the unique owner of a
  bin named `<n>`; a name matching bins in two members is an
  "ambiguous across packages — pass `-p <member>`" error.

The resolved bin lowers to `cmake --build --target
<crate>__bin__<n>` (or the unsuffixed `<crate>` / `<crate>-bin`
target for a package bin, per V44D-5).

### V44D-8 — `CrateKind` carries `Vec<BinTarget>`

**Proposed:** replace the single `bin_source: PathBuf` in the
`Bin` / `LibAndBin` variants with `bins: Vec<BinTarget>` where
`BinTarget { name: String, source: PathBuf }`. `resolve_kind`
builds the vec from the V44D-1/V44D-4 union, sorted by name,
after running the V44D-3 validity + collision checks. A
`Bin`/`LibAndBin` always has `bins.len() >= 1` (an empty bin
set resolves to `Lib` or the "no source" error, unchanged).
Downstream consumers (`workspace::Member`, `cmake_emit`,
`build`, `cli`) iterate the vec; the `len == 1` path reproduces
v0.3.1 exactly.

Two concrete struct changes ride on this and are **not**
optional plumbing — `cust run --bin` depends on them:

* **`build::BuildOutputs.executable: Option<PathBuf>`**
  ([build.rs](../../cust/src/build.rs)) becomes plural —
  `executables: Vec<(String, PathBuf)>` keyed by bin name
  (or equivalent map). One member can now emit N exes.
* **`cust run`'s exe-location step**
  ([cli.rs](../../cust/src/cli.rs) `run`, the
  `find_map(... out.executable)` lookup) selects by the
  V44D-6-resolved bin name instead of grabbing the member's
  sole executable. The `Finished … -> <exe>` reporting loop
  iterates the new vec.

* **`build::write_rewrite_tree`**
  ([build.rs](../../cust/src/build.rs)) currently discovers a
  single bin module graph from `kind.bin_source()`; it becomes
  a loop over `bins`, discovering one module graph per
  `BinTarget.source` (each `src/bin/<n>.c` is its own root
  that may `#cust use crate::<mod>`). The matching
  `cmake_emit::collect_view` bin-source collection
  (today one aggregated `bin_sources`) becomes per-bin
  `BinView`s (slice B).

### V44D-9 — `cust check` over every bin TU — **DEFERRED**

**Deferred (revised at slice C).** Originally locked as:
`cust check` runs `-fsyntax-only` over each `BinTarget.source`,
the same way v0.3.1 V31D-9 supposedly did for the single bin.
Implementation revealed the premise was wrong — the same way
v0.4.3 V43D-13 was. In the v0.4.2+ codebase, `cust check` runs
`build::run_phase1`, which surface-passes the **library half
only** and explicitly skips the bin half ("No surface pass on
the bin half"). It is a *tolerant* lib-surface-extraction pass
(V42D-15 / V43D-13), not a strict syntax checker: a
deliberately broken `src/bin/<n>.c` with `return 1 2 3;` passes
`cust check` clean today (verified live at slice C — exit 0,
`Checked hello-cstd`).

Extending that tolerant pass to bins would deliver none of the
"see syntax errors immediately" value the decision promised,
and making bins *strict* while the lib half stays *tolerant*
would be the same surprising asymmetry V43D-13 rejected. The
correct fix — a strict `cust check` for lib + bin + `tests/` —
is a `cust check` redesign that belongs in its own milestone
(tracked alongside the deferred V43D-13). Until then,
`cust check` ignores bins; bin syntax errors surface at
`cust build` / `cust run` time via the real CMake build, with
full file:line diagnostics from clang.

### V44D-10 — cwork dogfood

**Proposed:** `cwork/examples/hello-cstd` grows a `src/bin/`
with one extra bin (e.g. `src/bin/bench.c`) that consumes
`cstd`'s public surface differently from `main.c`, exercising:
multi-bin discovery, the `<crate>__bin__<name>` target,
`cust run -p hello-cstd --bin bench`, and the ambiguous-bin
error when `--bin` is omitted. Version bumps 0.4.3 → 0.4.4
land in the same slice.

### V44D-11 — Per-bin `[dependencies]` / `[clang]` — **DEFERRED**

**Deferred.** All bins of a member share the member's
`[dependencies]` and `[clang]` config. Cargo's per-target
`required-features` and per-bin dependency scoping are a
resolver concern that belongs with the v0.4.6 registry work,
not this mechanical multi-target milestone. This is the exact
schema question V31D-3 flagged as "don't have to be answered
now"; v0.4.4 still doesn't answer it.

### V44D-12 — `Cust.lock` unchanged

**Proposed: no schema bump.** Per V31D-10, the number and names
of a member's bins are properties of its `Cust.toml`, not the
resolved dependency graph. `Cust.lock` records member name /
version / path / deps — none of which change with multi-bin.
No lock churn.

## 6. Open questions (RQ-V44-N — resolved)

### RQ-V44-1 — `default-run` for `cust run` without `--bin`? — **DEFERRED**

**Deferred.** Cargo lets a package set `default-run = "<name>"`
so `cargo run` picks a default among multiple bins instead of
erroring. v0.4.4 ships only the V44D-6 "ambiguous ⇒ error"
behaviour — the safe default that matches `cargo run` when no
`default-run` is set. `default-run` is additive sugar: it adds
one `[package].default-run` key and one fallback branch ahead
of the V44D-6 error, with no change to any existing codepath,
so it can land in any later milestone without rework. Out of
scope for v0.4.4.

### RQ-V44-2 — `required-features` gating? — **DEFERRED**

**Deferred (no mechanism to build on).** Cargo can gate a bin
behind feature flags. cust has no feature system yet — features
are deferred with the v0.4.6 resolver. There is nothing to
gate, so there is nothing to design here. `required-features`
rides in with whenever features land; v0.4.4 does not
pre-design it.

### RQ-V44-3 — Unit tests inside `src/bin/*.c`? — **DEFERRED**

**Deferred (keep the V32D-11 exclusion).** v0.3.2
V32D-11/V32D-12 already exclude bin halves from `cust test`;
v0.4.4 keeps that exclusion unchanged for every bin, single or
multiple. `[[cust::test]]` functions in `src/bin/<n>.c` are not
discovered. This is the simplest option (no new code, zero
plugin change — §3) and preserves the existing invariant that
test coverage lives in the lib half + `tests/`. Discovering
bin-internal tests (Cargo does compile `#[test]` in bins) is a
follow-up that can lift the exclusion later without breaking
anything; v0.4.4 does not take it on.

## 7. Verification target

The v0.4.4 closeout (slice E) must demonstrate:

1. A member with `src/main.c` + `src/bin/bench.c` + a
   `[[bin]] name = "dump"` produces three executables at
   `target/<profile>/{<crate>,bench,dump}`, all from one
   `cust build`.
2. `cust run -p <member>` with >1 bin and no `--bin` exits
   non-zero with the §1 ambiguous-binary error listing all bin
   names sorted; `cust run -p <member> --bin bench` runs the
   right one and forwards `-- <argv>`. **Bare `cust run`** (no
   `-p`) in a workspace whose sole bin-member has >1 bin also
   hits the ambiguity error (Gap from v0.3.1's
   run-the-sole-exe path — V44D-6 rules apply unconditionally).
   `cust run --bin bench` with no `-p` resolves the owning
   member; a bin name owned by two members is the
   "ambiguous across packages" error.
3. `cust build --bin bench` rebuilds only `bench` (Ninja
   incremental); the other bins' exes are untouched.
4. A single-bin member (existing v0.3.1 shape) emits
   **byte-identical** `CMakeLists` to v0.4.3 — verified by an
   unchanged golden diff before the dogfood bin is added
   (V44D-5 regression guard).
5. Collision errors fire: `src/bin/<pkg>.c` next to
   `src/main.c` (V44D-3), duplicate `[[bin]] name` (V44D-4),
   leading-digit / bad-char stem (V44D-3 / V43D-8 reuse).
6. `[[bin]]` with an explicit `path` overrides the
   auto-discovered `src/bin/<name>.c`; without a `path` it
   collides (V44D-4).
7. `src/bin/sub/foo.c` and `src/bin/<name>/main.c` are
   silently ignored (V44D-1 no-recursion + V44D-2 deferral):
