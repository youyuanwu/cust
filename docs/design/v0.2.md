# cust v0.2 — module loader & plugin v0

**Status:** 🚧 **planning** (next milestone after v0.1).
**Parent doc:** [cust-design.md §17 roadmap](cust-design.md).
**Previous milestone:** [v0.1.md](v0.1.md) (shipped).

This file expands the one-line v0.2 roadmap bullet from
[cust-design.md](cust-design.md) into a working plan. Architectural
detail for individual subsystems lives in the parent design doc; this
file tracks **scope, sequencing, and open v0.2-internal decisions**.

---

## Scope

The big rocks, in suggested implementation order:

1. **Driver pre-pass for `#cust mod` / `#cust use` / `[[cust::cfg]]` /
   `[[cust::feature]]` / `[[cust::pub_macro]]`** — see [cust-design.md
   §4](cust-design.md) and [§9](cust-design.md) for the spec.
   Implementation strategy: Rust line scanner (decided, see V2D-1
   below).
2. **Multi-module scheduler (single-threaded).** Walk the module tree
   from `src/lib.c`, compile each `.c` to its own `.o` under
   `target/<profile>/build/<crate>/<mod>.o`, archive *all* `.o`s into
   `lib<name>.a`. `-jN` parallelism waits for v0.4.
3. **macOS support.** Mostly CI matrix expansion + confirming
   `llvm-ar`/`ar` discovery on macOS. Should "just work" since v0.1
   already shells out via `Command` with no Linux-isms. Lands here so
   the rest of v0.2 doesn't accumulate Linux-only assumptions.
4. **Plugin (`libcust_plugin.so`) v0.** New workspace member
   `plugin/` (C++); build script that invokes `cmake --build` (CMake
   is permitted **only** as the plugin's own build system, never in
   the user-facing hot path — see [cmake-and-portability.md](cmake-and-portability.md)).
   `clang::PluginASTAction` that emits per-module `.cust.h` fragments
   into `target/<profile>/.h-fragments/<crate>/`. Two-phase pipeline
   (surface → codegen) with fragment-header stamping and per-crate
   `target/` locking ([cust-design.md §4](cust-design.md), [§10](cust-design.md)).
5. **Naive crate-header concatenation** of the `[[cust::pub]]` subset
   into `target/<profile>/include/<crate>.h` ([cust-design.md §5](cust-design.md)).

Already shipped (originally v0.2, pulled forward to v0.1):

* `cust new` — see [v0.1.md "Deviations from spec"](v0.1.md).

## v0.2 design decisions

All four v0.2-internal decisions are settled. Recorded here for the
implementation PRs to point at.

### V2D-1: How does the pre-pass recognise `#cust …;` directives? — **decided: line scanner**

**Status:** ✅ decided (2026-06-03). Picks option A below.

We will **never** run a real C preprocessor in the pre-pass. The
hypothetical clang `-E` alternative only buys us the ability to put
`#cust mod` behind `#ifdef`, which the spec already says no to
([cust-design.md §4 "Features and modules"](cust-design.md)).

The Rust line scanner is **not throwaway** — it's the right shape
forever. Lifecycle:

* **v0.2:** ~150 lines, handles `#cust mod` + `#cust use`.
* **v0.3:** +50 lines for `[[cust::pub_macro]]` extraction
  ([cust-design.md §9](cust-design.md)) — `#define` is line-oriented
  with `\\\n` continuations; same scanner with a parser extension.
* **v0.4+:** +100 lines for module-level `[[cust::cfg]]` (see
  "Placement constraints" below).
* **Later:** unchanged. Anything richer goes to the plugin, which
  has the parsed AST and can drop nodes by attribute.

Build the v0.2 scanner with the invariants we want forever rather
than retrofitting them later:

1. **Real comment + string-literal state machine** from day 1 —
   3 states (code / line-comment / block-comment), `\\\n` line
   continuation, string + char literal recognition. Not regex.
2. **Real tokeniser for the `#cust …;` directive body** — so
   adding `[[cust::pub_macro]]` later is a parser extension, not a
   rewrite.
3. **Position-preserving rewrites** — every byte in the rewritten
   output maps back to a `(file, line, column)` in the source, so
   clang diagnostics in rewritten files still point at the user's
   code (`#line` directives are the standard tool).

Recognition rules (v0.2):

* `#cust <directive> …;` is recognised only when `#` is at column 0
  with no leading whitespace.
* Anything that looks like `#cust …` but isn't a directive the
  scanner recognises is a hard error, not a silent pass-through.
* The scanner ignores anything inside `#if 0` / `#ifdef` blocks
  *only insofar as it ignores nothing* — `#cust` is outside the C
  preprocessor's view. If a user writes `#cust mod` inside `#if 0`,
  the scanner still sees it and processes it. Documented loudly.

#### Placement constraints (v0.4 forward look)

Module-level `[[cust::cfg(...)]]` needs the scanner to find the
*end* of the gated decl. Rather than parse C, constrain placement:

* `[[cust::cfg]]` is legal on (a) `#cust mod` lines, (b) any decl
  ending in `;` at the top level, (c) brace-balanced
  struct/enum/union/function-definition blocks at the top level.
* The scanner finds the end with a `{ }` / `;` counter that
  respects strings, char literals, and comments (already required
  by invariant 1).
* Anything richer (cfg on a single statement inside a function
  body, cfg inside an `enum` body, etc.) is handled by the plugin
  at the AST level, not the pre-pass.

This is a v0.4 design note recorded here so the v0.2 scanner's
internal API doesn't paint us into a corner.

### V2D-2: Single workspace member for the plugin, or separate? — **decided: non-Cargo subdir**

**Status:** ✅ decided (2026-06-03). Picks option A below.

The plugin is C++ + CMake. Options for repo placement:

* **A — `plugin/` as a non-Cargo subdirectory** with a Rust
  build-time helper crate (`plugin-build/`) that shells out to
  `cmake --build`.
* **B — `plugin/` as a workspace member with `build.rs` that drives
  CMake**, exposing the resulting `.so` path via `cargo:rustc-env`.

A wins because CMake belongs at arm's length —
[cmake-and-portability.md](cmake-and-portability.md) is explicit that
CMake is never in the *user-facing* build hot path; tucking it behind
a non-Cargo subdir + a thin Rust shim keeps the boundary obvious in
the repo layout. The plugin's build cadence is also slower than the
driver's, so coupling it to every `cargo build` (option B) would
slow the inner loop for no benefit.

Concrete v0.2 shape:

* `plugin/CMakeLists.txt` + `plugin/src/*.cc` — standalone CMake
  project producing `libcust_plugin.so`.
* `plugin-build/` — small Rust crate exposed only as a workspace
  build helper; called explicitly via `cargo run -p plugin-build`
  (or a `cust bootstrap-plugin` subcommand if the ergonomics
  warrant). Not run automatically as part of `cargo build`.
* Discovery at driver runtime: the plugin path is looked up in
  `$CUST_PLUGIN` env var, falling back to
  `target/<profile>/libcust_plugin.so`. Missing-plugin error
  message points at `cargo run -p plugin-build`.

### V2D-3: Fragment-header convergence cap in v0.2 — **decided: 3, unchanged**

**Status:** ✅ decided (2026-06-03). Keep the spec value.

[cust-design.md §4](cust-design.md) commits to a cap of 3 iterations
with a clear divergence diagnostic. v0.2 ships that exact behaviour
verbatim — no spec change. If real-world `pub(repr)` graphs surface
legitimate 4+-iteration convergence, the spec already says raise the
cap rather than reject (the cap is a divergence detector, not a
complexity limit).

### V2D-4: Module name → file path mapping precedence — **decided: error**

**Status:** ✅ decided (2026-06-03). Picks option A below.

Spec: `#cust mod foo;` resolves to `foo.c` *or* `foo/mod.c`. v0.2
behaviour when **both** exist:

* **A — error.** "ambiguous: both `foo.c` and `foo/mod.c` exist".
* **B — prefer `foo/mod.c`.** Folder modules win, matching the
  intuition that a folder is a "bigger" thing.
* **C — prefer `foo.c`.** File modules win, matching the intuition
  that an explicit named file is more specific.

A wins. The cost of erroring is one rename; the cost of silently
picking the wrong one is a debugging session. Cargo has the same
rule for `foo.rs` vs `foo/mod.rs`. Diagnostic:

```
error: ambiguous module `foo` declared in src/lib.c:12 resolves to
  both `src/foo.c` and `src/foo/mod.c`
  hint: keep exactly one; delete the other or move its contents
```

## Verification target

A v0.2 build should produce, given a multi-module crate:

* `target/<profile>/build/<crate>/<each_mod>.o`
* `target/<profile>/.h-fragments/<crate>/<each_mod>.cust.h`
* `target/<profile>/include/<crate>.h` (concatenated `[[cust::pub]]`
  surface)
* `target/<profile>/lib<name>.a` (all objects archived)
* `target/compile_commands.json` (one entry per TU)

…on Linux *and* macOS. Integration test fixtures under
`cust/tests/fixtures/` grow with: a 2-module crate, a 3-module crate
with a `pub`/`pub(crate)` mix, a `pub(repr)` cycle that converges in
2 iterations, and a `pub(repr)` cycle that diverges (expected to
error with the §4 convergence diagnostic).
