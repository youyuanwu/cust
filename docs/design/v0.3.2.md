# cust v0.3.2 — `cust test` (Rust-style unit tests)

**Status:** ✅ **shipped** (2026-06-04).
**Parent doc:** [cust-design.md §17 roadmap](cust-design.md).
**Previous milestone:** [v0.3.1.md](v0.3.1.md) (shipped).

> **Subsequent history (2026-06-05):** v0.4.0 (V40D-6, V40D-10)
> retired the v0.3.2 driver pre-pass test scanner and the macro
> spellings it recognised. The plugin became the only discovery
> backend; the `--no-plugin` flag was redefined as a syntax-only
> escape hatch with no discovery promises. Several promises this
> document made about "the pre-pass stays in tree as the
> `--no-plugin` discovery path forever" (V32D-2 sidebar; below)
> are accordingly **revoked** — see [v0.4.0.md](v0.4.0.md)
> V40D-6 / V40D-10 / V40D-7 for the reasoning and the
> sentinel-marker enforcement mechanism that replaced the
> dual-backend design.

This file is the v0.3.2 record — scope, locked design decisions
(**V32D-N**), and the verification target. Like v0.3.1, this is
a small single-purpose milestone wedged between two larger ones:
v0.3.1 (binary crates) just landed; v0.4 (registry, build
scripts, plugin v1, parallelism) is the next big rock.

### Why a 0.3.2, not part of v0.4

v0.4 already carries six big-ticket items (registry, `cust add`,
semver, build scripts, plugin v1, `-jN`). Bolting the full
`[[cust::test]]` story onto v0.4 doubles the milestone, *and* couples
the test harness to plugin v1 — which is the wrong dependency
direction. Cargo's `cargo test` story is independently useful and
testable without the registry, and an early test harness gives us
something v0.4 itself can lean on (the registry / resolver / plugin
v1 are exactly the kind of code that benefits from a working unit-
test loop).

The trick: v0.4's [cust-design.md §10](cust-design.md) jobs #3
("test discovery") and §11 (`cust test`) plan to collect tests via
the *plugin*. Plugin v1 doesn't exist yet. v0.3.2 therefore lands
the *same observable contract* (`cust_test` macro, `__cust_tests[]`
table, fork-isolated runner, `cust test` CLI) using a
**driver pre-pass** as the discovery backend — exactly mirroring
how `[[cust::pub_macro]]` is owned by the driver pre-pass today
(§9). When plugin v1 lands in v0.4, it joins as a *second*
discovery backend behind the same `__cust_tests[]` interface (see
V32D-2); the pre-pass stays in tree as the `--no-plugin` path.
Everything else (macro expansion, runner template, fork harness,
CLI) stays put across the transition.

This split also means the milestone is mechanical: it touches
manifest (test profile bits), build pipeline (test-build variant),
a new pre-pass scanner, a generated runner TU, a new `cust test`
subcommand, and the prelude (test/assert macros). No new contracts
that v0.4 can invalidate.

### Why v0.3.2 ships before plugin v1 (not after)

The natural counter-question to V32D-2: if plugin v1 is the
"correct" home for test discovery, why not build plugin v1
*first* and skip the driver pre-pass scanner entirely?

The call is closer than the rest of v0.3.2 implies, but it lands
on "v0.3.2 first":

1. **Plugin v1 is much bigger than the test harness.** Per
   [cust-design.md §10](cust-design.md), plugin v1 owns five
   jobs simultaneously: full `[[cust::pub(repr)]]` body export
   (struct / enum / union pretty-printing with bitfields,
   alignment, nested and anonymous types — all the corner
   cases plugin v0 sidesteps by emitting opaque tags only),
   AST-based test discovery, derive codegen for `Eq` / `Hash`
   / `Debug`, per-TU semantic checks (`must_use` type-level,
   `no_panic` reachability, `const` constexpr-equivalence,
   `unsafe` call-site enforcement, `repr` validation), and
   phase-isolation enforcement with the
   `ProgramAction == ParseSyntaxOnly` invariant plus the §4
   fragment-stamping fixed-point loop. Plugin v0 was the
   entire content of v0.2; plugin v1 is 2–3× that. Pulling it
   forward turns v0.3.2 into "v0.4 starts now" and loses the
   "small milestone wedged between two big ones" shape that
   v0.3.1 proved out.

2. **Plugin-v1-first does not shrink v0.3.2 by much.** Even
   with plugin v1 in hand, the v0.3.2 plan still has to write:
   the runner template (`cust_test_run`, fork loop, output
   formatter, exit codes), the `cust test` CLI subcommand,
   workspace iteration, the test-build variant in `build.rs`
   (`-DCUST_TEST_BUILD=1`, fresh `target/<profile>/test/<crate>/`
   tree), the cstd dogfood tests, the V32D-11 bin-only error,
   and the V32D-12 workspace-bare semantics. That's 5 of the
   6 v0.3.2 slices. Only slice B (`test_scanner.rs`) is the
   part you'd skip — and that's ~150 LOC of regex modeled on
   the existing `mod_scanner.rs` plus ~100 LOC of unit tests.
   Trading ~250 LOC of bridge code for pulling forward a
   milestone 2–3× the size of v0.2 is not a good ratio.

3. **The pre-pass is not throwaway code.** This is the V32D-2
   reframing — the pre-pass scanner is a *first-class*
   implementation of the discovery contract, not scaffolding
   to be deleted. When plugin v1 ships, the pre-pass becomes
   the `cust check --no-plugin` (§9 "Fallback when the plugin
   is not loaded") backend for test discovery, plus the
   `--no-plugin` discovery backend that we ship forever for
   downstream consumers who don't load `libcust_plugin.so`.
   Same lifecycle as the v0.1 `cust_pub` macro-only fallback,
   which is still in the prelude alongside v0.2's plugin
   surface extraction — both paths coexist permanently.

4. **Tests-first accelerates plugin v1 development.** The cust
   *driver* is tested with `cargo test` already, but
   cust-the-language has zero unit-test story today. Every
   subsequent milestone — including plugin v1 itself — benefits
   from being able to write `cust_test int test_resolve_dep(void)`
   against driver-internal C testbed crates. If plugin v1
   ships before `cust test`, plugin-v1 development happens
   without `cust test` to validate its outputs. If `cust test`
   ships first, plugin v1 *uses* it from day one (the cstd
   tests written for v0.3.2 become a regression suite that
   plugin v1's AST discovery has to keep passing — the test
   suite literally validates the swap).

5. **Plugin v1 is the highest-risk v0.4 item.** Unknown clang
   API surface area, ABI churn risk, the §10 phase-isolation
   invariant to nail, the §4 fixed-point loop to wire up.
   Front-loading it means v0.4's other rocks (registry, build
   scripts, `-jN`) wait on the riskiest one. v0.3.2-first
   keeps plugin v1 inside the v0.4 batch where the
   surrounding rocks can absorb the risk schedule, and lets
   v0.3.2 ship a useful capability on a predictable timeline.

**Trade-off accepted:** the regex-based scanner has a real
user-visible limitation (V32D-2). Documented in the prelude and
shipped knowingly. v0.4 plugin v1 lifts it. None of v0.3.2's
other contracts are revisited by that swap.

```sh
     Running cwork/target/debug/test/cstd

```

Plus the user-facing macros, lifted into the prelude:

```c
#cust use cstd;

cust_test int test_max_i32(void) {
    cust_assert_eq(cstd_max_i32(3, 7), 7);
    cust_assert_eq(cstd_max_i32(-1, -2), -1);
    return 0;
}

cust_test void test_strlen_hello(void) {
    cust_assert(cstd_strlen("hello") == 5);
}
```

The shape mirrors Rust's `#[test] fn test_foo() { assert_eq!(...); }`
beat-for-beat: function-attribute marker, in-source colocated tests,
`assert_*` family that aborts the running test on failure, per-test
isolation so one failed test doesn't take the rest down.

## Scope

Implementation order matters here — earlier items unblock later
ones. Locked **V32D-N** decisions appear below the scope list.

1. **Prelude additions.** `cust_test`, `cust_test_ignore`,
   `cust_panic(msg)`, `cust_assert(expr)`, `cust_assert_eq(a, b)`,
   `cust_assert_ne(a, b)`. `cust_test` expands to one of two forms
   depending on a `CUST_TEST_BUILD` define injected only during
   the test variant build:

   ```c
   #ifdef CUST_TEST_BUILD
   #  define cust_test         __attribute__((annotate("cust::test")))
   #  define cust_test_ignore  __attribute__((annotate("cust::test_ignore")))
   #else
   #  define cust_test         __attribute__((unused)) static
   #  define cust_test_ignore  __attribute__((unused)) static
   #endif
   ```

   Outside a test build, `cust_test`-annotated functions become
   `static unused` — the compiler still type-checks them (so
   tests can't bitrot), but they cost zero bytes in the release
   archive and don't pollute the public surface.

   The assertion macros are defined alongside, also gated on
   `CUST_TEST_BUILD` so they vanish from non-test builds and
   leave no `cust_panic_impl` undefined-reference at link time:

   ```c
   #ifdef CUST_TEST_BUILD
   _Noreturn void cust_panic_impl(const char *file, int line, const char *msg);
   #  define cust_panic(msg)         cust_panic_impl(__FILE__, __LINE__, (msg))
   #  define cust_assert(e)          ((e) ? (void)0 : cust_panic("assertion failed: " #e))
   #  define cust_assert_eq(a, b) \
         ((a) == (b) ? (void)0 : cust_panic("assertion failed: `(" #a ") == (" #b ")`"))
   #  define cust_assert_ne(a, b) \
         ((a) != (b) ? (void)0 : cust_panic("assertion failed: `(" #a ") != (" #b ")`"))
   #else
   #  define cust_panic(msg)         ((void)(msg))
   #  define cust_assert(e)          ((void)0)
   #  define cust_assert_eq(a, b)    ((void)0)
   #  define cust_assert_ne(a, b)    ((void)0)
   #endif
   ```

   `cust_panic_impl` is **not** in the prelude — it is defined
   inside the generated runner TU (`cust_test_main.c`, scope
   item 4), so the symbol exists exactly when `CUST_TEST_BUILD`
   is set. (**V32D-1**, **V32D-3**.)

2. **Test discovery — driver pre-pass.** A new
   `cust/src/test_scanner.rs` walks the same module set
   `mod_scanner` already produces and matches the line-oriented
   regex
   `^\s*cust_test(?:_ignore)?\s+(int|void)\s+([A-Za-z_]\w*)\s*\(\s*void\s*\)`
   against each module's source. Result: per-crate
   `Vec<TestEntry { module, name, fn_kind, ignored, file, line }>`.
   The shape and constraints mirror `mod_scanner` / pre-pass §9
   exactly. (**V32D-2**.)

3. **Test-build variant.** When `cust test` runs against a member,
   the build pipeline runs in a **fresh `target/<profile>/test/<crate>/`
   tree** (NOT alongside the lib build — keeps Cargo's "test build
   is a different unit" invariant) with `-DCUST_TEST_BUILD=1` added
   to every TU's flags. The library's own per-TU objects are
   recompiled in this variant; the resulting test executable is at
   `target/<profile>/test/<crate>` (no extension, Cargo parity).
   (**V32D-4**, **V32D-5**.)

4. **Generated runner TU.** A driver-emitted
   `target/<profile>/test/<crate>/cust_test_main.c` is concatenated
   from four parts:

   * A small fixed preamble declaring the runner's table shape:

     ```c
     enum cust_test_fn_kind { CUST_TEST_FN_VOID, CUST_TEST_FN_INT };
     struct cust_test_entry {
         const char *qname;       /* "module::name" */
         void       *fn_ptr;      /* cast to int(*)(void) or void(*)(void) per fn_kind */
         enum cust_test_fn_kind fn_kind;
         int         ignored;     /* 1 if cust_test_ignore-marked */
         const char *file;
         int         line;
     };
     ```

   * `extern` forward decls for each discovered test fn.
   * A constant `__cust_tests[]` array of `struct cust_test_entry`,
     one entry per discovered test, sorted by `qname` for stable
     output.
   * A canned `cust_test_run(argc, argv, tests, n_tests)` body
     plus `int main(int argc, char **argv) { return
     cust_test_run(argc, argv, __cust_tests, N); }`, and the
     `cust_panic_impl` definition referenced by the prelude
     assertion macros (writes the message + `file:line` to
     stderr, then `_exit(101)`).

   The runner body is itself a driver template (not a shipped
   runtime crate) — the same template the v0.4 plugin-discovery
   backend will reuse unchanged, just fed a `__cust_tests[]`
   built from the AST instead of the regex scanner. (**V32D-6**.)

5. **Fork-isolated runner.** `cust_test_run` does:

   1. Parse a single positional `<filter>` and the flag
      `--list` (Cargo parity), forwarded from `cust test [filter]
      [-- --list]`.
   2. For each entry in `__cust_tests[]` whose `qname` contains
      `<filter>` (or unconditionally if no filter): if
      `entry.ignored` is set, print `test <qname> ... ignored`
      and bump the `ignored` counter — do **not** fork.
      Otherwise: `fork()`, child casts `fn_ptr` per
      `entry.fn_kind` and calls it, exits 0 on
      `int`-returning-0 or `void`-returning, exits non-zero on
      `int`-returning-non-zero, or aborts via `cust_panic_impl`
      (which writes `<msg>` and `at <file>:<line>` to stderr
      and calls `_exit(101)` — Rust's `cargo test` exit code
      for an assertion failure).
   3. Parent prints `test <qname> ... ok` / `FAILED` per
      forked test, accumulates `passed` / `failed` / `ignored`
      / `filtered_out` counts, prints the closing summary line
      (Cargo-format: `test result: ok. P passed; F failed; I
      ignored[; X filtered out]`).
   4. Process exit code: `0` if all forked tests passed, `1`
      if any failed.

   Tests matching the `--list` flag print one line per entry
   (`<qname>: test`) and exit 0 without forking anything.

   Linux-only — `fork()`, `waitpid()`, `WIFEXITED`/`WIFSIGNALED`,
   `_exit()`. Matches the cust "Linux only until v1" stance.
   (**V32D-7**, **V32D-8**.)

6. **`cust test` subcommand.** New CLI entry:

   ```
   cust test [-p <member>] [--release] [<filter>] [-- <runner-args>]
   ```

   Behaviour:

   * No `-p`: runs tests for every library member (or
     library half of every lib+bin member) in topological order,
     one test binary per member.
   * `-p <member>`: scopes to one member (+ its transitive deps
     for the build, but only the named member's tests run).
   * `<filter>`: forwarded as the first runner argv. Cargo-style
     substring match.
   * `-- <runner-args>`: appended after the filter, so users can
     pass `--list` (the only v0.3.2 runner flag — `--nocapture`,
     `--test-threads`, `--exact` all deferred).
   * Exit code: 0 on full pass across every test binary; 1 on
     any failure or any zero-test crate when invoked with `-p`.
   (**V32D-9**, **V32D-10**.)

7. **Bin-crate tests rejected.** v0.3.2 only tests the *library
   half* of a crate. Lib + bin members: tests run, but only over
   `src/lib.c` + library modules (not `src/main.c`). Bin-only
   members: `cust test -p <bin-only>` errors with

   ```
   error: workspace member `<name>` is a bin-only crate;
     cust test v0.3.2 only runs unit tests in library crates
     (lib+bin members test their library half only)
   ```

   This is **not** a contract — v0.4 will lift it once multi-bin
   per crate lands. (**V32D-11**.)

8. **Workspace bare `cust test`.** Runs every testable member.
   "Testable" = lib or lib+bin. Bin-only members are skipped
   silently (not an error — workspaces mixing bins and libs are
   normal). Topological order, one fresh test binary per crate,
   per-member status header + per-test lines. Final summary
   counts across the whole workspace. (**V32D-12**.)

9. **cstd dogfood.** Add `test_*` functions colocated in
   `cwork/ccrates/cstd/src/math.c` and `cwork/ccrates/cstd/src/mem.c`
   (4–6 tests). `cust test -p cstd` from `cwork/` is the headline
   verification command. Plus one deliberate-failure test gated
   on `CUST_FAIL_DEMO=1` so `examples/` can demonstrate the
   `FAILED` path without flunking CI. (Optional — covered by
   `examples/test-failure-demo/` if it's worth a separate
   example crate.)

10. **Documentation.** Per the milestone routine:

    * New section in [cust-design.md §11](cust-design.md):
      "v0.3.2 implementation — driver pre-pass discovery" subsection
      documenting the two-backend model (V32D-2): pre-pass
      ships in v0.3.2, plugin v1 joins as a second backend in
      v0.4, both live behind the same `__cust_tests[]` contract.
    * [cust-design.md §10 job #3](cust-design.md) correction:
      the existing paragraph mentions "ctor-based registration"
      as the planned v0.4 mechanism. That predates V32D-3;
      both backends actually emit a static `__cust_tests[]`
      table from a driver- or plugin-known list. Rewrite that
      paragraph to match.
    * Roadmap bullet flipped to ✅ shipped on close.
    * v0.3.2.md gets a "Shipped deltas" + "Verification" closing
      section (added at slice-E close, matching v0.3.1.md).

## Decisions (V32D-N — locked)

### V32D-1 — Where do unit tests live?

**Locked: A. Colocated in `src/**.c`** (Cargo parity). A test
function is any `cust_test`-marked function. No `#[cfg(test)] mod
tests { ... }` block — cust doesn't have a `cfg` story yet, and
`cust_test` itself already disables the function in non-test builds
via the `CUST_TEST_BUILD` macro guard (V32D-3).

`cust check` does **not** define `CUST_TEST_BUILD`, so it sees
`cust_test`-marked functions as silent `static unused` and emits
no warning or error for them — they simply aren't part of the
library surface from `cust check`'s perspective. A future
`cust check --tests` (deferred to v0.4) is the explicit
opt-in to syntax-check test code.

Integration tests under `tests/` are **deferred to v0.4**, when the
multi-target layout (`tests/` + `examples/` + `benches/`) lands
together. v0.3.2 is unit-tests-only.

**No `Cust.toml` schema changes in v0.3.2.** v0.3.2 deliberately
adds no manifest surface — no `[[test]]` array, no
`[profile.test]` (deferred to v0.4 with `[[cust::test(inline)]]`
and the timeout work), no new `[package]` keys. Test discovery
is entirely a property of the source files (the `cust_test`
marker) and the active subcommand (`cust test` flips the
`CUST_TEST_BUILD` define).

### V32D-2 — Discovery mechanism: pre-pass vs plugin?

**Locked: A. Driver pre-pass is the *first* discovery backend;
plugin v1 is added as a *second* backend in v0.4.** Both backends
coexist forever — the pre-pass is the permanent home for
`cust check --no-plugin` (§9 "Fallback when the plugin is not
loaded") test discovery and for any downstream consumer who
doesn't load `libcust_plugin.so`. The contract (`cust_test`
macro + `__cust_tests[]` table + fork harness + runner template
+ `cust test` CLI) is identical regardless of which backend
populates the table. The pre-pass scanner shape mirrors
`mod_scanner` / `[[cust::pub_macro]]` extraction (line-oriented,
regex-driven, no AST), which is the same pattern §9 uses for
other attributes that have to live on the driver side
permanently.

Framing this as "two backends behind one stable interface"
(rather than "v0.3.2 stopgap that v0.4 swaps in") is deliberate:

* When plugin v1 lands in v0.4, the swap is *additive* — a new
  `plugin_test_scanner` source that the driver prefers when the
  plugin is loaded — not a replacement. The v0.3.2 pre-pass
  scanner stays in tree.
* It gives `cust check --no-plugin` (a documented v0.5+ flag
  per §9) a working test-discovery path on day one, rather
  than "silent no-op for `[[cust::test]]`" the way other
  fallback-disabled attributes degrade.
* It eliminates the "throwaway scaffolding" framing — the
  pre-pass scanner is a first-class long-lived component, not
  a placeholder.
* It validates the `__cust_tests[]` table contract end-to-end
  before plugin v1 work starts; the v0.4 plugin-discovery
  implementation has a complete regression suite (the cstd
  tests written for v0.3.2) to keep passing.

**Trade-off accepted:** the regex can be fooled by tests written on
unusual lines (e.g. `cust_test\n  int test_foo(void)`). v0.3.2
requires the marker, return type, and name on one line —
documented in the prelude header for `cust_test`. v0.4 plugin v1
adds a second backend that has no such restriction; users who
want AST-grade discovery use it, users who want a plugin-free
build keep the pre-pass restriction. Both paths covered.

### V32D-3 — `cust_test` macro shape?

**Locked: B.** Test build: `__attribute__((annotate("cust::test")))`
(external linkage, plugin annotation that the v0.4 plugin-discovery
backend keys off). Non-test build: `__attribute__((unused)) static`.
The non-test form preserves type-checking (so tests can't bitrot
when the underlying library API changes) without polluting the
release archive or the public crate header — `cust_test`-marked
functions are never `cust_pub`-marked, and the static-unused form
means they won't even reach the linker in a non-test build.

**Why not ctor-based registration today?** Because the v0.4
plugin v1 will *also* avoid the ctor route. [cust-design.md §10
job 3](cust-design.md) currently documents the ctor switch as a
reaction to section-based discovery not surviving LTO — but a
static `__cust_tests[]` table emitted from a driver- or
plugin-known list sidesteps *both* hazards (no section reliance,
no ctor ordering cost). The v0.3.2 pre-pass and the v0.4
plugin v1 backend both produce the same table shape, so neither
needs the ctor mechanism the §10 paragraph mentions. That
paragraph is corrected in the slice-E doc update (scope item 10).

### V32D-4 — Where do test-build objects live?

**Locked: A.** `target/<profile>/test/<crate>/` — a fully fresh
build tree per member, never colocated with the lib build. This
matches the way Cargo isolates `target/<profile>/deps/<crate>-<hash>`
test binaries from `target/<profile>/deps/<crate>-<hash>` lib
artefacts (different hash inputs = different paths). For cust the
discriminator is the `test/` segment rather than a hash, because
the test build is the only "alternate build" we have in 0.3.x.

This means a `cust build` followed by `cust test` does **not**
share object files. That's fine for v0.3.2 — incremental
compilation is a separate v0.3.x or v0.4 question.

### V32D-5 — Test binary path / naming?

**Locked: A.** `target/<profile>/test/<crate>` (no extension,
matches `target/<profile>/<crate>` for bin members from v0.3.1).
**Not** `<crate>-test` or `<crate>.test` — the parent directory
already disambiguates, and a plain `<crate>` name keeps the
"binary you just ran" identifiable in `ps`.

### V32D-6 — Where does the runner live?

**Locked: B.** Emitted by the driver as
`target/<profile>/test/<crate>/cust_test_main.c`, compiled as
part of the test build, linked into the test binary. The runner
body is a static template inside `cust/src/test_runner.rs`
(string constant), not a shipped runtime crate. Rationale:

* Avoids a chicken-and-egg "the test runner is itself a cust
  crate that needs to be built" loop.
* Matches what plugin v1 will do in v0.4 — only the test-list
  generator changes, the runner template is shared.
* Zero new manifest concepts (no `[dev-dependencies]`-on-the-
  runner, no implicit-crate magic).

A future v0.5+ "shippable test runtime" can replace the template
with a real `cust_test` lib crate behind the same generated
`main`. Not blocking.

### V32D-7 — Per-test isolation?

**Locked: A. Fork-per-test.** Matches the [cust-design.md §11
"Per-test process isolation"](cust-design.md) plan beat-for-beat.
v0.3.2 is Linux-only (`fork`, `waitpid`, `WIFEXITED`/`WIFSIGNALED`,
`_exit`) — no Windows code path because the rest of cust is
Linux-only too.

**Note — stricter than stock cargo.** Cargo's `libtest` runs all
`#[test]`s of a binary as **threads of one process**, with
panic-unwind isolation only (`catch_unwind` per worker). Our model
matches **`cargo-nextest`**'s "one test per process" approach
instead. We picked it because:

* C has no `catch_unwind`. A `cust_panic` from a worker thread
  can't be recovered cleanly without `setjmp` / `longjmp`
  gymnastics, and even that doesn't survive a `SIGSEGV` /
  `abort()` / UB-induced crash — which would take the entire test
  binary down (the well-known cargo footgun that motivates
  nextest).
* `fork()` is cheap on Linux. v0.3.x test suites (cstd-sized,
  tens of tests) won't notice; multi-thousand-test crates can opt
  out via `[[cust::test(inline)]]` when v0.4 lands.
* It makes `--test-threads N` (deferred to v0.4) a *parent-side*
  concern — "how many forks in flight" — rather than a child-side
  threadpool. Simpler runner, no shared mutable state inside the
  test binary.

The user-facing UX still reads as Rust-style (`cust_test` ≈
`#[test]`, `cust_assert_eq` ≈ `assert_eq!`, substring filter,
`... ok` / `FAILED` lines, exit code 101 on failure) — only the
isolation strength is stricter.

`[[cust::test(inline)]]` (opt-out from fork isolation, matching
stock cargo's thread model) is **deferred to v0.4** — needs the
plugin-v1 attribute argument parser, and the use case is "shared
in-process cache" which the v0.3.2 demo workload (cstd) doesn't
have anyway.

### V32D-8 — Per-test timeout?

**Locked: B. No timeout in v0.3.2.** Adding `SIGTERM`-then-100ms-
then-`SIGKILL` requires either `setitimer` per fork or a
parent-side `waitpid(WNOHANG)` + select loop. Both are
straightforward but require a `[profile.test]` parse path that
v0.3.2 doesn't otherwise need. Defer to v0.4 alongside
`[profile.test]` and sanitizer-flag plumbing.

**Consequence:** an infinite-loop test in v0.3.2 hangs `cust
test`. Documented in the v0.3.2 doc and in the prelude header
for `cust_test`. Users can Ctrl-C; the signal reaches the test
subprocess via the same process group.

### V32D-9 — `cust test` filter semantics?

**Locked: A. Substring match against `module::name`** (Cargo
parity). `cust test mem` matches `mem::test_strlen_hello` and
`mem::test_strlen_empty` but not `math::test_max_i32`. Single
positional argument; multi-filter (`cust test foo bar` runs both)
**deferred to v0.4** (Cargo accepts it, but v0.3.2 picks the
simpler grammar).

`--exact` (require full equality of `module::name` to filter)
deferred to v0.4 alongside multi-filter.

### V32D-10 — `cust test` arg forwarding shape?

**Locked: A.** Mirror `cust run`'s clap shape from v0.3.1: a
`<filter>` positional plus everything after `--` forwarded as
extra runner argv. The runner sees `<filter> <forwarded...>`,
parses `<filter>` itself (positional), and parses `--list` /
future flags from the rest. Keeps the CLI surface symmetric with
`cust run` (V31D-7 / V31D-8 stay the templates).

### V32D-11 — Tests in bin-only crates?

**Locked: B. Error.** v0.3.2 only tests library code. Justification:
the bin-only path requires either (a) building the bin's TUs into a
test binary that *replaces* `cust_main` with the test runner main
(which is the multi-target story v0.4 owns), or (b) building the
bin twice (once normally, once for tests), which doubles the test-
build pipeline for marginal value. Defer cleanly to v0.4.

Lib+bin members are fine — `cust test -p <name>` tests the library
half only. The error message is explicit about this (`bin-only
crate`, not just `no library`).

### V32D-12 — Workspace-wide bare `cust test`?

**Locked: A.** Runs every testable member in topological order
(reuses the existing edge-aware iteration from
`workspace::build_workspace`). Per-member status header is
printed before each member's runner block. Summary at the end
aggregates across all members:

```
test result: ok. 12 passed; 0 failed; 0 ignored across 3 crates
```

Bin-only members are silently skipped (V32D-11 only errors when
the user *explicitly* asked for a bin-only member via `-p`).

## Resolved minor questions (locked to simplest option)

These were flagged as *non-blocking* during drafting. Each is now
locked to the simplest available option for v0.3.2 ship; the
"defer to v0.4" answers are tracked again in the Deferrals table.

* **RQ-V32-1 — `[[cust::test]]` attribute spelling.**
  **Locked: macro-only.** v0.3.2 ships `cust_test` (the macro)
  as the sole user-facing surface. The `[[cust::test]]`
  attribute spelling is **not** recognised by the v0.3.2
  pre-pass scanner. Rationale: the pre-pass regex would have
  to grow C23-attribute parsing for one attribute, and there's
  no behavioural difference \u2014 both forms expand to the same
  `annotate("cust::test")` underneath. v0.4 plugin v1 adds
  attribute-spelling support as a free side effect of having
  an AST. Documented in the prelude header next to `cust_test`.

* **RQ-V32-2 — Coloured `ok` / `FAILED` output.**
  **Locked: monochrome.** No new dependency, no terminal
  detection, no `NO_COLOR` env-var dance. v0.4's broader
  diagnostic-rendering pass owns colour for the whole driver,
  test runner included.

* **RQ-V32-3 — `cust_assert_eq` value formatting.**
  **Locked: expression text + `file:line` only.** On failure:

  ```
  test math::test_max_i32 ... FAILED
      assertion failed: `(left == right)`
        left:  `cstd_max_i32(3, 7)`
        right: `7`
        at cwork/ccrates/cstd/src/math.c:42
  ```

  The values themselves are *not* printed — doing so requires
  per-type `_Generic` printers that v0.3.2 doesn't ship. Users
  who need the runtime values today can add a `printf` above
  the assertion. v0.4 (`[[cust::derive(Debug)]]` lands
  alongside plugin v1) brings real value pretty-printing.

* **RQ-V32-4 — Test discovery and incremental compilation.**
  **Locked: not v0.3.2's problem.** Test-build objects live in a
  fresh `target/<profile>/test/<crate>/` tree (V32D-4) and the
  whole tree rebuilds every `cust test` invocation — same as
  the current `cust build` behaviour. When the incremental-
  compilation milestone lands (v0.3.x or v0.4), the test
  scanner's output hash will need to participate in the
  test-binary freshness check. That work is owned by
  whichever milestone adds incrementality, not by v0.3.2.
