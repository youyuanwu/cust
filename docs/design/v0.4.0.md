# cust v0.4.0 — plugin v1 (AST-based surface, pub_repr, test discovery)

**Status:** ✅ **shipped** (2026-06-05).
**Parent doc:** [cust-design.md §17 roadmap](cust-design.md).
**Previous milestone:** [v0.3.2.md](v0.3.2.md) (shipped).
**Minimum clang:** 19 (matches plugin v0; dev box is on 21.1.8).

This file is the v0.4.0 record — scope, in-progress design
decisions (**V40D-N**), and the verification target. v0.4.0 is
the *plugin v1* slice of the v0.4 series: it grows
`libcust_plugin.so` from the v0.2 surface-extraction stub into
the full AST-driven plugin that cust-design.md §10 has been
describing all along. Subsequent v0.4.x milestones own the
network half (registry, `cust add`, semver resolution), build
scripts, parallelism, and the `cust test` follow-ups
(`--nocapture`, `--exact`, `tests/`, ...).

### Why a 0.4.0, not "all of v0.4 at once"

The original `cust-design.md` §17 v0.4 bullet bundled six big-
ticket items into one milestone: registry, `cust add`, semver,
build scripts, plugin v1, `-jN` parallelism. v0.3.1 and v0.3.2
established the "small single-purpose milestone wedged between
big ones" cadence; v0.4.0 continues it. Plugin v1 stands alone
because:

1. **Plugin v1 is the largest single piece of v0.4.** Per the
   [v0.3.2 V32D-2 sidebar](v0.3.2.md), plugin v1 owns five jobs
   simultaneously: full `[[cust::pub_repr]]` body export
   (struct / enum / union pretty-printing with bitfields,
   alignment, anonymous nested types — all the corner cases
   plugin v0 sidesteps by emitting opaque tags only), AST-based
   test discovery, derive codegen for `Eq` / `Hash` / `Debug`,
   per-TU semantic checks (`must_use` type-level, `no_panic`
   reachability, `const` constexpr-equivalence, `unsafe`
   call-site enforcement, `repr` validation), and phase-isolation
   enforcement with the `ProgramAction == ParseSyntaxOnly`
   invariant plus the §4 fragment-stamping fixed-point loop.
   v0.4.0 *does not ship all five* (see V40D-2); it ships the
   subset that unblocks the rest of v0.4.

2. **Plugin v1 has no network dependency.** Registry / `cust
   add` / semver resolution / source-hash lockfile / sandbox
   policy for build scripts are independent work streams, and
   each carries its own risk surface. Bundling them with plugin
   v1 doubles the milestone and entangles two risk streams
   that don't need to be entangled.

3. **Plugin v1 *unblocks* the rest of v0.4.** The v0.3.2
   driver pre-pass test scanner has the single-line restriction
   baked into its V32D-2 contract; lifting it requires the AST.
   The §4 circular-`pub_repr` fixed-point loop requires the
   plugin to actually emit struct bodies into fragment headers.
   The `[profile.test]` / `[[cust::test(inline)]]` /
   `should_panic` follow-ups want the AST-attribute form. Each
   later v0.4.x milestone can lean on plugin v1; building them
   on the pre-pass would mean throwing scaffolding away twice.

4. **No backward-compat burden.** cust has no external users yet.
   v0.4.0 is the last milestone where we can flip the
   decl-annotation surface from "macros for everything" to
   "C23 attributes for everything" without a migration story.
   We take the flip (V40D-7 — see also the next subsection).
   The on-disk contracts (fragment-header byte format, generated
   runner TU shape, `__cust_tests[]` schema) stay byte-identical
   to what plugin v0 + the v0.3.2 pre-pass produce today (V40D-8),
   so the §4 stamping invariant survives the upgrade unchanged.

### Macros vs C23 attributes — the long-term split

Both surfaces exist in v0.3.x. Asking "do we keep one or both
long-term?" is a fork in the road that v0.4.0 closes. The lock
is:

* **Decl annotation: C23 attribute spelling only.** Everything
  the plugin recognises on a decl uses `[[cust::pub]]`,
  `[[cust::pub_crate]]`, `[[cust::pub_repr]]`, `[[cust::test]]`,
  `[[cust::test_ignore]]`. The matching v0.3.x prelude
  macros (`cust_pub`, `cust_pub_t`, `cust_pub_crate`,
  `cust_test`, `cust_test_ignore`) are **deleted** in v0.4.0.
  cwork is rewritten as part of the milestone.
* **Function-like helpers: macros only, forever.** `cust_assert`,
  `cust_assert_eq`, `cust_assert_ne`, `cust_panic`, plus future
  `defer!`, `unreachable!`, `likely!`, `cust_main` etc. The
  plugin literally cannot replace these — they need use-site
  expansion (sourceloc capture via `__FILE__` / `__LINE__`,
  conditional eval, control-flow injection) that the AST
  layer can't reach.

The reasoning behind dropping the dual-spelling "both
recognised" middle ground:

1. **Parametrised forms are awkward as macros.** `cust_pub_crate`
   and `cust_pub_repr` need to be separate per-variant
   macros; the C23 attribute form is just as natural with
   three separate attribute names (one name = one shape).
   The v0.3.x prelude already has a per-variant macro
   shape (`cust_pub_crate`, `cust_pub_repr` planned), which
   doesn't scale either way to attributes like
   `[[cust::link("z", kind = "static")]]` or
   `[[cust::deprecated("msg", since = "0.5")]]` (those need
   the C23 form for real arg parsing).
2. **Decl-kind awareness lives in one place.** v0.3.x has
   `cust_pub` vs `cust_pub_t` because visibility on a typedef
   warns. With the plugin owning attribute lowering
   (`ParsedAttrInfo`) we just teach it to attach
   `visibility("default")` for function/var decls and skip
   it for type decls. One attribute, one user-visible spelling,
   one place that knows the rule.
3. **One recognition path in the plugin.** Plugin v1 only
   parses `[[cust::*]]`; no `annotate("cust::*")` legacy path.
   Shrinks the plugin and the ATTRIBUTE-SEMANTICS.md doc.
4. **`--no-plugin` becomes honest.** Without macros doing
   visibility lifting unconditionally, `--no-plugin` can no
   longer produce a linkable `.a`. v0.4.0 documents `cust
   check --no-plugin` as **syntax-check only**, with no link
   promise (V40D-10). `cust build --no-plugin` and `cust
   test --no-plugin` are both rejected. The escape-hatch use
   case (clangd / IDE / fast lint loop) keeps working;
   pretending otherwise was always a fiction.

This decision IS the substance of V40D-7. The other V40D-N
decisions below assume it.

### Headline outcome

```c
// src/parser/ast.c in `parser` crate
#cust use crate::parser::types;       // sibling module

[[cust::pub_repr]] struct ast_node {  // body — pub_repr
    enum ast_kind kind;
    union {
        struct { ast_node *lhs, *rhs; } binop;
        long literal;
    };
} __attribute__((aligned(16)));

[[cust::pub]] ast_node *ast_parse(const char *src);
```

→ `target/<profile>/.h-fragments/parser/parser__ast.cust.h`
(emitted by plugin v1, byte-stable on no source change):

```c
/* @generated by cust plugin v1 — DO NOT EDIT */
/* Forward declarations of [[cust::pub]] items. */

struct ast_node {
    enum ast_kind kind;
    union {
        struct { struct ast_node *lhs, *rhs; } binop;
        long literal;
    };
} __attribute__((aligned(16)));

struct ast_node *ast_parse(const char *src);
```

Plus tests from any `src/**.c` in the crate get discovered
from the AST, without the v0.3.2 pre-pass single-line
restriction:

```c
// allowed in v0.4.0; would have been rejected by the v0.3.2 pre-pass
[[cust::test]]
int test_parser_handles_unicode_identifiers
(void)
{
    /* ... */
    return 0;
}
```

`cust test -p parser` still produces the same Cargo-shape
output; only the *discovery backend* and the *decl-annotation
spelling* changed.

## Scope

In suggested implementation order. In-progress V40D-N decisions
appear below the scope list. Anything marked **❓ open** is a
question the v0.4.0 design discussion needs to lock before the
first slice ships; anything marked **🔒 locked** is settled and
just listed here for context.

1. **Phase-mode awareness in the plugin.** Detect
   `ParseSyntaxOnly` vs `EmitLLVMOnly` / `EmitObj` via
   `CompilerInstance::getFrontendOpts().ProgramAction` (§10
   "Phase-mode detection"). Phase 1 emits fragment headers + AST
   checks only; phase 2 does semantic checks (and, post-v0.4.0,
   derive codegen). V40D-5.

2. **C23 attribute spelling — the only decl-annotation
   surface.** Register `ParsedAttrInfo` subclasses for
   `[[cust::pub]]`, `[[cust::pub_crate]]`, `[[cust::pub_repr]]`,
   `[[cust::test]]`, `[[cust::test_ignore]]`. The attribute
   handlers attach an `AnnotateAttr` to the decl that the AST
   consumer recognises. The legacy `annotate("cust::*")` path
   (plugin v0's recognition surface) is **removed** — plugin
   v1 only sees the C23 form. V40D-7.

3. **Decl-annotation macros retired + cwork rewrite.** The v0.3.x
   prelude macros `cust_pub`, `cust_pub_t`, `cust_pub_crate`,
   `cust_test`, `cust_test_ignore` are deleted from
   `cust/src/prelude.h`. cwork's cstd and hello-cstd are
   rewritten in the same commit to the C23 spelling (sed-class
   change — ~12 `cust_pub` + ~6 `cust_pub_t` + ~7 `cust_test`
   sites). Function-like macros (`cust_assert*`, `cust_panic`,
   `cust_main`) stay. V40D-7 framing; V40D-14 covers the
   plugin's compensating behaviour for `[[cust::test]]` in
   non-test builds (it has to lift the function to internal
   linkage since there's no longer a `cust_test` macro
   expanding to `static __attribute__((unused))`).

4. **`pub_crate` semantics.** Plugin v0 already accepts
   `annotate("cust::pub_crate")` and emits it to the same
   fragment header as `pub`. v0.4.0 splits: `pub_crate` decls
   go into the per-module fragment header so sibling modules
   can use them, but are filtered out of the per-crate
   concatenated header (§5). V40D-3.

5. **`pub_repr` body export.** For
   `[[cust::pub_repr]]`-tagged struct / union / enum, the
   plugin pretty-prints the full body into the fragment
   header instead of just the opaque tag. Bitfields,
   alignment, anonymous nested struct/union, attribute
   survival (`packed`, `aligned`), enum underlying type
   (`enum E : uint8_t`) are all in scope. Function/var decls
   tagged `[[cust::pub_repr]]` are rejected with a clear
   error. V40D-4 covers the pretty-printer strategy.

6. **Enum support.** Plugin v0 emits nothing for `EnumDecl`.
   v0.4.0 adds enum forward decls and (for `pub_repr` enums)
   full body export. V40D-4 sub-case.

7. **AST-based test discovery.** Plugin walks for
   `[[cust::test]]` and `[[cust::test_ignore]]` decls. Emits
   the same `__cust_tests[]` entries the v0.3.2 pre-pass
   scanner emits today — same `(qname, fn_ptr, fn_kind,
   ignored, file, line)` tuple, same generated runner TU
   shape. The runner template is unchanged from v0.3.2.
   V40D-6 covers the driver-side flow.

8. **Driver: plugin mandatory for `cust build` and `cust
   test`.** Today plugin discovery is silent on missing
   plugin (`cust/src/plugin.rs`). v0.4.0 upgrades both
   subcommands to hard-error with the V40D-12 wording. The
   v0.3.2 test-discovery pre-pass scanner
   (`cust/src/test_scanner.rs`) is **deleted** as part of
   this slice — there is no longer a fallback discovery path
   for `cust test`. The `mod_scanner` and `pub_macro`
   pre-passes are independent and stay (V40D-13).

9. **Fixed-point iteration loop for circular `pub_repr`.** §4
   phase 1 specifies an iterate-until-byte-identical loop with
   a 3-iteration cap. Plugin v0 / today's driver run phase 1
   once and never iterate. v0.4.0 adds the loop in the driver
   (not the plugin — the plugin remains stateless per TU).
   Diagnostic on divergence at the 4th iteration uses the
   §4 verbatim error wording. V40D-11.

10. **`cust check --no-plugin` flag.** The honest downgrade:
    syntax-only mode, no decl-annotation promises. Skips
    `-fplugin=...`, adds `-Wno-unknown-attributes` so clang
    doesn't warn on the now-unrecognised `[[cust::*]]`
    attributes, no fragment headers emitted, no test
    discovery. `cust build --no-plugin` and `cust test
    --no-plugin` are both rejected. V40D-10.

11. **Documentation.** §10 plugin jobs reflect the v0.4.0 split
    (pre-pass owns `pub_macro` only; plugin owns everything
    else, including all decl annotation). §11 v0.3.2
    subsection flips its "v0.4 lifts the single-line
    restriction" wording to "v0.4.0 lifted it; the pre-pass
    test scanner is gone". §9 attribute catalogue: rows
    rewritten to list only the C23 spelling, no "macro form
    OR C23 form" annotations. §17 roadmap gains a v0.4.0
    bullet between v0.3.2 and the now-trimmed v0.4 entry.

## Decisions (V40D-N — locked)

### V40D-1 — Does plugin v1 replace plugin v0, or augment it?

**Locked: A. Replace.** Plugin v1 is the only `libcust_plugin.so`
shape v0.4.0 supports. The v0.2 annotate-only recognition
path in `plugin/src/plugin.cc` is removed in the same commit
that lands `ParsedAttrInfo` recognition for the C23 form
(V40D-7). No cross-version compatibility — a driver built
from v0.4.0 sources expects a plugin built from v0.4.0
sources, and the plugin discovery layer fails fast when it
doesn't get one.

The driver and plugin are built from the same source tree
and assumed to match (RQ-V40-1: no capability negotiation
symbol). Mismatch surfaces as a clang error — unknown
`[[cust::*]]` attribute, missing `ParsedAttrInfo`
registration, or the V40D-12 "plugin not found" error if
the `.so` is absent entirely. v0.5+ may grow a real
negotiation layer if and when an external plugin ecosystem
materialises.

Alternative considered — "keep plugin v0's annotate path
alive so the same plugin binary can serve a v0.3.x driver
in parallel" — rejected because there's no v0.3.x driver to
serve. The repo only ships from one source tree.

### V40D-2 — What's in plugin v1 vs deferred to v0.5+?

**Locked: A. In v0.4.0** (the "five plugin v1 jobs" trimmed to
the smallest subset that unblocks the rest of v0.4):

* ✅ Phase-mode detection.
* ✅ Fragment header synthesis (full surface: pub / pub_crate / pub_repr).
* ✅ C23 attribute spelling.
* ✅ AST-based test discovery.

**Deferred to v0.5+** (each gets its own milestone or rolls
into v0.7 per §17):

* ⏳ Derive codegen (`[[cust::derive(Eq, Hash, Debug)]]`) — v0.7.
* ⏳ `must_use` type-level enforcement — v0.7.
* ⏳ `no_panic` per-TU reachability heuristic — v0.7.
* ⏳ `const` constexpr-equivalent enforcement — v0.7.
* ⏳ `unsafe` call-site enforcement — v0.7.
* ⏳ `repr(C | packed | transparent)` validation — v0.7.
* ⏳ `deprecated`, `ctor`/`dtor`, `asm_export`, `doc`, `link`,
   `bench` — v0.7+, individually small.

Rationale: derive + `must_use` + `no_panic` + `const` +
`unsafe` + `repr` validation is the "per-TU semantic checks"
bucket from §10 job #1, all of which produce diagnostics
rather than fragment-header or test-table output. They share
no infrastructure with the v0.4.0 work and can ship one at a
time without rebuilding the plugin's architecture each time.
v0.4.0 ships the infrastructure (phase-mode, attribute
recognition, AST walker, file-output contracts); v0.7 fills
in the semantic-check leaves.

### V40D-3 — How does `pub_crate` differ from `pub` in fragment headers?

**Locked: A. Both end up in the per-module fragment header;
only `pub` makes it into the per-crate concatenated header.**
Concretely:

* `target/<profile>/.h-fragments/<crate>/<module>.cust.h`
  contains forward decls for *both* `[[cust::pub]]` and
  `[[cust::pub_crate]]` items in `<module>`. Sibling modules
  in the same crate `#cust use crate::<module>;` and reach
  both.
* `target/<profile>/build/<crate>/include/<crate>.h` (the
  concatenated crate header from §5) contains *only* `pub`
  items. `pub_crate` items are filtered out at the concat
  step.

Implementation: each emitted fragment-header line carries a
single-char prefix comment (`/*p*/` or `/*c*/`) that the
driver's concat step uses to filter. The concat step strips
the prefix comments on the way out so the published header
is clean. Alternative considered — separate
`<module>.pub.cust.h` and `<module>.pub_crate.cust.h` files
— rejected for doubling the file count and complicating the
stamping invariant.

Symbol-localisation behaviour at link time (§6) is unchanged:
`pub_crate` symbols are localised by `llvm-objcopy
--localize-hidden` at the crate-link step; `pub` symbols
survive into the final artifact.

### V40D-4 — `pub_repr` body export — how is the body printed?

**Locked: A. Custom pretty-printer over the AST, not
clang's `SourceManager::getSourceText`.** The naive
"copy the original source text byte-for-byte between the
`{` and `}`" approach has three blockers:

1. The original text may contain `#include`d macros that
   only expand correctly in the original TU's preprocessor
   environment. Fragment headers must be self-contained.
2. The original text may reference `static const` constants
   from the same TU (e.g. `int buf[BUFSZ];`) that are not
   exported. The pretty-printer can resolve `BUFSZ` to its
   value via `ConstantExpr::getResultAsAPSInt`; raw text
   copy can't.
3. Anonymous nested types inside the body need stable
   synthetic names so two different `pub_repr` structs
   sharing an anonymous inner type each get their own
   inner-type decl in the fragment header.

The pretty-printer is a `RecursiveASTVisitor` subclass that
walks the `RecordDecl` / `EnumDecl` and emits C source.
Coverage:

* **Basic fields:** `<type> <name>;`. Types resolved via
  `QualType::print` with `PrintingPolicy::PolishForDeclaration`.
* **Bitfields:** `<type> <name> : <width>;` via
  `FieldDecl::isBitField` + `getBitWidthValue`.
* **Alignment:** `__attribute__((aligned(N)))` after the
  closing brace if `RecordDecl::getMaxAlignment()` exceeds
  the natural alignment.
* **Packing:** `__attribute__((packed))` if
  `RecordDecl::hasAttr<PackedAttr>()`.
* **Anonymous nested struct/union:** emitted in-line at the
  field site, recursively.
* **Named nested types** (e.g. an `enum` defined inside a
  `pub_repr struct`): emitted *before* the struct in the
  fragment header (own line), then referenced by tag from
  inside the struct body.
* **Enums:** `enum <name> [: <underlying>] { V0 = …, V1 = …, };`.
  Underlying type emitted only if explicitly specified
  (C23 `enum E : T`). Discriminant values always emitted
  explicitly (not implicit) so a future enum-variant reorder
  doesn't change the public ABI silently.
* **Unions:** same as structs, with `union` keyword.

What's **out of scope** for v0.4.0's pretty-printer (each
emits a clear error, not silent acceptance):

* Pointer-to-member types (C++ only).
* `_BitInt(N)` width > 64 (C23 — wait for OQ).
* `_Atomic` qualifiers (need OQ on ABI).
* `__attribute__((vector_size(N)))` (need OQ on portable
  representation).

Error wording for unsupported kinds:

```
error: cannot export body of `<decl>`: <reason>
  hint: drop `pub_repr` and export as opaque tag with
        `[[cust::pub]]`, or open an issue describing the use case
```

### V40D-5 — Phase-mode invariants?

**Locked: A. Plugin enforces** that fragment headers AND test
discovery sidecar files (RQ-V40-2) are emitted **only** in
`ParseSyntaxOnly` (phase 1) and **never** in `EmitLLVMOnly` /
`EmitObj` (phase 2). Both outputs share the same gate.
Concretely:

* `CustASTConsumer::HandleTranslationUnit` checks
  `CI.getFrontendOpts().ProgramAction == ParseSyntaxOnly`.
* If true and `fragment-out` / `test-sidecar-out` was passed:
  emit the corresponding file.
* If false and either was passed: hard error
  ("phase 2 plugin invocation must not write fragment headers
   or sidecar files — this is a driver bug").
* If true and both were absent: silent no-op (today's
  IDE-side `clangd` invocation path).
* If false and both were absent: run semantic checks
  only (the v0.7 leaf — no-op in v0.4.0).

Unit test: `test_phase_isolation.cc` (in `plugin/test/`,
stood up in slice A per the slice plan) invokes the plugin
twice on the same TU with the two `ProgramAction` values,
asserts the phase-1 outputs equal the on-disk files, and
asserts phase 2 writes nothing.

Driver-side invariant: phase 1 invocations pass
`-fplugin-arg-cust-fragment-out=<path>` and (in test builds)
`-fplugin-arg-cust-test-sidecar-out=<path>`; phase 2
invocations pass neither. `build_cflags` in
`cust/src/build.rs` already threads `fragment_out:
Option<&Path>` through both call sites; v0.4.0 adds
`test_sidecar_out: Option<&Path>` next to it and makes sure
phase 2 always passes `None` for both.

### V40D-6 — Test discovery: plugin or pre-pass scanner?

**Locked: A. Plugin only — the v0.3.2 pre-pass scanner is
deleted in v0.4.0.** Today the driver's `run_test_build()`
calls `test_scanner::scan` on every module source. v0.4.0
flips it to:

```
// Plugin is mandatory for `cust test` per V40D-12.
// During phase 1 the plugin emits per-TU test entries into
// sidecar files alongside fragment headers. The runner-
// template generator reads those files and emits the
// __cust_tests[] table.
let entries = collect_plugin_test_entries(&plan)?;
```

The plugin emits each TU's test entries to a sidecar file
`target/<profile>/.test-discovery/<crate>/<module>.cust.tests`
(one entry per line, format per RQ-V40-2). The runner-
template generator reads all files matching that glob and
emits the `__cust_tests[]` table.

There is no fallback discovery path — `--no-plugin` does
not apply to `cust test` (V40D-10 rejects it). `cust/src/
test_scanner.rs` and its 17 unit tests are deleted in the
same commit. The driver pre-pass that handles `pub_macro`
(V40D-13) is unrelated and stays.

Alternative considered — "keep the pre-pass as a
verification cross-check" — rejected. The pre-pass
single-line restriction (V32D-2) is a strict subset of
what the AST sees; a cross-check would only ever flag
"plugin found a test the pre-pass missed," which is the
expected case. Dead weight.

Alternative also considered — "retarget the pre-pass to
scan for the `[[cust::test]]` token instead of `cust_test`
and keep it as the `--no-plugin` path" — rejected because
`--no-plugin` no longer promises any decl-annotation
functionality (V40D-10). The pre-pass would scan, find
entries, but then have nowhere to plumb them (the test
runner needs the plugin anyway to produce a linkable test
binary). Dead weight.

### V40D-7 — Decl-annotation surface: macros, C23 attributes, or both?

**Locked: A. C23 attributes only.** Decl-annotation macros
(`cust_pub`, `cust_pub_t`, `cust_pub_crate`, `cust_test`,
`cust_test_ignore`) are deleted from `cust/src/prelude.h` in
v0.4.0 slice E. The four-point rationale lives in the
"Macros vs C23 attributes — the long-term split" subsection
above; this entry records the implementation contract.

Implementation:

* `ParsedAttrInfoRegistry` registers by attribute *name*,
  not name + arguments — and clang's expression-parser for
  C++/C23 attribute arguments silently drops identifiers
  that aren't in scope (verified by AST-dump during slice A:
  `[[cust::pub(crate)]]` with `OptArgs = 1` produces a 0-arg
  ParsedAttr at `handleDeclAttribute` time, the identifier
  `crate` having been lost in the parser). There is no
  ParsedAttrInfo hook to override the parser. So v0.4.0
  registers **five** recognisers in `plugin/src/plugin.cc`,
  one per attribute name (no parens, no dispatch needed):
  * `cust::pub`         → `AnnotateAttr("cust::pub")`
  * `cust::pub_crate`   → `AnnotateAttr("cust::pub_crate")`
  * `cust::pub_repr`    → `AnnotateAttr("cust::pub_repr")`
  * `cust::test`        → `AnnotateAttr("cust::test")`
  * `cust::test_ignore` → `AnnotateAttr("cust::test_ignore")`
  Each recogniser accepts 0 args. The AST visitor sees a
  single recognition path keyed on the annotation string.
  Naming convention: snake_case underscored modifiers
  (`pub_crate`, `pub_repr`, `test_ignore`) match the
  `AnnotateAttr` payload strings the legacy v0.3.x macros
  already use, so the rewrite from macro to C23 form is
  mechanical and the plugin's internal recognition surface
  doesn't change.
* Earlier draft proposed a single parameterised `cust::pub`
  recogniser with `getArg(0)` dispatch on `crate` / `repr`.
  Rejected during slice A implementation when AST-dump
  showed clang's expression-parser silently drops the
  identifier args; there is no clean way to make
  `[[cust::pub(crate)]]` parse as a 1-identifier-arg
  attribute via the ParsedAttrInfo API.
* `cust_pub`, `cust_pub_t`, `cust_pub_crate`, `cust_test`,
  `cust_test_ignore` removed from `cust/src/prelude.h` in
  the same commit. Function-like macros (`cust_assert`,
  `cust_assert_eq`, `cust_assert_ne`, `cust_panic`,
  `cust_main`) stay.
* `[[cust::pub]]` on a function/var decl: plugin attaches
  `visibility("default")` automatically (decl-kind aware —
  this is what `cust_pub` vs `cust_pub_t` used to encode
  manually).
* `[[cust::pub]]` on a typedef/struct/union/enum decl:
  plugin skips the visibility lift (no-op on type decls;
  used to be `cust_pub_t`'s job).
* `[[cust::test]]` on a function decl in a non-test build
  (`CUST_TEST_BUILD` undefined): plugin attaches
  `__attribute__((internal_linkage, unused))` so the function
  doesn't leak into the regular artifact. This replaces what
  `cust_test`'s non-test branch did via macro expansion.
  See V40D-14.
* `[[cust::test]]` in a test build (`CUST_TEST_BUILD`
  defined): plugin emits a `__cust_tests[]` entry into the
  sidecar file (V40D-6, RQ-V40-2) and leaves the function
  alone.

Alternative rejected — **"both spellings, equivalent;
macros remain documented default"** (this entry's prior
draft) — rejected because:

* It's a permanent tax (two source spellings, two recognition
  paths in the plugin, a fallback test matrix to maintain).
* Decl-kind awareness in the plugin (this entry's third and
  fourth bullets) is strictly easier than the macro split
  (`cust_pub` / `cust_pub_t`) we have today, not harder.
* Without backward-compat constraints, there's no reason
  to keep both alive other than aesthetic inertia.

Alternative also rejected — **"keep decl macros as thin
wrappers around the C23 form"** (e.g. `#define cust_pub
[[cust::pub]] __attribute__((visibility("default")))`) —
rejected because under `--no-plugin` the C23 form trips
`-Wunknown-attributes`; suppressing it in the prelude
with `#pragma clang diagnostic ignored
"-Wunknown-attributes"` is a sledgehammer that hides real
attribute typos. Cleaner to require the user to write the
C23 form and let `--no-plugin` add `-Wno-unknown-attributes`
explicitly at the driver level (V40D-10).

### V40D-8 — Fragment-header byte stability within v0.4.0?

**Locked: A. The bytes plugin v1 emits for a given AST are
deterministic and stable across plugin rebuilds.** No promise
of bytewise stability vs. plugin v0 — the source-level surface
changed (C23 attributes instead of `annotate(...)` macros)
and cwork is rewritten as part of the milestone, so the
fragment-header churn is bounded and expected.

Within v0.4.0 the stamping invariant from §4 — "byte-identical
fragment header → downstream modules don't rebuild" — still
requires that re-running the plugin on the same TU produces
the same bytes. This is enforced by:

* Deterministic `PrintingPolicy` settings (`TerseOutput =
  true`, `PolishForDeclaration = true`, `SuppressInitializers
  = true`); no clang version-specific flags that flip on us.
* A golden-file test in `plugin/test/` that locks the byte
  output for a representative TU containing one of each
  supported decl kind (function, typedef, opaque record,
  `pub_repr` struct with bitfields, `pub_repr` enum with
  explicit underlying, `extern` var). Re-running the test
  with the same plugin must reproduce; a different plugin
  build that changes the bytes fails the test (forcing the
  author to either revert the change or update the golden).
* Decl iteration order is `ASTContext::getTranslationUnitDecl
  ()->decls()`, which clang documents as source order. The
  plugin does not sort.

Within-rebuild stability is what makes the stamping invariant
functional. Cross-version stability ("upgrading the plugin
doesn't rebuild the world") is a non-goal post-v0.4.0 — if
v0.5 changes the fragment-header format, the next build
invalidates every cached fragment and re-emits; this is fine
because the plugin discovery layer also recognises the new
plugin and re-runs everything regardless.

### V40D-9 — Test surface for each plugin-owned attribute?

**Locked: A.** One column of tests per attribute, not three.
The v0.3-style "with_plugin / without_plugin /
fallback_behavior" matrix is moot: under V40D-10
`--no-plugin` doesn't promise any decl-annotation
functionality, so there's no fallback path to test.
ATTRIBUTE-SEMANTICS.md (the divergence-documenting doc §9
promises) becomes a one-line entry per attribute
documenting "recognised by the plugin only" and is
otherwise unnecessary in v0.4.0.

| Attribute | v0.4.0 test |
|---|---|
| `[[cust::pub]]` (function/var) | fragment header lists the decl; symbol has `visibility("default")` in the emitted object |
| `[[cust::pub]]` (typedef/record/enum) | fragment header lists the opaque tag; no visibility lift (and no `-Wignored-attributes` warning) |
| `[[cust::pub_crate]]` | fragment header (sibling reach) yes, crate header no; symbol localised at crate-link step |
| `[[cust::pub_repr]]` (record/enum) | full body in fragment header per V40D-4 |
| `[[cust::pub_repr]]` (function/var) | hard error with V40D-4 wording |
| `[[cust::test]]` (test build) | plugin emits `__cust_tests[]` sidecar entry |
| `[[cust::test]]` (non-test build) | plugin attaches `internal_linkage + unused`; function does NOT appear in the regular artifact (`nm` check) |
| `[[cust::test_ignore]]` | sidecar entry has `ignored = true`; runner template skips it |

The non-test-build behaviour for `[[cust::test]]` (last row

### V40D-10 — `cust check --no-plugin` semantics?

> **⚠️ SUPERSEDED (2026-06-14).** The `--no-plugin` flag was
> **removed** ahead of the incremental-`cust check` milestone:
> it offered a strict subset of the real (plugin-backed) check
> and could pass code the real check rejects (a false green).
> The same reasoning retired the *implicit* plugin-less fallback
> on `cust check` (warn + syntax-only): the plugin is now
> **mandatory for every subcommand** that resolves it — `build`
> / `check` / `test` / `run` all hard-error if the plugin is
> missing (V40D-12). The historical decision below is retained
> for context. See [incremental-check.md](incremental-check.md)
> and [cust-design.md §10](cust-design.md).

**Locked: A. Syntax-only escape hatch, no decl-annotation
promises.** The flag is opt-out (default is plugin-on). Per
subcommand:

* `cust check --no-plugin` — skip `-fplugin=...`. Add
  `-Wno-unknown-attributes` so clang doesn't warn on the
  now-unrecognised `[[cust::*]]` attributes; the syntax
  check still runs over the entire TU. No fragment headers
  emitted, no test discovery, no `__cust_tests[]` files
  generated. Output is only as useful as plain
  `clang -fsyntax-only` over each TU — useful for IDE / fast
  lint loop, useless for verifying cust-specific semantics.
* `cust build --no-plugin` — **rejected.** Building requires
  fragment headers. Error wording:

  ```
  error: `--no-plugin` is incompatible with `cust build`
         (fragment headers are required to link multi-module crates)
    hint: drop `--no-plugin`, or use `cust check --no-plugin`
          for a syntax-only pass
  ```
* `cust test --no-plugin` — **rejected** (new in V40D-10).
  Test discovery requires the plugin (V40D-6 deleted the
  pre-pass scanner). Error wording:

  ```
  error: `--no-plugin` is incompatible with `cust test`
         (test discovery requires the plugin)
    hint: drop `--no-plugin`
  ```

Driver flag is on the root `cust` invocation, not per
subcommand, so `cust --no-plugin check` and `cust check
--no-plugin` are both accepted (V40D-10a — clap-level
placement decided during impl). `cust --no-plugin build`
and `cust --no-plugin test` produce the same hard errors as
the per-subcommand forms.

### V40D-11 — Fixed-point loop placement + divergence wording?

**Locked: A.** The loop wraps **the per-crate phase-1 sweep**
in `cust/src/build.rs` (specifically `compile_surface_pass`,
the function that today walks the crate's modules once and
invokes clang with `-fsyntax-only` + `-fplugin-arg-cust-
fragment-out=<path>` for each). Per-crate, not per-module —
the cycle detection is over the inter-module fragment-header
dependency graph and only the whole sweep is meaningful as a
fixed-point unit.

Iteration mechanics:

* Iteration N writes fragment headers to a scratch tree
  `target/<profile>/.h-fragments/<crate>/.iter-N/` rather
  than the canonical `.h-fragments/<crate>/`.
* After each iteration, the driver byte-compares each
  module's `.iter-N/<module>.cust.h` against
  `.iter-(N-1)/<module>.cust.h` (or against the canonical
  fragment from the previous run for iter 1). Converged
  when every module matches.
* On convergence: atomically `rename` the converged
  `.iter-N/` directory contents over `.h-fragments/<crate>/`,
  preserving the per-file stamping-skip behaviour the
  plugin already implements (matching files are skipped at
  write time inside the plugin; this is the outer-loop
  equivalent).
* On non-convergence after N=3: emit the V40D-11 error
  (verbatim per §4), list the still-wobbling modules,
  exit 1.

Reuse the §4 verbatim error wording:

```
error: circular `[[cust::pub_repr]]` dependency did not converge
  in 3 iterations between modules: <module-a>, <module-b>[, …]
  hint: break the cycle by exporting one side as `[[cust::pub]]`
        (opaque) instead of `[[cust::pub_repr]]`
```

Listed modules are the set whose fragment headers still
differed between iterations 2 and 3 (the "still wobbling"
set). Cap is a divergence detector, not a complexity limit;
if a real-world `pub_repr` cycle empirically needs 4
iterations to converge, raise the cap to 5 and rerun (§4
"raise the cap rather than rejecting them"). v0.4.0 ships
with cap = 3 and a hidden `CUST_FIXED_POINT_CAP=<n>` env
var for debugging.

The plugin remains stateless per TU — every iteration
re-invokes clang on each module the same way. The plugin
doesn't know which iteration it's on; it just writes to
the path the driver gave it via `-fplugin-arg-cust-
fragment-out=`. Concurrent crate builds remain safe because
the per-crate `target/<crate>/.lock` from §4 "Concurrent
builds" serialises any two builds of the same crate.

### V40D-12 — Plugin loading: hard error vs silent skip?

**Locked: B. Hard error when plugin is configured-required
but missing.** Today plugin discovery is silent (per
`cust/src/plugin.rs` doc comment: "Discovery is silent when
the plugin is absent"). v0.4.0 changes this for `cust build`
and `cust test` (both need the plugin for fragment headers
and/or discovery) — both upgrade "plugin missing" from
silent skip to:

```
error: cust plugin (libcust_plugin.so) not found
  searched:
    $CUST_PLUGIN: <value or "not set">
    <exe-dir>/libcust_plugin.so: <exists? / not found>
  hint: build the plugin with `cargo run -p plugin-build`
```

`cust check` keeps silent-skip behaviour (single-TU
syntax-only doesn't strictly need the plugin), but emits a
warning if the plugin is absent so users discover the
problem before it bites them in `cust build`.

### V40D-13 — `pub_macro` extraction: who owns it?

**Locked: A. Driver pre-pass keeps owning it —
architecturally.** §9 attribute catalogue puts
`[[cust::pub_macro]]` under "driver pre-pass" because macro
definitions are erased by clang's preprocessor before the
plugin sees the AST. Even with plugin v1's expanded surface,
the *AST* still doesn't have the macro decls. A future
plugin v2 using clang `PPCallbacks` could move this into the
plugin (the §9 parenthetical), but that's out of scope for
v0.4.0.

**Implementation note:** v0.4.0 only commits to the
architectural decision. The actual `pub_macro` pre-pass
extractor does NOT exist in v0.4.0 (or anywhere else today
— `cust/src/mod_scanner.rs` only handles `#cust mod` and
`#cust use`, not `[[cust::pub_macro]]`). v0.4.0 leaves
`pub_macro` as a no-op the same way v0.3.x does; the
extractor lands in whichever later milestone first needs
it (probably v0.4.x or v0.5 when a real crate wants to
export an object-like macro). This V40D entry exists to
pin the ownership before that lands.

Net effect for v0.4.0: zero discovery pre-passes (the v0.3.2
test-discovery pre-pass is deleted per V40D-6; the
`pub_macro` pre-pass doesn't exist yet). `mod_scanner` (for
`#cust mod` / `#cust use`) is a different layer entirely
(driver pragmas, not attribute discovery) and stays.

### V40D-14 — `[[cust::test]]` behaviour in non-test builds?

**Locked: A. Plugin attaches `internal_linkage + unused` so
the test function doesn't leak into the regular artifact.**
v0.3.2's `cust_test` macro encoded this at macro-expansion
time:

```c
// v0.3.2 prelude (deleted in v0.4.0):
#ifdef CUST_TEST_BUILD
#  define cust_test __attribute__((annotate("cust::test")))
#else
#  define cust_test __attribute__((unused)) static
#endif
```

Without the macro, the plugin has to do the equivalent
at AST-attach time. Per V40D-7's five-recogniser model, the
`cust::test` AND `cust::test_ignore` recognisers both run
this logic:

* Always attaches `AnnotateAttr` with payload
  `"cust::test"` or `"cust::test_ignore"` respectively, for
  the AST consumer to find.
* If `CUST_TEST_BUILD` is **not** defined: also attaches
  `InternalLinkageAttr` and `UnusedAttr`. Effect: the
  function emits as `static` in the object and clang doesn't
  warn about an unreferenced decl.
* If `CUST_TEST_BUILD` **is** defined: only the annotate
  attribute. Function gets normal external linkage so the
  generated runner TU can reference it via `extern`.

Detection of `CUST_TEST_BUILD` from inside the
`ParsedAttrInfo::handleDeclAttribute` callback (which runs
during parsing, *before* `CustASTConsumer` exists):

* The callback signature is
  `AttrHandling handleDeclAttribute(Sema &S, Decl *D, const
  ParsedAttr &Attr) override`.
* Read the macro through the Sema's preprocessor:
  `S.getPreprocessor().getMacroInfo(
  S.getPreprocessor().getIdentifierInfo("CUST_TEST_BUILD"))`
  returns non-null when the macro is defined.
* Cache the lookup on `S` via a side-channel `llvm::DenseMap
  <Sema *, bool>` keyed by Sema pointer (one Sema per TU;
  the entry is short-lived). Avoids re-walking the macro
  table on every `[[cust::test]]` attachment in a TU with
  many tests.

Alternative considered — "leave the function with external
linkage and rely on `--gc-sections` / link-time elimination"
— rejected because (a) it only works with `-ffunction-
sections -fdata-sections -Wl,--gc-sections`, which we'd
have to thread through `build_cflags` unconditionally; (b)
static libraries (`.a`) don't run the linker at archive
time, so the test symbols would still appear in `nm
libcstd.a` until something else links the archive. The
plugin-attached `internal_linkage` is the honest answer.

Alternative also considered — "don't even codegen the
function in non-test builds (plugin removes it from the
AST)" — rejected because removing decls from the AST mid-
compilation is fragile (other decls may reference the
removed one in non-obvious ways) and clang's plugin
interface doesn't support it cleanly.

## Open questions (RQ-V40-N — locked)

Small questions; each picked the simplest option that meets
the milestone's needs.

* **RQ-V40-1. Plugin capability negotiation — yes or no?**
  **Locked: no.** The driver and plugin are built from the
  same source tree (`cargo build && cargo run -p plugin-build`).
  V40D-1 already commits to "plugin v1 replaces plugin v0; no
  cross-version compatibility." If the plugin loads but is
  built against a different driver's expectations, the
  symptom is a clang error (unknown attribute, missing
  `ParsedAttrInfo` registration, etc.) which is already
  surfaced clearly. Adding a `cust_plugin_capabilities()`
  symbol + JSON-parsing logic in the driver is infrastructure
  v0.4.0 doesn't need. Reconsider in v0.5+ if and when an
  external plugin ecosystem materialises.

* **RQ-V40-2. Where do `__cust_tests[]` sidecar files live,
  and what's the format?**
  **Locked.** Path:
  `target/<profile>/.test-discovery/<crate>/<module>.cust.tests`
  — sibling to `.h-fragments/`, same lifecycle as fragment
  headers (regenerated on every phase 1, atomically swapped,
  byte-stamping skips downstream work). Reuses the directory
  infrastructure the fragment-header layer already has.

  Format: TSV, one test per line, no header line.

  ```
  <qname>\t<fn_kind>\t<ignored>\t<file>\t<line>
  ```

  Fields:
  * `qname` — `::`-separated, matches v0.3.2's `TestEntry::
    module + "::" + name`. Drops the `lib` prefix for the
    crate root, same rule as the v0.3.2 runner template.
    Test names are valid C identifiers per V40D-7's
    `ParsedAttrInfo` registration (anything else fails clang
    parsing before the plugin runs), so the `qname` never
    contains literal tab / newline / `\`.
  * `fn_kind` — literal string `int` or `void`. Matches
    `TestEntry::fn_kind` (`FnKind::Int` / `FnKind::Void`).
  * `ignored` — literal `0` or `1`.
  * `file` — absolute path to the source file. Spaces
    allowed (TSV separator is tab); the driver normalises
    to workspace-relative in the runner template output.
    A file path containing a literal tab or newline is
    rejected by the plugin with `"sidecar file path
    contains illegal character (tab or newline): <path>"`.
  * `line` — 1-based decimal integer.

  No version field: the path lives under `.test-discovery/`
  which is regenerated unconditionally on every phase 1, so
  a format change in a future plugin just regenerates with
  the new shape. Driver and plugin are built from the same
  source tree per RQ-V40-1; they agree on the format.

* **RQ-V40-3. Plugin build mechanism?**
  **Locked: keep V2D-2 unchanged.** `cargo run -p plugin-build`
  shells out to `cmake -B build && cmake --build build`.
  Contributors who forget step 2 hit the V40D-12 hard error
  on the next `cust build`, which tells them what to do. No
  wrapper script needed; if the two-step flow becomes painful
  empirically, add the script then.

* **RQ-V40-4. Capability-set encoding?**
  **Locked: moot — dropped per RQ-V40-1.** No capability
  symbol, no encoding to choose.

* **RQ-V40-5. Split function-like macros out of `prelude.h`?**
  **Locked: no.** The prelude is "things every TU gets via
  `-include`," and function-like macros qualify. Splitting
  would mean threading two `-include` flags through
  `build_cflags` for zero v0.4.0 benefit. Reconsider in
  v0.5+ if the file gets crowded.

* **RQ-V40-6. Phase-2 plugin invocation — drop or keep
  `-fplugin=...`?**
  **Locked: keep loaded.** Today `build_cflags` adds
  `-fplugin=...` whenever `plan.plugin` is `Some`, in both
  phases. Stripping it for phase 2 means adding a phase-aware
  branch in `build_cflags` plus the matching phase-mode no-op
  inside the plugin (V40D-5 already specifies the no-op).
  Smaller diff to leave the flag in.
