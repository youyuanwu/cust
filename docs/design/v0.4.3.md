# cust v0.4.3 — `tests/` integration tests

**Status:** � **shipped** (opened 2026-06-12, closed 2026-06-13).
**Parent doc:** [cust-design.md §11 `cust test`](cust-design.md).
**Previous milestone:** [v0.4.2.md](v0.4.2.md) (shipped 2026-06-11).
**Builds on:** v0.3.2 unit-test harness (V32D-1 … V32D-12) and
v0.4.0 plugin-based test discovery (V40D-6, V40D-7, V40D-14)
and v0.4.2 CMake/Ninja build backend (V42D-13, V42D-14, V42D-16).
**Minimum clang / CMake / Ninja:** unchanged from v0.4.2
(clang 19, CMake 3.21, any Ninja).

This file is the v0.4.3 record — scope, in-progress design
decisions (**V43D-N**), open questions, and the verification
target. v0.4.3 adds integration tests under each crate's
`tests/` directory: one `.c` file = one test executable
linked against the crate's **public** surface only (same shape
as a downstream `#cust use crate::<crate>;` consumer). Unit
tests under `src/**.c` (V32D-1) continue to work unchanged.

The originally-planned v0.4.3 scope ("`--jobs` / `-jN` polish")
collapsed into [v0.4.2 slice D](v0.4.2.md#slice-d----jobs--cust_jobs-plumbing-0be1184)
when V42D-13 let one Ninja graph own intra- and inter-crate
parallelism; `cust build -jN` already lowers to `cmake
--build -j N`. v0.4.3 reslots to integration tests, which were
formerly grouped under v0.4.5 — see "Roadmap impact" below.

## 1. Headline outcome

A crate gains a sibling `tests/` directory next to `src/`:

```
cwork/ccrates/cstd/
├── Cust.toml
├── src/
│   ├── lib.c
│   ├── math.c
│   └── …
└── tests/                      # NEW in v0.4.3
    ├── basic.c                 # one integration-test exe per file
    └── alloc_pressure.c
```

`cust test` against cwork now runs unit tests **and**
integration tests, in dep order over the workspace:

```console
$ cust test
   Compiling cstd v0.4.2
   Compiling hello-cstd v0.4.2
   Configuring workspace (cmake, target/debug/cmake)
   Building   workspace (ninja -j8)

     Running unittests src/lib.c (target/debug/test/cstd/cstd)

running 10 tests
test geom::test_point_distance_sq_3_4_5 ... ok
…
test mem::test_strlen_hello ... ok

test result: ok. 10 passed; 0 failed; 0 ignored

     Running tests/basic.c (target/debug/test/cstd/basic/basic)

running 4 tests
test test_min_max_roundtrip ... ok
test test_strlen_empty_after_alloc ... ok
test test_geom_via_public_surface ... ok
test test_version_is_0_4_2 ... ok

test result: ok. 4 passed; 0 failed; 0 ignored

     Running tests/alloc_pressure.c (target/debug/test/cstd/alloc_pressure/alloc_pressure)

running 2 tests
test test_string_grow_until_oom ... ok
test test_vec_push_many ... ok

test result: ok. 2 passed; 0 failed; 0 ignored
```

The output banner format mirrors Cargo's:

* Unit tests: `Running unittests src/lib.c (<exe>)`
* Integration tests: `Running tests/<file>.c (<exe>)`

Test-output line format inside each exe is unchanged from
v0.3.2 (V32D-7): `test <qname> ... ok|FAILED|ignored`.

**Linkage** (locked, V43D-3): each `tests/<file>.c` exe links
**only** against the CUT's public surface — `lib<crate>.a` and
the concatenated `<crate>.h` published at
`target/<profile>/build/<crate>/include/<crate>.h`. It does
**not** see `[[cust::pub_crate]]` or unannotated decls. This
makes the integration-test exe a real downstream consumer
of the crate, structurally identical to a sibling workspace
member that has `cstd = { path = "../cstd" }` in its
`Cust.toml`.

## 2. What v0.4.3 ships (MVP)

In-scope:

* New `tests/<file>.c` discovery convention (V43D-1).
* One CMake `add_executable(<crate>__itest__<stem> EXCLUDE_FROM_ALL ...)`
  target per integration-test file (V43D-5), emitted into
  the same workspace `CMakeLists.txt` v0.4.2 already
  generates (V42D-13).
* Each integration-test exe uses the same
  `[[cust::test]]`-based discovery as unit tests (V43D-4):
  the plugin emits a per-TU sidecar at
  `target/<profile>/.test-sidecars/<crate>/tests/<stem>.tsv`;
  the driver concatenates them into a per-exe runner TU at
  `target/<profile>/cmake/cust_itest_main_<crate>__<stem>.c`.
* Public-surface linkage (V43D-3): runner TU + integration
  source compile against `<crate>.h` and link against
  `lib<crate>.a`. No private headers, no `.cust.h` fragment
  access, no `[[cust::pub_crate]]` reach.
* `cust test -p <member>` continues to scope; an integration-
  test–only filter (e.g. `cust test --test <stem>`) is
  **deferred** (V43D-9 open question — see §6).
* Existing v0.3.2 test isolation guarantees apply unchanged:
  fork-per-test (V32D-7), per-test cwd under
  `target/<profile>/test/<crate>/<stem>/` (V32D-4 generalised
  to integration exes), exit code 101 on assertion failure
  (matches `cargo test`).

Out of scope — deferred to v0.4.5 ("`cust test` follow-ups"):

* **Shared helper code under `tests/`** (`tests/common/mod.c`
  or any other helper-sharing convention — V43D-2 flipped
  from locked-in-MVP to deferred). The compile-into-every-
  integration-exe machinery is mechanically straightforward
  but adds a second `MemberView`-shaped emission path
  (helper TUs vs integration TUs) and another precedent for
  filesystem-driven discovery rules. v0.4.3 MVP keeps the
  emission model crisp: every `.c` file under `tests/` (top
  level only) is a separate integration-test exe, full stop.
  Users who need shared code in v0.4.3 can `#cust use crate::<crate>;`
  and put helpers behind `[[cust::pub]]` in the CUT (less
  ideal — pollutes the public surface), or duplicate the
  helper inline per file (acceptable for the typical
  3–5-line helper that's common in early dogfood).
* `[[cust::test(inline)]]`, `should_panic`, per-test
  timeouts, `--nocapture` / `--exact` / `--test-threads N`
  flag plumbing, `[profile.test]` plumbing.
* Multi-file integration tests in Cargo's `tests/<name>/main.c`
  + sibling-modules form (the doc-locked deferral to v0.4.5
  for full Cargo-parity multi-file integration tests).
* Explicit `[[test]]` manifest entries (`Cust.toml` table
  with `name = "..."`, `path = "..."`, `harness = false`).
  v0.4.3 keeps "discovery is filesystem-driven, no manifest
  surface" — same posture v0.3.2 took for unit tests (V32D-1
  closer "No `Cust.toml` schema changes in v0.3.2").
* Doc tests (`cust test --doc`), benchmarks (`benches/`),
  examples (`examples/`). All on a separate v0.4.x track.

Out of scope — already shipped, just reaffirmed:

* The fork-per-test isolation harness (V32D-7) and runner
  template (`cust/src/test_runner_template.c`) — used
  verbatim by the new integration runner.
* `[[cust::test]]` plugin attribute + sidecar-emit mechanism
  (V40D-6) — used verbatim; integration TUs go through the
  same plugin invocation with `-DCUST_TEST_BUILD=1`.
* The v0.4.2 CMake-driven build backend (V42D-13, V42D-14,
  V42D-16) — extended with one extra
  `add_executable(<crate>__itest__<stem> EXCLUDE_FROM_ALL …)`
  per integration file, no other shape changes.

## 3. Why v0.4.3 now — and not v0.4.5

The original v0.4.x ordering put integration tests in v0.4.5
behind multi-bin (v0.4.4). Two pieces of evidence flipped this:

1. **Slice D pull-forward.** v0.4.3 was originally scoped as
   `--jobs` / `-jN` polish (cust-design.md §17). v0.4.2 V42D-13
   collapsed both intra- and inter-crate parallelism into one
   Ninja graph, and slice D shipped `--jobs` plumbing as part
   of v0.4.2. The v0.4.3 slot is now empty.

2. **Integration tests are independently shippable.**
   The MVP scope above touches three files (`cmake_emit.rs`,
   `workspace.rs`, one new module) and reuses v0.3.2's harness
   + v0.4.0's discovery + v0.4.2's CMake backend wholesale.
   Multi-bin (v0.4.4) is a deeper change — it adds a new
   `[[bin]]` array shape, multiple `add_executable` targets
   per member, and CLI changes for `cust run -p <member> --bin
   <name>` — that doesn't *block* integration tests in any
   direction. Both can ship in either order.

The independence is asymmetric in one direction: when v0.4.4
multi-bin lands, integration tests of a multi-bin crate
should still see only the **library** half (V43D-3 stays the
same — no integration tests of bin halves). This is the
same constraint v0.3.2 V32D-11 already encodes for unit tests
of bin-only members.

## 4. Boundary with v0.4.0 plugin & v0.4.2 CMake backend

v0.4.0 V40D-6 emits one test-discovery sidecar per **module**
inside the crate-under-test. v0.4.3 extends that to one
sidecar per integration-test TU as well, written to a sibling
path under `target/<profile>/.test-sidecars/<crate>/tests/`
instead of `target/<profile>/.test-sidecars/<crate>/`:

```
target/<profile>/.test-sidecars/cstd/
├── cstd__lib.tsv           # unit-test discovery (existing — V40D-6)
├── cstd__math.tsv
├── cstd__mem.tsv
├── cstd__geom.tsv
├── cstd__types.tsv
└── tests/                  # NEW in v0.4.3 — V43D-6 path scheme
    ├── basic.tsv
    └── alloc_pressure.tsv
```

The plugin needs **zero changes** to support this: it already
emits one sidecar per TU regardless of what the TU is for,
keyed off `-fplugin-arg-cust-test-sidecar-out=<path>`. The
v0.4.3 driver just passes a different `-out=` path per
integration TU. Same plugin binary, same sidecar format
(V40D-6 TSV), same `test_discovery::parse` reader.

The v0.4.2 CMake backend (`cmake_emit::collect_view` →
`emit_workspace_cmakelists`) gains one new emission per
integration file:

```cmake
# ---- crate: cstd (integration test: basic) ----
add_executable(cstd__itest__basic EXCLUDE_FROM_ALL
    "/path/to/cwork/ccrates/cstd/tests/basic.c"
    "/path/to/target/debug/cmake/cust_itest_main_cstd__basic.c"
)
set_target_properties(cstd__itest__basic PROPERTIES
    RUNTIME_OUTPUT_DIRECTORY "/path/to/target/debug/test/cstd/basic"
    OUTPUT_NAME basic
)
target_include_directories(cstd__itest__basic PRIVATE
    "/path/to/target/debug/build/cstd/include"      # for <crate>.h
)
target_link_libraries(cstd__itest__basic PRIVATE
    cstd                                            # the CMake lib target
)
target_compile_options(cstd__itest__basic PRIVATE
    # Same compile options as the unit-test target, including:
    "-DCUST_TEST_BUILD=1"
    "SHELL:-fplugin=…/libcust_plugin.so"
    "-fplugin-arg-cust-test-sidecar-out=…/.test-sidecars/cstd/tests/basic.tsv"
    # …
)
```

`EXCLUDE_FROM_ALL` matches the v0.4.2 slice F refinement
(V42D-14): integration test targets exist unconditionally in
the CMakeLists but stay out of the default `cmake --build`
graph. `cust test` reaches them via `--target
<crate>__itest__<stem>` exactly the way it already reaches
`<crate>__test`.

## 5. Decisions (V43D-N — in progress)

### V43D-1 — Discovery: `tests/*.c` only, recursive forbidden

**Locked: A. Every `.c` file at the top level of `<crate>/tests/`
is one integration-test source. Subdirectories are NOT
recursed. Non-`.c` files are ignored. Within a crate,
integration exes run in alphabetical-by-`<stem>` order
(Cargo parity); the discovery walk sorts stems so the
`Running tests/<file>.c` banner sequence is deterministic
regardless of filesystem readdir order.**

Matches Cargo's `tests/<file>.rs` exactly. A user who wants
multi-file integration tests waits for v0.4.5's Cargo
`tests/<name>/main.c` form (deferred — V43D-9), or duplicates
helpers inline per file in the meantime (V43D-2 deferral note).

**Rejected: B. Recurse into `tests/**/*.c` and treat each
file as its own exe.** Loses the "one file = one exe" property
that makes the `Running tests/<file>.c` banner unambiguous,
and pre-commits to a subdir-semantics decision that v0.4.5's
`tests/<name>/main.c` form needs to revisit. Cargo's recursion
rule is "top-level only," and we mirror it.

**Rejected: C. Discovery from `Cust.toml` `[[test]]` array
only.** Forces every test to be manifest-registered, which is
hostile to "add a file, get a test" iteration. Deferred to
v0.4.5 as an *additive* layer (filesystem-discovered tests
still work; manifest entries override paths or add custom
names).

### V43D-2 — `tests/common/` exclusion — **DEFERRED to v0.4.5**

**Deferred.** Originally locked as: files matching
`tests/common/*.c` are shared helper TUs, compiled into every
integration test exe in the same crate's `tests/` dir, with
`tests/common/` on the include path. Removed from v0.4.3 MVP
to keep the emission model crisp: in v0.4.3 every `.c` file
under `tests/` (top level only) is a separate integration-test
exe, full stop. Any subdirectory under `tests/` is silently
ignored (consistent with V43D-1's no-recursion rule).

The full helper-sharing design (`tests/common/mod.c` + the
`tests/common/` include path + the `mod.c`-required-if-dir-
exists rule + the per-exe rebuild-on-common-change edges)
lands in v0.4.5 alongside the Cargo `tests/<name>/main.c`
multi-file form; both share the "how does v0.4.3's no-
subdirs-under-tests/ rule extend?" question, and pricing
them together is cheaper than pricing each separately.

Until then, users who need shared helper code across
integration tests can:

* Promote the helper to the CUT's public surface with
  `[[cust::pub]]` — least ideal, pollutes the production
  API for testing-only code.
* Duplicate the helper inline per integration file —
  acceptable for short helpers (`cstd_test_alloc` wrapper
  fits in ~5 lines) and what the v0.4.3 cwork dogfood
  does (V43D-7).
* Add a `tests-shared` sibling crate in the workspace and
  depend on it from `Cust.toml` — heavier but works today
  via v0.3 workspace deps.

No design constraint imposed on v0.4.5: when the time comes,
`tests/common/mod.c` will either match this section's
original proposal or take Cargo's `tests/<name>/main.c +
tests/<name>/<sibling>.c` form depending on which the v0.4.5
design pass prefers.

### V43D-3 — Linkage: public surface only

**Locked: A. Each integration test exe links against
`lib<crate>.a` (the CUT's static archive) and includes
`<crate>.h` (the concatenated public header). No
`.h-fragments/<crate>/*.cust.h` access, no
`[[cust::pub_crate]]` reach, no `src/*.h` access.**

This makes integration tests **structurally identical to a
downstream workspace member**. The integration test exe sees
exactly what `hello-cstd` sees today, no more. This:

* validates the publicness of the surface: if a test needs a
  decl that wasn't `[[cust::pub]]`-marked, the user discovers
  it at integration-test time instead of at "second downstream
  consumer" time;
* keeps the test-build CMake target shape simple (no special
  fragment-header include paths);
* preserves v0.3.2 V32D-1's "unit tests = colocated in `src/`"
  posture for crate-private testing.

What integration tests CANNOT reach — and **how the boundary
is enforced** (two distinct mechanisms, don't conflate them):

* `[[cust::pub_crate]]` decls in the CUT — **header-level**
  enforcement. These don't appear in the published
  `<crate>.h`, so a `tests/*.c` that references one fails at
  **compile time** with an implicit-declaration / unknown-
  identifier error. (The symbol may still have external
  linkage in `lib<crate>.a`; the boundary is the header, not
  the archive's symbol table. A user who hand-writes a
  forward declaration could defeat it — that's accepted, the
  same way a downstream consumer who forward-declares a
  private symbol can.)
* Unannotated `static` decls in the CUT — **linkage-level**
  enforcement. These have internal linkage and aren't in the
  archive's exported symbol table, so even a hand-written
  forward declaration fails at **link time** with an
  undefined-reference error.
* `[[cust::test]]` decls in the CUT's `src/**.c` files —
  **linkage-level**. The V40D-14 plugin pass marks them
  `InternalLinkageAttr` in non-`CUST_TEST_BUILD` builds, and
  the CUT's archive here is the v0.4.2 lib half (not the
  test half), so they're absent from the exported symbol
  table. See V43D-7 for the consequence on the cwork dogfood.

**Rejected: B. Integration tests see the same surface as
unit tests (crate-private).** Loses the Cargo-parity "tests/
is a downstream consumer" property, which is the
architectural value-add of having a separate `tests/` dir at
all. Users who want crate-private testing already have
unit tests under `src/`.

**Rejected: C. Per-`[[test]]` configurable.** Defers the
question rather than answering it; adds manifest surface
v0.4.3 explicitly avoids (V43D-1's deferral logic).

### V43D-4 — Test-fn discovery: same `[[cust::test]]` mechanism

**Locked: A. Integration-test fns are declared the same way
as unit tests — with `[[cust::test]]`. One integration-test
`.c` file holds N test fns; the per-exe runner iterates them.**

Reuses v0.4.0 V40D-6 plugin sidecar emission, v0.3.2 V32D-7
fork-per-test harness, and v0.3.2's `test_runner_template.c`
verbatim. The new code is:

* Driver-side: extend `cmake_emit::collect_view` to recognise
  `tests/*.c` and populate one extra `MemberView::integration_tests`
  Vec; add a **sibling** `emit_integration_test_executable`
  (not an extension of `emit_test_executable` — integration
  exes *link against* `lib<crate>.a` and compile only
  `tests/<stem>.c` + the runner TU, whereas the unit-test
  target *recompiles* the lib sources into the exe, so the
  two emitters have structurally different source/link
  shapes); extend `write_test_runner_tu` (today named for
  unit tests) to also produce per-integration-exe runner TUs.
* No plugin changes.
* No `test_runner_template.c` changes.

**Rejected: B. Each `tests/<file>.c` is itself a single test
(Cargo's pre-2015 model where the file *is* the test, with
its own `int main`).** This is older Cargo behaviour and
would mean integration test files contain `int main(void)`,
which conflicts with the V32D-7 fork-harness's
runner-supplied `main`. Loses the "N tests per file" property
that makes assertion-failure isolation useful.

**Rejected: C. Hybrid (user picks per file).** Doubles the
mental model + the implementation surface; no user demand
yet.

### V43D-5 — One CMake target per integration file

**Locked: A. One `add_executable(<crate>__itest__<stem>
EXCLUDE_FROM_ALL ...)` per `tests/<file>.c`. Naming:
`<crate>__itest__<stem>` (CMake target name) /
`<stem>` (`OUTPUT_NAME`, on-disk filename).**

The double-underscore `__itest__` infix:

* Disambiguates from the unit-test target
  (`<crate>__test`, V42D-14) at the CMake target-name level.
* Makes `--target <crate>__itest__<stem>` selectable from
  `cust test --test <stem>` (when v0.4.5 lands that flag).
* Survives `<stem>` containing characters CMake target names
  forbid: `<stem>` is restricted to `[A-Za-z0-9_]` and
  hyphens (V43D-8 below); other characters error at config
  time.

`RUNTIME_OUTPUT_DIRECTORY = target/<profile>/test/<crate>/<stem>/`
plus `OUTPUT_NAME = <stem>` (no infix) lands the on-disk file
at `target/<profile>/test/<crate>/<stem>/<stem>`, matching the
`Running tests/<file>.c (target/.../test/<crate>/<stem>/<stem>)`
banner. The per-stem subdirectory is deliberate: it is also
the exe's cwd (V43D-11), mirroring the unit-test invariant
"cwd = the directory containing the exe" (V32D-4, where the
unit exe is `test/<crate>/<crate>` and cwd is `test/<crate>/`).
Without the subdir the on-disk file `test/<crate>/<stem>`
and the per-stem cwd directory `test/<crate>/<stem>/` would
collide (a file and a directory of the same name).

**Rejected: B. One `add_executable(<crate>__itest_all ...)`
with all integration sources linked together, runner iterates
across-file boundaries.** Loses the "one file = one fork" /
"one file = one assertion-failure boundary" property. Cargo
ships one exe per file specifically because users expect a
failing test in `basic.c` not to abort the test discovery
in `alloc_pressure.c`.

**Rejected: C. Lift integration tests into the v0.4.4 multi-
bin design.** Architecturally wrong: integration tests are
not "binaries the crate produces for downstream consumers,"
they're test harnesses with a runner-supplied main. They
should live next to `<crate>__test`, not next to bin targets.

### V43D-6 — Sidecar path scheme: `tests/` subdir

**Locked: A. Integration-test sidecars live at
`target/<profile>/.test-sidecars/<crate>/tests/<stem>.tsv`.
Unit-test sidecars stay at
`target/<profile>/.test-sidecars/<crate>/<qname>.tsv` (V40D-6).**

The extra `tests/` subdirectory:

* Avoids a name collision in the rare case a unit-test
  module qname matches a `tests/` file stem (e.g. `cstd`
  has a `src/foo/mod.c` ⇒ qname `foo` *and* a `tests/foo.c`
  ⇒ stem `foo`). Without the subdir these would race for
  the same sidecar path.
* Mirrors the `tests/` source-tree subdir, making the cache
  layout self-documenting.
* Costs one `fs::create_dir_all` per crate-with-integration-
  tests at config time, paid once per `cust test` run.

The path is constructed by `target_layout::test_sidecar_path`
(today `(crate_name: &str, qualified_name: &str)`; extended to
take an additional `Origin::Unit | Origin::Integration` enum so
the integration variant inserts the `tests/` path segment).

### V43D-7 — cwork dogfood: cstd gets a `tests/` dir

**Locked: A. `cwork/ccrates/cstd/tests/basic.c` ships in the
v0.4.3 closeout (slice D-equivalent), with 3–4 integration
tests covering the public surface (`cstd_version`,
`cstd_max_i32`, `cstd_strlen`, `cstd_string_new_in` + free).**

Mirrors the v0.4.0 cwork rewrite pattern (cust-design.md §17
v0.4.0 bullet "Cwork rewrite as part of slice E"). The
dogfood validates V43D-3 (public-surface linkage) end-to-end:
if the tests need anything that isn't `[[cust::pub]]`-exported,
they fail to compile, surfacing the surface gap.

No `tests/common/` directory ships — V43D-2 is deferred. The
few helpers the cwork integration tests need (e.g. a
leak-checking alloc wrapper) are inlined per file. When v0.4.5
adds shared-helper support, the cwork dogfood will be revisited
to exercise the new shape.

### V43D-8 — `<stem>` validity rules

**Locked: A. `<stem>` is the file basename minus `.c`. It
must match `[A-Za-z][A-Za-z0-9_-]*` (same shape as a Cust
package name). Stems that don't validate error at config
time with `tests/<file>.c: stem '<bad>' must match
[A-Za-z][A-Za-z0-9_-]* (used as the CMake target name + on-
disk filename)`. Stems that collide with the unit-test exe
name (`<stem> == <crate>`) error with `tests/<crate>.c:
integration-test stem '<crate>' collides with the unit-test
executable; rename the file`.**

The CMake-target-name characters that CMake actually forbids
are tighter than `[A-Za-z0-9_]` (no leading digit, no
hyphens in some contexts), but `__itest__` infix + the
`[A-Za-z]` leading-char rule mean the full CMake target name
`<crate>__itest__<stem>` always validates. The on-disk
collision check protects the v0.3.2 unit-test exe path: the
unit exe is the *file* `target/<profile>/test/<crate>/<crate>`,
while a `tests/<crate>.c` would want the *directory*
`target/<profile>/test/<crate>/<crate>/` for its per-stem cwd
(V43D-5/V43D-11) — a file and directory of the same name
can't coexist, so `<stem> == <crate>` is rejected at config
time.

**Rejected: B. Allow any filesystem-valid stem and quote
heavily.** Punts CMake-name quoting hazards downstream;
v0.4.4 multi-bin design has independently confirmed the
`[A-Za-z][A-Za-z0-9_-]*` rule for `[[bin]] name = …`.

### V43D-9 — `--test <stem>` filter flag

**Open (deferred to v0.4.5 unless cheap).** Cargo's `cargo
test --test <name>` builds + runs only the named integration
test exe. The v0.4.3 implementation cost is "tiny" because
`cmake --build --target <crate>__itest__<stem>` is already
how single-exe selection works (V43D-5).

The deferral question is whether v0.4.3 should ship `--test
<stem>` or only `cust test` (all integration tests). Open
arguments for shipping:

* Implementation is 10 lines (clap flag + one `--target`
  passthrough).
* It's how users actually iterate on a single test file.

Against:

* Adds a CLI surface that needs documentation, error
  messages for unknown stems, and interaction rules with
  the existing positional filter (which the v0.3.2
  harness already accepts — see `test_runner_template.c`
  argv parsing).
* The v0.4.5 flag-plumbing milestone explicitly groups
  `--exact`, `--nocapture`, `--test-threads N`, `--test
  <name>` together for one cohesive design pass.

Lean: defer to v0.4.5 with the rest of the CLI work, ship
v0.4.3 with bare `cust test` only. The v0.4.5 deferral note
in this doc will reference V43D-9.

### V43D-10 — Test failure exit code propagation

**Locked: A. `cust test` exits non-zero iff ANY test exe
(unit or integration, across all members) reported a
non-zero exit. Otherwise exits 0.**

Unchanged from v0.3.2 V32D-9 behaviour for unit tests;
explicitly extended to cover the new integration exes. The
test-runner template's existing exit-101-on-assertion-failure
(V32D-7) maps to "this exe failed" for cust-test's
aggregation purposes.

### V43D-11 — Per-test cwd for integration tests

**Locked: A. Each integration test exe runs with cwd =
`target/<profile>/test/<crate>/<stem>/` — the directory that
contains the exe itself (the exe is
`target/<profile>/test/<crate>/<stem>/<stem>`, V43D-5). The
directory is created fresh per `cust test` run.**

Mirrors v0.3.2 V32D-4 for unit tests exactly: there the cwd
(`test/<crate>/`) is the directory containing the unit exe
(`test/<crate>/<crate>`); here the cwd (`test/<crate>/<stem>/`)
is the directory containing the integration exe
(`test/<crate>/<stem>/<stem>`). The per-stem subdirectory
keeps integration tests from clobbering each other's output
files. Same fork-per-test still applies inside each exe
(V32D-7).

The per-stem subdir is the natural disambiguator: there's
only one unit-test exe per crate but N integration exes, and
nesting each integration exe in its own `<stem>/` directory
is what lets the exe file and the per-stem cwd directory
coexist (V43D-5 — otherwise `test/<crate>/<stem>` would have
to be both a file and a directory).

### V43D-12 — Empty `tests/` directory

**Locked: A. An empty `tests/` directory (or one with only
non-`.c` files at the top level) is silently ignored. Same
as no `tests/` dir at all. Subdirectories under `tests/`
(including any future `tests/common/`) are also silently
ignored in v0.4.3 — V43D-1 explicitly forbids recursion and
V43D-2 defers the helper-sharing design.**

Cargo-parity. Avoids a configure-time error for users who
git-add an empty `tests/` skeleton.

When v0.4.5 lifts the subdirectory restriction (the V43D-2
deferral + the Cargo `tests/<name>/main.c` form), this rule
relaxes accordingly; v0.4.3 keeps the strict "flat tests/
or nothing" shape so the relaxation is purely additive.

### V43D-13 — `cust check` and integration tests — **DEFERRED**

**Deferred (revised at slice E).** Originally locked as:
`cust check` runs the surface pass over `tests/*.c` the same
way it does for `src/**.c`, reporting syntax errors with
file:line. Implementation revealed the premise was wrong:
**`cust check` in v0.4.2 (V42D-15) is a *tolerant*
surface-extraction pass, not a strict syntax checker** — it
suppresses clang diagnostics (stderr → `/dev/null`) because
unresolved cross-module references are expected during
fragment extraction. A deliberately broken `src/**.c` file
passes `cust check` clean today. Extending that tolerant pass
to `tests/*.c` would add code that delivers none of the
"see syntax errors immediately" value the decision promised,
and making `tests/` *strict* while `src/` stays *tolerant*
would be a surprising asymmetry.

The correct fix — making `cust check` strict for both
`src/**.c` and `tests/*.c` — is a `cust check` redesign that
belongs in its own milestone (tracked alongside the v0.4.5
`cust test` follow-ups). Until then, `cust check` ignores
`tests/` entirely (no walk, no surface pass): integration
syntax errors surface at `cust test` time via the real CMake
build, with full file:line diagnostics from clang.

RQ-V43-1's `cust check --no-plugin` extension is deferred
with it (there's nothing for the scanner to walk in `tests/`
until `cust check` gains a strict mode).

## 6. Open questions (RQ-V43-N)

### RQ-V43-1 — How does `cust check --no-plugin` (V40D-10) discover integration tests? — **DEFERRED with V43D-13**

**Deferred.** Originally locked as: extend the existing
`mod_scanner` regex pass to walk `<crate>/tests/*.c`. This
rides on V43D-13 (`cust check` over `tests/`), which slice E
deferred once implementation revealed `cust check` is a
*tolerant* surface pass, not a strict syntax checker. With
`cust check` ignoring `tests/` entirely in v0.4.3, there's
nothing for the `--no-plugin` scanner to walk there yet. The
extension lands when `cust check` gains a strict mode (same
milestone as V43D-13).

### RQ-V43-2 — Integration test compile-time defines (asymmetric `CUST_TEST_BUILD=1`)

**Locked: the asymmetric build is safe.** Integration tests
compile with `-DCUST_TEST_BUILD=1`; the CUT's `lib<crate>.a`
they link against was built without it. The define only
affects two well-understood sites:

1. The v0.3.2 V32D-3 prelude's panic-impl forward-decl,
   which the runner template (compiled in the same exe with
   `CUST_TEST_BUILD=1`) provides exactly once.
2. The plugin's V40D-14 attribute-attach branch on
   `[[cust::test]]` decls (external linkage in the
   integration TU vs. `static unused` in the CUT's lib
   half — but `[[cust::test]]` fns aren't part of the
   public surface anyway, so the linker sees nothing
   asymmetric across the boundary).

The public surface (`[[cust::pub]]` decls in `<crate>.h`)
is identical regardless of `CUST_TEST_BUILD`, so the linker
contract between the test exe and the lib archive is
satisfied. No additional design work needed; the slice C
runner-TU generation simply passes `-DCUST_TEST_BUILD=1`
on the integration target's `target_compile_options`
exactly the way the v0.4.2 V42D-14 unit-test target already
does.
