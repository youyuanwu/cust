# cust v0.3 — workspaces & path dependencies

**Status:** ✅ **ready for implementation** (2026-06-04).
**Parent doc:** [cust-design.md §17 roadmap](cust-design.md).
**Previous milestone:** [v0.2.md](v0.2.md) (shipped).

This file is the planning doc for v0.3 — scope, locked design
decisions (V3D-N), and the verification target. Architectural detail
for individual subsystems lives in the parent design doc; anything
that lands as a permanent contract during v0.3 gets its own
`docs/spec/<name>.md` and is referenced from here.

### Reframed scope (2026-06-04)

Earlier drafts of v0.3 tried to land a registry, network fetch, and
semver resolution algorithm in one milestone. That conflates *three*
independent risks: the dep-graph plumbing inside the driver, the
registry wire protocol (a v1-blocking contract — OQ-12), and the
algorithmic resolver. **v0.3 now ships only the first.** Concretely:

* **In scope:** workspaces (`[workspace]` in `Cust.toml`), path
  dependencies (`dep = { path = "../other" }`), and the multi-
  crate build orchestration those imply.
* **Out of scope, deferred:** registry, network fetch, git deps,
  semver resolution, `cust add` against a registry. All to v0.4+
  once the in-tree dep plumbing is proven.

Why this split is the right shape: every piece of v0.3 work (member
discovery, topological build order, cross-crate header lookup, lock
shape for path-only graphs, the new `#cust use <name>;` syntax) is a
prerequisite for the network-dep milestone. Doing it first against
in-tree deps lets us shake out the resolver/scheduler/link plumbing
with zero network risk and zero version-selection risk, then layer
the registry on top as a second `Index` source. None of v0.3's
contracts (lock format, dep dir layout, `#cust use` syntax) are
invalidated by the later registry work.

This also pulls **workspaces** forward from the original v0.8 slot
in [cust-design.md §17](cust-design.md); see the "Roadmap impact"
section at the end of this doc.

### Headline outcome

A user can lay out a repo like

```
my_project/
├── Cust.toml          # [workspace] members = ["app", "util"]
├── Cust.lock
├── app/
│   ├── Cust.toml      # depends on util = { path = "../util" }
│   └── src/lib.c
└── util/
    ├── Cust.toml
    └── src/lib.c
```

run `cust build` at the workspace root, and have both crates
compile in topological order (`util` first, then `app`), with
`app`'s sources able to `#cust use util;` to reach `util`'s
`[[cust::pub]]` surface. Single shared `target/` at the workspace
root, single `Cust.lock`, single `compile_commands.json`.

## Scope

The big rocks, in suggested implementation order. Decisions called
out as **V3D-N** below the scope list are blockers for the
corresponding rocks.

1. **`[workspace]` manifest table.** Parse `[workspace]` in
   `Cust.toml`; require a `members` list of relative paths (string
   array). Reject unknown subkeys at parse time, matching v0.1/v0.2
   strictness. Workspace shape (virtual root vs root-is-also-a-
   member) and globbing in `members` are **V3D-1** and **V3D-2**
   respectively.
2. **Manifest discovery, workspace-aware.** Extend the existing
   "walk up from cwd looking for `Cust.toml`" algorithm
   ([v0.1.md "Manifest discovery"](v0.1.md)) to identify the
   workspace root: keep walking past the first `Cust.toml` if it's
   a member of an enclosing `[workspace]`. This matches Cargo's
   algorithm. After v0.3 the *workspace root* is the canonical
   anchor; the per-crate manifest is the *member* manifest.
3. **Path-dep manifest schema activation.** Relax the v0.1/v0.2
   "populated `[dependencies]` is an error" check in
   `cust/src/manifest.rs` and accept the path-dep form
   `dep = { path = "../other" }` plus the `{ path, features,
   default-features, optional }` table form. `version`, `git`,
   `tag`, `branch`, `rev`, and `[target.'cfg(...)'.dependencies]`
   stay rejected at parse time with `not yet supported in v0.3`
   errors. `[build-dependencies]` and `[dev-dependencies]` stay
   gated (no test harness yet; build scripts deferred per the
   table below). See **V3D-3** for the exact accept/reject list.
4. **Path resolution.** A path dep is resolved by `(parent
   manifest dir).join(path).canonicalize()`. The result must point
   at another crate inside the same workspace's `members` set
   (workspace boundary is enforced — out-of-workspace path deps are
   rejected with a clear "must be a workspace member" error;
   rationale in **V3D-4**). The dep graph is then built from these
   edges.
5. **Topological build scheduler.** Single-threaded for v0.3
   (parallelism stays at v0.4). Build members in reverse topological
   order; each member uses the v0.2 pipeline unchanged. Cycle
   detection: any cycle in the path-dep graph is rejected with
   `error: dependency cycle: a → b → a` (self-cycle prints
   `error: dependency cycle: a → a`). The cycle's display order
   starts at the alphabetically-first member name in the cycle to
   make the error deterministic across runs. The build is short-
   circuited per crate by the existing v0.2 stamping
   (`.cust-version`, fragment-header byte-equal) — no new
   incrementality work in v0.3.
6. **Per-member target layout.** Shared `target/` at the
   workspace root. Per-member outputs land under
   `target/<profile>/build/<member-name>/...` (the v0.2 path,
   already keyed by crate name), and the new dep dir lives at
   `target/<profile>/deps/<member-name>/` as a **symlink** to the
   member's own build dir (decided in **V3D-5**), so consumers
   reach `lib<name>.a` and `include/<name>.h` via a stable
   workspace-root-relative path that does not depend on where in
   the source tree the producer crate lives. The symlink is
   recreated after each producer build to handle the
   first-build-of-a-new-member case.
7. **Dep surface wiring.** A new directive `#cust use <name>;`
   (no `crate::` prefix, distinct from v0.2's `#cust use
   crate::<module>;`) is recognised by `mod_scanner` and rewritten
   by the driver to `#include
   "<workspace-root>/target/<profile>/deps/<name>/include/<name>.h"`
   — the path is absolute so the resulting `#include` is
   independent of where the consumer source file lives in the
   workspace tree. The producer crate must declare the symbol as
   `[[cust::pub]]` for it to appear in the consumer's view
   (`pub(crate)` is module-private within the producer, and stays
   invisible at the dep boundary). See **V3D-6** for the exact
   syntax decision.
8. **Final link for library consumer crates.** A consumer's
   `lib<name>.a` contains only that consumer's own objects;
   transitive dep `.a`s are **not** bundled, matching Cargo's
   `rlib` semantics. Downstream binary consumers do the cross-
   crate link themselves; v0.3 ships no binary-crate support (see
   **V3D-7**), so this case is exercised in v0.3 only by the
   integration test that drives `clang` over a workspace's
   `lib<root>.a` plus each dep's `lib<dep>.a` from the shared
   `target/<profile>/deps/<dep>/` path.
9. **`Cust.lock` (path-only form).** Even with no version
   resolution, the lock records the resolved graph for
   diagnostics: workspace member list, edges between them, and the
   `workspace_root` absolute path at lock time (so a moved-but-
   not-rebuilt workspace produces a clear error). Format: TOML;
   `lock_format_version = 1`. The lock lives at the workspace
   root next to the root `Cust.toml`. No source hashes in v0.3
   (path deps bypass hashing — Cargo does the same; see **V3D-8**
   for the reasoning and the v0.4+ extension shape for when
   registry deps land).

   **Single-crate (non-workspace) builds.** A `Cust.toml` with no
   enclosing `[workspace]` continues to work exactly as in v0.2:
   no `Cust.lock` is emitted (there are no edges), and a populated
   `[dependencies]` table is rejected at parse time with `error:
   path dependencies require a [workspace]; add it to a parent
   Cust.toml`. This keeps the v0.1/v0.2 single-crate happy path
   untouched.
10. **CLI surface bumps.**
    * `cust build [-p|--package <member>]` — without the flag,
      build every workspace member; with it, build that member
      and its transitive path deps. Long form `--package` matches
      Cargo; short form `-p` is the same alias.
    * `cust check [-p|--package <member>]` — same scoping.
    * `cust clean` — unchanged (still wipes the whole `target/`
      since it's shared).
    * `cust new <path> [--name <name>] [--lib]` — unchanged
      from v0.1; v0.3 does **not** add `--workspace` or `cust
      init --workspace` (creating workspaces is a one-line
      `Cust.toml` edit and not worth a flag yet).
    * `cust add` — **not in v0.3**. Without a registry, the only
      thing `cust add` could do is edit `[dependencies]` to add a
      path entry, which is a one-line manual edit. Deferred to
      v0.4 alongside registry deps where it pays its weight.

## v0.3 design decisions

All nine decisions below are locked (✅ decided 2026-06-04). Each
records the question, the options that were on the table, and the
concrete behaviour the implementer commits to. Order matches the
dependency arrows between the scope items, not the scope list
order.

### V3D-1: Workspace shape — **decided: B (both, like Cargo)**

**Status:** ✅ decided (2026-06-04). Picks option B below.

**Question:** Does v0.3 support both *virtual* workspaces (root
`Cust.toml` has only `[workspace]`, no `[package]`) and *root-is-
also-a-member* workspaces (root has both)?

Live options:

* **A — virtual only.** Workspace root has `[workspace]` and
  nothing else; the root directory contains no `src/`. All crates
  are explicit `members`. Pro: simplest; no ambiguity over what
  "the root crate" means. Con: forces a `members = ["."]` ish
  pattern for users who want one crate at the top.
* **B — both, like Cargo.** Root manifest may have *either* or
  *both* of `[workspace]` and `[package]`. If both, the root is
  itself an implicit member. Pro: matches Cargo; users coming
  from Rust get no surprise. Con: extra parser cases; the
  "implicit member" rule is a small footgun (renaming root
  package silently changes member set).
* **C — root-is-always-a-member.** Root manifest must have
  `[package]`; `[workspace]` adds *extra* members. Pro: no
  virtual-vs-real distinction. Con: forces every workspace to
  have a "main" crate at the root, which doesn't match real-world
  Rust workspaces (e.g. `cargo new --workspace` creates a
  virtual root).

**Decision: B.** Match Cargo. Both virtual and root-is-also-a-member
workspaces are accepted. If the root manifest has both `[package]`
and `[workspace]`, the root is an implicit member and does not need
to appear in `members`. If the root has only `[workspace]`, it is a
virtual workspace and the root directory must not contain `src/`
(rejected with `error: virtual workspace root must not contain
src/`). The "implicit member" footgun is mitigated by
`cust build -v` logging the resolved member list (including which
entry is the implicit root) to stderr at the start of every build.

### V3D-2: Member list — literal paths only, or globs? — **decided: A (literal only)**

**Status:** ✅ decided (2026-06-04). Picks option A below.

**Question:** Does `members` accept globs (`members = ["crates/*"]`)
or only literal directory paths in v0.3?

Live options:

* **A — literal only.** `members = ["app", "util", "crates/cli"]`.
  Pro: trivial; no glob library; no ambiguity about which dirs
  match. Con: large workspaces (10+ members) get verbose.
* **B — globs from day 1.** Pull in `glob` crate; accept
  `crates/*`. Pro: matches Cargo. Con: glob semantics on
  workspace-relative paths have edge cases (trailing slash,
  hidden dirs, symlinks); shippable but more test surface.
* **C — literal in v0.3, globs in v0.4.** Pro: smallest v0.3.
  Con: existing Cargo users will be surprised.

**Decision: A.** Each entry in `members` is a literal relative
directory path; glob characters (`*`, `?`, `[`) in member entries
are rejected at parse time with `error: globs in members are v0.4+
(list directories literally)`. Paths are interpreted relative to
the workspace root, must not escape it (`..` segments resolving
above the root are rejected), and must point at a directory that
contains a `Cust.toml`. Globs land in v0.4 as an additive
extension that does not break v0.3 manifests.

### V3D-3: Manifest dep-spec strictness — **decided: A (strict whitelist)**

**Status:** ✅ decided (2026-06-04). Picks option A below.

**Question:** Which `[dependencies]` entry shapes does v0.3
*accept* vs. *reject* at parse time?

The full schema [cust-design.md §3](cust-design.md) lists
`version`, `path`, `git`, `tag`, `branch`, `rev`, `features`,
`default-features`, `optional`, and the
`[target.'cfg(...)'.dependencies]` table.

v0.3-specific options:

* **A — strict whitelist** of (`path`, `features`,
  `default-features`, `optional`). Reject `version`, `git`,
  `tag`, `branch`, `rev`, and `[target.'cfg(...)'.dependencies]`
  at parse time with `not yet supported in v0.3, available in
  v0.4` errors. Pro: early, clear errors; matches v0.1/v0.2
  house style. Con: a manifest valid at v0.4 doesn't parse at
  v0.3.
* **B — accept `version` as well**, but require it to coexist
  with `path` (so `dep = { path = "../x", version = "0.2.0" }`
  is legal) and reject `version` *alone*. Pro: matches Cargo's
  habit of recording both for crates that may be published.
  Con: late error model; user types `version` thinking it'll
  download something and gets a confusing build.
* **C — accept `path`-only, reject everything else including
  `features`/`default-features`/`optional`.** Pro: smallest
  surface. Con: features are needed almost immediately
  for real crates.

**Decision: A.** Accepted entry shapes in v0.3:

* `dep = { path = "../x" }`
* `dep = { path = "../x", features = ["…"] }`
* `dep = { path = "../x", default-features = false }`
* `dep = { path = "../x", optional = true }`
* any combination of the above.

Everything else is a parse-time error pointing at v0.4. The bare
semver shorthand (`dep = "1.4"`) is rejected since there is
nothing to resolve against. `[target.'cfg(…)'.dependencies]`
tables are rejected wholesale (the cfg-expression evaluator is
v0.4+). `[build-dependencies]` and `[dev-dependencies]` stay
gated at "must be empty" per the v0.1/v0.2 rule (no build scripts
and no test harness in v0.3).

### V3D-4: Workspace-boundary enforcement for path deps — **decided: A (reject out-of-workspace paths)**

**Status:** ✅ decided (2026-06-04). Picks option A below.

**Question:** Are out-of-workspace path deps (`{ path =
"../../some_other_repo" }`) accepted in v0.3?

Live options:

* **A — reject out-of-workspace paths.** Path dep targets must
  resolve to a directory listed in the enclosing workspace's
  `members`. Pro: every dep is governed by the same lockfile and
  shared `target/`; no ambiguity about "whose target dir does
  this dep build into". Con: less flexible than Cargo (Cargo
  silently makes the out-of-tree path dep a "non-workspace
  member").
* **B — accept; the dep is built into the consumer's
  `target/`.** Pro: matches Cargo's flexibility. Con: the
  out-of-tree dep doesn't see the consumer's workspace lockfile
  or `target/`, so a transitive dep cycle with the consumer's
  workspace creates ambiguous resolution.
* **C — accept but warn**, with the warning saying "you probably
  want this as a workspace member".

**Decision: A.** Every path dep must resolve (via
`parent_manifest_dir.join(path).canonicalize()`) to a directory
listed in the enclosing workspace's `members` set. Out-of-workspace
resolutions are rejected with `error: dependency '<name>' resolves
to <abs path> which is not a workspace member; add it to
[workspace] members or move it inside the workspace tree`. This
guarantees a single shared `target/` and a single `Cust.lock`
govern every dep, and matches v0.3's framing ("in-tree only,
registry layered on top later"). The Cargo-style "non-workspace
path dep" relaxation can be revisited in v0.4+ once the registry
source exists to compare it against.

### V3D-5: Dep dir layout under shared `target/` — **decided: A (symlink farm)**

**Status:** ✅ decided (2026-06-04). Picks option A below.

**Question:** What lives at `target/<profile>/deps/<member-name>/`?

Live options:

* **A — symlink farm.** `target/<profile>/deps/<name>/` is a
  symlink to `target/<profile>/build/<name>/` (which already
  holds the v0.2 outputs for that member). Zero copy. Pro:
  trivial; fast; storage-efficient. Con: symlinks on tar/cp/git
  archive operations need care; Windows portability story is
  ugly (NTFS symlinks need admin or developer mode).
* **B — file copy.** Driver copies `lib<name>.a` and
  `include/<name>.h` from the producer's build dir to the
  consumer-facing dep dir after each producer build. Pro: no
  symlink portability question. Con: extra IO; cache discipline
  needs to clean stale copies.
* **C — no copy, no symlink — consumers compile against the
  producer's build dir directly.** The driver injects
  `-I<workspace-root>/target/<profile>/build/<name>/include` and
  links against `<workspace-root>/target/<profile>/build/<name>/lib<name>.a`
  directly. Pro: simplest of all; no extra fs layout. Con: leaks
  the producer's internal build-dir shape into the consumer's
  `compile_commands.json`, coupling it to the v0.2 layout in a
  way that the registry layer (v0.4) will then have to translate.

**Decision: A.** After each member's v0.2 build completes, the
driver creates `target/<profile>/deps/<name>/` as a symlink to the
member's `target/<profile>/build/<name>/` directory. Consumers
compile with `-I<workspace-root>/target/<profile>/deps/<name>/include`
and link against `<workspace-root>/target/<profile>/deps/<name>/lib<name>.a`,
both stable paths that v0.4 will keep when the registry source
lands (registry deps will materialise into the same shape under a
different sibling, e.g. `target/<profile>/deps/<name>-<version>-<hash>/`).
Windows is out of scope for v0.3 (deferred per the table below);
the Windows port in v0.5 may switch to **B** (file copy) if the
NTFS symlink permission story still bites.

### V3D-6: `#cust use` for cross-crate imports — **decided: A (`#cust use <name>;`)**

**Status:** ✅ decided (2026-06-04). Picks option A below.

**Question:** What's the syntax for "this source file imports
crate `<name>`'s public surface"?

Live options:

* **A — `#cust use <name>;`** (no `crate::` prefix). Pro: short;
  parallels Rust's `use foo;` for extern crates. Con: subtly
  similar to v0.2's `#cust use crate::<module>;` so the reader
  has to know that `crate::` means "intra-crate" and bare
  means "cross-crate".
* **B — `#cust use extern::<name>;`** Pro: unambiguous;
  future-proof for `super::`, `self::`. Con: nobody writes that
  in Rust; reads as bureaucratic.
* **C — overload `#cust use crate::<name>;` on resolution:**
  the driver first checks for a sibling module of that name,
  falls back to dep lookup. Pro: one syntax. Con: silent
  shadowing — a sibling module named the same as a dep wins,
  and the user has no syntactic way to disambiguate. Bad
  failure mode.

**Decision: A.** `#cust use <name>;` (bare ident, no `crate::`)
is the cross-crate import syntax; the existing
`#cust use crate::<module>;` stays the intra-crate import syntax
unchanged. The driver pre-pass distinguishes them by the literal
`crate::` prefix — the two paths through `mod_scanner` never
shadow each other. `<name>` must match an entry in the
consumer's `[dependencies]` table; otherwise a parse-time error
points at the missing dep. Rewrites to `#include
"<workspace-root>/target/<profile>/deps/<name>/include/<name>.h"`
(absolute path so the `#include` is independent of the
consumer source file's location in the workspace).

### V3D-7: Binary crates (`[[bin]]`) in v0.3? — **decided: A (library-only)**

**Status:** ✅ decided (2026-06-04). Picks option A below.

**Question:** Does v0.3 ship `crate-type = ["bin"]` / `cust
run`, or is the milestone library-only?

Live options:

* **A — library-only.** Workspaces with `staticlib` /
  (future) `cdylib` / (future) `rlib` members only. `cust run`
  stays at v0.4. Pro: smaller v0.3; defers the question of how
  multiple binaries in one workspace are named and dispatched
  (`cust run --bin foo`).
* **B — single-binary support.** A member crate may set
  `crate-type = ["bin"]`; the link step produces
  `target/<profile>/<member-name>` instead of `lib<name>.a`.
  `cust run` is *still* deferred to v0.4 (just build the binary,
  user runs it directly). Pro: makes workspaces actually useful
  for shipping things in v0.3. Con: doubles the link-step test
  surface.
* **C — full `cust run` + multiple binaries per crate (`[[bin]]`
  array in `Cust.toml`).** Pro: feature-complete. Con: large
  v0.3 scope.

**Decision: A.** v0.3 is library-only: every workspace member
produces a `staticlib` (the v0.2 default). `crate-type = ["bin"]`
is rejected at manifest parse time with `error: binary crates are
v0.4+`. `cust run` is not added. Binaries land in v0.4 alongside
`[[cust::test]]` (tests are conceptually binary crates linked
against the lib), where the link-step test surface can be
exercised in one milestone.

### V3D-8: `Cust.lock` for path-only graphs — **decided: A (emit minimal lock)**

**Status:** ✅ decided (2026-06-04). Picks option A below.

**Question:** Does v0.3 emit `Cust.lock` at all, given there's
nothing to resolve?

Live options:

* **A — yes, emit a minimal lock.** Records workspace members,
  edges, `lock_format_version = 1`, the absolute path of the
  workspace root at lock time, and a reserved-for-future
  `[metadata]` section. Pro: forces us to lock the lock schema
  *before* the registry needs it; v0.4 only has to *extend*
  the existing format with hashes and version pins.
  Con: lockfile that does no version-locking is a slight
  conceptual oddity for users.
* **B — no, defer `Cust.lock` to v0.4** when there's real
  pinning to do. Pro: smaller v0.3. Con: v0.4 ships *both* the
  registry resolver and the lock format in one milestone,
  bringing back the same "two risks in one bag" problem v0.3
  was designed to avoid.

**Decision: A.** v0.3 emits `Cust.lock` at the workspace root
with `lock_format_version = 1`. The v0.3 schema is the v1 draft;
v0.4 *extends* it with per-dep `[[dep.source]]` entries (kind,
registry, version, sha256, …) and bumps to `lock_format_version =
2` only if the extension turns out to be non-additive. The
schema is documented in `docs/spec/cust-lock.md` shipped
alongside this milestone. Initial v0.3 fields:

* `lock_format_version = 1`
* `workspace_root = "<absolute path at lock time>"` — a
  moved-but-not-rebuilt workspace produces a clear
  "workspace root changed; run `cust build` to regenerate"
  error instead of silently linking against stale paths.
* `[[member]]` table per workspace member: `name`, `version`,
  `path` (workspace-root-relative), and `dependencies` (the
  resolved edge list, names only — path is implied by the
  target member's own entry).

No source hashes in v0.3 (path deps don't need them; the v0.2
fragment-header stamping handles incrementality at the file
level). Source hashes are the v0.4 extension.

A separate `docs/spec/cust-lock.md` is the planned permanent home
for the schema; v0.3 produces the v1 draft (path-only); v0.4
extends it.

### V3D-9: Workspace inheritance (`[workspace.dependencies]`)? — **decided: B (defer to v0.4)**

**Status:** ✅ decided (2026-06-04). Picks option B below.

**Question:** Does v0.3 ship `[workspace.dependencies]` + per-member
`dep = { workspace = true }` (Cargo's workspace inheritance,
[cust-design.md §16 OQ-6](cust-design.md))?

Live options:

* **A — yes.** Lets a workspace centralise its dep specs in one
  place. Pro: matches Cargo; nice ergonomics. Con: another schema
  surface to lock in v0.3; only pays off once *registry* deps land
  (path deps are already centralised because they're literal
  relative paths).
* **B — no, defer to v0.4** alongside the registry. Pro: smaller
  v0.3 schema. Con: v0.4 manifest schema breaks v0.3 manifests
  that wrote `[workspace.dependencies]` (though no v0.3 manifest
  would, since v0.3 rejects it at parse time).

**Decision: B.** `[workspace.dependencies]` and the per-member
`dep = { workspace = true }` shape are rejected at parse time in
v0.3 with `error: workspace dependency inheritance is v0.4+`.
Path deps are already centralised structurally (the relative
path *is* the dedup), so inheritance buys nothing in a path-only
world; it pays its weight in v0.4 once registry deps land and
version strings start repeating across members.

## Out of scope for v0.3 (deferred per [cust-design.md §17](cust-design.md))

| Item | Lands in |
|---|---|
| Registry, network fetch, sparse-index protocol | v0.4 |
| `cust add <name>@<ver>` against a registry | v0.4 |
| Semver version resolution / `Cust.lock` source hashes | v0.4 |
| Full Cargo-grade resolver (PubGrub + backtracking) | v0.5+ |
| Cross-version dep coexistence in one graph | v0.5+ |
| Workspace inheritance (`[workspace.dependencies]`, OQ-6) | v0.4 (V3D-9) |
| `members` globs | v0.4 (V3D-2) |

**Rationale for the bigger deferrals:**

* **Registry / network → v0.4.** v0.3's central thesis is that the
  registry should be added as a second `Index` source on top of
  the workspace-member resolver, not as a coupled rewrite. The
  resolver/scheduler/link plumbing in v0.3 is exactly what v0.4
  reuses.
* **Build scripts → v0.4.** Earlier v0.3 drafts bundled
  `build.cust.c` in because registry crates would need them. With
  the registry deferred, build scripts have no v0.3 caller — even
  the test fixture would be hand-written. They land in v0.4
  alongside the first registry crate that wants one.
* **`cust add` → v0.4.** Without a registry, the only thing
  `cust add` could do is mutate `[dependencies]` to add a path
  entry — a one-line edit. The command is worth its weight only
  when it talks to a registry.
* **Full resolver, version coexistence → v0.5+.** Path-only graphs
  have no version dimension to resolve.

## Verification target

A v0.3 build of the headline-outcome workspace (`my_project` with
members `app` and `util`, where `app` depends on `util` via
`path = "../util"`) produces:

* **At the workspace root:**
  * `Cust.lock` with `lock_format_version = 1`, the two members,
    and the edge `app → util`.
  * `target/compile_commands.json` covering both members' TUs
    (paired entries per the v0.2 clangd story; single file at
    `target/` per the v0.1 layout, not per-profile).
* **Per-member, under `target/<profile>/build/<member>/`:**
  * `util/<modules>.o`, `util/libutil.a`, `util/include/util.h`
    (the v0.2 outputs, unchanged shape).
  * `app/<modules>.o`, `app/libapp.a`, `app/include/app.h`.
* **Dep view, under `target/<profile>/deps/<member>/`:**
  * `util` and `app` each present as a symlink to the
    corresponding `build/<member>/` directory (per V3D-5). The
    consumer's compile commands reach `util`'s archive and
    header through this path so the build is independent of the
    member's source-tree location.
* **Topological scheduling:** `util` builds before `app`; if
  `util` is unchanged across two `cust build` runs, `app` does
  not rebuild (existing v0.2 stamping handles this once the
  edge is encoded in the scheduler).
* **Negative paths:**
  * A `Cust.toml` with `[dependencies] foo = "1.0"` produces
    `error: version specs are v0.4+ (use { path = "..." }
    instead)` at parse time.
  * A path dep in a non-workspace `Cust.toml` produces
    `error: path dependencies require a [workspace]; add it to
    a parent Cust.toml`.
  * A path dep resolving outside the workspace produces
    `error: dependency 'bar' resolves to <abs path> which is
    not a workspace member`.
  * An import cycle (`a → b → a` via path deps) produces
    `error: dependency cycle: a → b → a`; a self-cycle
    produces `error: dependency cycle: a → a`.
  * `#cust use util;` in `app/src/lib.c` without `util` in
    `[dependencies]` produces `error: unknown dep 'util' — add
    it to [dependencies]`.

Linux only; Windows deferred per the table above. macOS is not
a supported platform.

Test count at v0.3 close: target **75+ unit + 50+ integration**,
`cargo clippy --all-targets` silent under the workspace
pedantic+nursery profile. Concrete numbers fold in as the slices
land.

## Open contracts established by v0.3

These are the contracts v0.3 commits to a permanent shape for, and
which therefore need a `docs/spec/<name>.md` doc before v0.3 ships
(OQ-12):

* **`Cust.lock` schema (path-only form)** — `docs/spec/cust-lock.md`.
  Driven by V3D-8. The v0.4 milestone *extends* the schema rather
  than replacing it; the lock format version bumps to `2` only if
  the extension is non-additive.
* **`[workspace]` schema** — folded into the existing manifest
  section of [cust-design.md §3](cust-design.md). Subkeys frozen
  at v0.3: `members`. v0.4 adds `dependencies` (V3D-9) and
  potentially `resolver = "2"`.
* **`#cust use <name>;` syntax for cross-crate imports** —
  recorded in the parent design's §4 alongside the existing
  `#cust use crate::<module>;` syntax. Driven by V3D-6.
* **Workspace target dir layout** — extends the v0.1 layout note
  with the workspace-root-relative shape. v0.3 *extends* the
  layout (single shared `target/`, per-member subdirs); no
  existing single-crate path is repurposed.

A v0.3 PR that lands without the corresponding spec doc is not
done.

## Roadmap impact (for [cust-design.md §17](cust-design.md))

This reframing changes the parent design's roadmap. The §17 bullet
list needs the following edits when v0.3 starts landing:

* **v0.3** — was "dependency resolver & registry"; becomes
  "workspaces & path dependencies".
* **v0.4** — gains the deferred work from the old v0.3 scope:
  registry (initial wire protocol), `cust add`, version resolution,
  `Cust.lock` source hashes, build scripts (`build.cust.c`) + the
  §12 hang-protection timeout. The existing v0.4 contents (plugin
  v1, fixed-point iteration, `[[cust::test]]`, fork-based test
  isolation, `cust_pub_repr`, `-jN` parallelism) stay.
* **v0.8** — was "workspaces, `build.cust.c`, generated-source
  splicing". Workspaces and build scripts move out (to v0.3 and
  v0.4 respectively). v0.8 retains generated-source splicing
  beyond `include_generated!` and any workspace-extension work
  (e.g. `[workspace.metadata]` consumer conventions) that didn't
  fit in v0.3.
* **OQ-6 (workspace inheritance)** — partially shipped in v0.3
  (the workspace itself exists) but the `[workspace.dependencies]`
  + `dep = { workspace = true }` shape stays at v0.4 per V3D-9.

The roadmap edit lands in the same PR that flips this doc from
🚧 planning to ✅ shipped, so that §17 never points at a v0.3
description that's stale.
