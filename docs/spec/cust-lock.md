# `Cust.lock` schema

**Status:** v1 (shipped in cust v0.3).
**Owning contract:** OQ-12 (contract surface versioning) in
[../design/cust-design.md](../design/cust-design.md).
**Driven by:** V3D-8 in [../design/v0.3.md](../design/v0.3.md).

`Cust.lock` records the resolved workspace dependency graph. v0.3
ships the v1 schema below; v0.4 will **extend** the same schema
with per-dep source metadata (registry, sha256, version) and bump
`lock_format_version` to `2` *only* if the extension is
non-additive.

The lock file lives at the **workspace root** (sibling to the
`Cust.toml` declaring `[workspace]`). It is intended to be
committed to version control by the workspace owner — for binary
crates Cargo-style, by convention.

A single-crate (non-workspace) project does **not** produce a
`Cust.lock` in v0.3: there are no edges to record. v0.4 may
revisit this when registry deps land.

---

## File location

```text
<workspace-root>/Cust.lock
```

Sibling to the workspace root's `Cust.toml`. The CLI's
`Workspace::discover` algorithm walks up from cwd to identify the
workspace root; the lock lives there regardless of where in the
tree the user invoked `cust build`.

`Cust.lock` is mtime-stable across no-op builds: the driver
re-renders the file's full content on every successful build but
only writes to disk when the rendered bytes differ. Consumers
that watch the file (IDEs, CI invalidation) can rely on the
mtime not advancing when no graph change occurred.

## Top-level schema (v1)

```toml
# @generated by cust — DO NOT EDIT by hand.
# This file records the resolved workspace member graph.
# v0.3: path deps only; no version pins or source hashes.

lock_format_version = 1
workspace_root = "/abs/path/to/workspace/at/lock/time"

[[member]]
name = "<package name>"
version = "<package version>"
path = "<workspace-root-relative path>"
dependencies = ["<dep name>", "..."]
```

### Top-level keys

| Key | Type | Required | Description |
|---|---|---|---|
| `lock_format_version` | integer | yes | Schema version. v1 in v0.3. Bumps only on non-additive change. |
| `workspace_root` | string | yes | Absolute path of the workspace root at the time the lock was written. Used to detect moved-but-not-rebuilt workspaces in v0.4+ (v0.3 records but does not yet check). |

### `[[member]]` array of tables

One entry per workspace member, **sorted alphabetically by `name`**.
Implicit-root members (root-is-also-a-member shape) are recorded the
same as listed members; the workspace-root-relative `path` for the
implicit root is `"."`.

| Key | Type | Required | Description |
|---|---|---|---|
| `name` | string | yes | Member's `[package].name`. |
| `version` | string | yes | Member's `[package].version`. v0.3 records this for diagnostic display only; v0.4 will use it for the registry-extension. |
| `path` | string | yes | Workspace-root-relative directory containing the member's `Cust.toml`. The implicit root member's `path` is `"."`. |
| `dependencies` | array of string | yes | Names of other workspace members this member depends on, **alphabetically sorted**. Names rather than paths — the target's own `[[member]]` entry resolves the path. |

### Example

For the headline-outcome workspace in v0.3.md (`app` depending on
`util`):

```toml
# @generated by cust — DO NOT EDIT by hand.
# This file records the resolved workspace member graph.
# v0.3: path deps only; no version pins or source hashes.

lock_format_version = 1
workspace_root = "/home/user/my_project"

[[member]]
name = "app"
version = "0.1.0"
path = "app"
dependencies = ["util"]

[[member]]
name = "util"
version = "0.1.0"
path = "util"
dependencies = []
```

## Versioning rules

| Change shape | Action |
|---|---|
| New optional key on `[[member]]` (e.g. v0.4's `[[member.source]]`) | Additive — v1 keeps |
| New optional top-level key (e.g. v0.4's `[registry]` block) | Additive — v1 keeps |
| Removing a key, renaming a key, changing a type | Breaking — bump to v2 |
| Changing the meaning of an existing value | Breaking — bump to v2 |
| Adding a new required key to an existing table | Breaking — bump to v2 |

A v0.3 cust driver reading a `lock_format_version > 1` lock file
errors with `error: Cust.lock format version <n> is newer than
this cust supports (1); upgrade cust`. A v0.4+ driver reading a
v1 lock is expected to either accept it as-is (when the missing
fields are optional) or rewrite it on the next build.

## Out of scope (v0.4+)

The v0.4 milestone extends this schema with:

* Per-dep `[[member.source]]` tables tagging each dep as
  `kind = "path"` (current default), `kind = "registry"`, or
  `kind = "git"`, along with the source-specific metadata
  (registry URL, version, sha256, git rev).
* A top-level `[registry]` table identifying which registry
  configuration was used to resolve the graph.
* A top-level `[metadata.cust]` table for future driver-internal
  pins that the user shouldn't touch.

Source hashes (sha256 of the source tarball) are also a v0.4
addition. v0.3's path-only graphs don't need them — the v0.2
fragment-header stamping already handles file-level
incrementality, and path deps have no "source you fetched from a
remote you might want to verify" dimension.

---

## Editing the file by hand

Don't. Edit your `Cust.toml` files and re-run `cust build`; the
driver regenerates the lock. The `@generated` banner is part of
the contract — hand-edits are not preserved across builds.

A future `cust update [<member>]` command (planned for v0.4
alongside the registry resolver) will be the supported way to
recompute the lock without otherwise rebuilding.

---

## Compatibility with `cargo`'s `Cargo.lock`

The two formats are intentionally distinct (different file name,
different schema). `Cargo.lock` carries Cargo-specific concepts
(`package.source = "registry+https://..."`, checksums in
`[metadata]`, `resolver` version) that don't map cleanly onto the
cust dependency model. Side-by-side coexistence of
`Cargo.lock` and `Cust.lock` in the same repository is supported
and expected for projects that ship both Rust and cust crates.
